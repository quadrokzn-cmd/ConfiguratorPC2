# 2026-05-07 — Мини-фикс: парсеры + перепрогон матчинга

## 1. Какая задача была поставлена

После диагностики Волны 2 нужно было закрыть три причины аномальной маржи 89–99 %, найденные днём ранее:

1. `name_attrs_parser.print_speed_ppm` хватал «3» из «А3» вместо реальной скорости из «≥ 20» (regex прыгал на первую цифру после «печати»).
2. `name_attrs_parser.max_format` не ловил «в формате А3», «формат сканирования А3», «поддержка А3» — только узкий «формат [печати] A4/A3» в начале фразы.
3. `ingest/card_parser._parse_items` не извлекал per-position цены для multi-position лотов — 47 % тендеров (76 шт., 303 позиции) пропущены матчингом.

Затем — перепрогон матчинга на той же БД (без повторного похода в zakupki) и сравнение «до vs после».

## 2. Как я её решал

Пошагово, четыре части (A → B → C → D), плана как такового не строил — задача линейная.

**A. `name_attrs_parser.py`**

- `parse_print_speed_ppm` переписал в две стадии: сначала ищу окно после слова «скорость» (до `;` или 200 символов), внутри — два явных «маркер→число» паттерна с lookbehind `(?<![АAaа])`: `(?:не\s+менее|≥|>=|от)\s*(\d+)` и `(\d+)\s*(?:стр\.?\s*/\s*мин|стр/мин|ppm)`. Если ни один не сработал — fallback на старую эвристику «первое число после "печати"», тоже с lookbehind. Идея — **явные маркеры приоритетнее эвристики**, число НЕ может стоять сразу после буквы «А»/«A».
- `parse_max_format` расширил с одного regex до четырёх: «формат [печати/сканирования/копирования] A3», «в формате A3», «поддержка A3 / возможность … A3», «сканирование/печать [текст] A3». Собираю все совпадения в `set` и применяю правило «**A3 побеждает A4 при одновременном упоминании**» (A3-устройство умеет и A4 — это не противоречие).
- `parse_print_technology` оставил как есть: уже корректен после Волны 2, эквивалентность лазер/LED/электрография обрабатывается в `attribute_rules.py`. Для DoD написал ещё пару тестов (`лазерная`, `светодиодная`) — оказалось, они уже были, но я добавил третий (`струйная`).
- 7 новых тестов в `tests/test_match_name_parser.py` (24 всего). Все зелёные.

**B. `card_parser.py`**

Реверс-инжиниринг по двум фикстурам показал: zakupki в реальной разметке использует заголовок **«Цена за ед., ₽»**, а в коде искалось «Цена за единицу» / «Цена единицы». Substring-поиск не находил совпадения — отсюда `nmck_per_unit=None` для всех multi-position лотов. Заменил список на `["Цена за ед", "Цена единицы"]` — стало находить.

Дополнительно: при работающем извлечении цены вылезли «строки-единицы» (имена «Гигабит в секунду», «Ампер-час (3,6 кКл)», «Градус Цельсия») из expander rows вложенной таблицы характеристик. Они проходили `_is_meaningful_position_name` (длина > 12, два слова), но не имели ни КТРУ, ни qty, ни price. Добавил жёсткий фильтр: позиция без КТРУ и без распарсенных `qty_raw`/`price_raw` отбрасывается. Это уменьшило число `tender_items` на 47 строк (мусор) и оставило только настоящие позиции.

5 новых тестов в `tests/test_ingest_card_parser.py` (7 всего): три multi-position лота из диагностики (по 3 точки проверки в каждом), single-position, лот без структуры цен. Все зелёные.

**C. `scripts/reparse_cards.py`**

Идемпотентный скрипт: SELECT всех `(reg_number, url, raw_html)` из `tenders` (162 строки), для каждой — `parse_card → compute_flags → upsert_tender`. `upsert_tender` уже делает full DELETE+INSERT для `tender_items` внутри транзакции, так что не оставляет грязи. Печатает «до/после» по `items_total / items_with_per_unit / multi-position items без per-unit`. Прогнал дважды — повторный запуск даёт `Δ=0`.

Бонус: `scripts/_dump_raw_html.py` — выгрузка raw_html выбранных reg_numbers в `tests/fixtures/raw_html/`. Использован для реверса и тестов.

**D. Перепрогон матчинга**

`scripts/run_matching.py` с `full_recompute=True`. Зафиксировал «до» через psql (одну простыню запросов) и затем запустил.

## 3. Решил ли — да / нет / частично

**Технически — да, по DoD.** Все четыре пункта DoD закрыты:

- print_speed_ppm не хватает число из «А3»/«A4» — тест `test_print_speed_does_not_grab_digit_from_a3` проходит.
- max_format ловит расширенные паттерны — 4 новых теста.
- card_parser извлекает per-position цены — 3 теста на raw_html-фикстурах.
- scripts/reparse_cards.py идемпотентен — повторный прогон даёт Δ=0.

**По бизнес-результату — частично.** Маржа практически не изменилась:

| метрика | до | после | дельта |
|---|---|---|---|
| `tender_items` всего | 485 | 438 | −47 (мусор отфильтрован) |
| Items с `nmck_per_unit` | 79 | 438 | **+359** |
| Multi-position items без per-unit | 406 | 0 | **−406** |
| `matches.match_type='primary'` | 79 | 187 | +108 |
| Тендеров с primary | 79 | 144 | +65 |
| Маржа median per tender | 89.14 % | 88.77 % | −0.37 пп |
| Маржа max | 99.42 % | 99.76 % | +0.34 пп |

Маржа не упала, потому что:

- **46 / 187 primaries имеют пустые `checks`** — у этих лотов в `name` есть характеристики, но **не из нашей 5-атрибутной схемы critical** (`max_format`/`colorness`/`duplex`/`print_speed_ppm`/`print_technology`). Заказчики на zakupki указывают другое: ресурс картриджа, dpi, время выхода первого отпечатка, класс энергоэффективности — и полагаются на КТРУ-код для базовых параметров. Примеры: [0373100056024000064](https://zakupki.gov.ru/epz/order/notice/ea20/view/common-info.html?regNumber=0373100056024000064) («Суммарная емкость автоподачи ≥ 50 л, тонер ≥ 3 ШТ, dpi ≥ 600»), [0319100017826000023](https://zakupki.gov.ru/epz/order/notice/ea20/view/common-info.html?regNumber=0319100017826000023) («время первого отпечатка ≤ 7 С, класс A, dpi ≥ 1200»). У этих 46 лотов и в `tender_items.required_attrs_jsonb` пусто — это не вопрос «достать атрибуты из таблицы карточки в name», в самой карточке полей нашей схемы нет.
- **35 / 187 primaries — это SKU с `n/a` для critical-атрибутов**: они проходят с `needs_manual_verification=True` по решению собственника №19 (никогда не отбрасывать — пометить ручной проверкой). Это правильное поведение, но оно держит дешёвые SOHO-SKU в ответах.
- Остальные 106 матчей — реальные совпадения, у них тоже маржа высокая (median 86.6 %), потому что лоты zakupki часто публикуются с НМЦК на 80–90 % выше нашего cost_base. Это **не баг матчинга, а реалия рынка**.

## 4. Эффективно ли решение, что можно было лучше

**Что получилось хорошо:**

- Реверс HTML занял 5 минут, потому что фикстуры лежали локально в `tests/fixtures/raw_html/` — сэкономил один поход в zakupki и зафиксировал эталонные данные для будущих регрессий.
- `reparse_cards.py` через существующий `upsert_tender` — никакого дублирования транзакционной логики. Идемпотентность даром.
- Подход «явный маркер → число» вместо «первая цифра после "печати"» — устойчивее к шумам в `name`. Это паттерн, который стоит держать в голове для остальных attribute parsers.

**Что было бы лучше:**

- Я не сразу понял, что текущий `card_parser` не извлекал per-unit **вообще** — даже для single-position. БД заполнялась через `derive_single_position_nmck` в matcher (`UPDATE … nmck_per_unit = nmck_total/qty WHERE count(items)=1`). Только после прогона `parse_card` на single-position фикстуре стало видно, что и там `nmck_per_unit=None`. На это потерял минут 10. Урок: при фиксе парсера всегда **сначала прогнать его на эталонных данных** — не строить гипотезы по тому, что в БД.
- Ожидание собственника «маржа упадёт после фикса» оказалось завышенным. Если бы я сразу промерил долю primary с пустыми `checks` (после Волны 2, до моего фикса), стало бы видно, что главный рычаг — не name-parser, а структурный парсер карточки (`required_attrs_jsonb`). Я бы тогда это явно подсветил перед стартом и сэкономил собственнику ожидание «вот сейчас всё починится».

## 5. Как было и как стало

**Было (после Волны 2 + диагностики, до этого мини-фикса):**
- 162 лота, 485 позиций, 79 с per-unit (только single-position через `derive_single_position_nmck`).
- 76 multi-position тендеров (303 позиции) полностью без матчей — вылетают на стадии «items_no_nmck_per_unit» в `MatchingStats`.
- 79 primary матчей, медиана маржи 89.14 %, max 99.42 %.
- name-parser хватает «3» из «А3» вместо «20» из «≥ 20». На лоте 0373100031126000016 SKU `hp:7WN42B` с `print_speed_ppm=7` проходит фильтр (ложно успешный).
- name-parser не ловит «в формате A3» — на лоте 0358200055826000034 `max_format` не извлекался, и SKU без формата проходил без проверки.

**Стало (после фикса + reparse + перематчинг):**
- 162 лота, 438 позиций, **все 438 с per-unit** (мусорные строки-единицы отфильтрованы, multi-position наполнены).
- 0 multi-position items без per-unit.
- 187 primary матчей (на 144 тендерах), медиана маржи 88.77 %, max 99.76 %.
- На 0373100031126000016 `print_speed_ppm` теперь возвращает 20; SKU `hp:7WN42B` (7 ppm) **отбрасывается** атрибутным фильтром — primary стал другим. На 0358200055826000034 `max_format=A3` извлекается и попадает в `rule_hits.checks` (сейчас SKU всё равно проходит как `n/a` с manual-флагом — это уже не вопрос парсера).
- 12 новых тестов: всего 177 (было 165), все зелёные.

**Открытые вопросы к оркестратору** (для постановки следующих чатов):

1. **БАГ в `card_parser._extract_position_attrs` — пропущен expander-блок с полной таблицей характеристик.** Исходный диагноз про «нет атрибутов в карточке» был неверным — обнаружен по скриншоту собственника от 2026-05-07. На zakupki у каждой позиции есть **раскрывающийся блок** (chevron) с полной таблицей характеристик: «Наименование характеристики | Значение | Ед.измерения | Инструкция». Эта таблица в HTML лежит в `<tr class="truInfo_…" style="display:none">` — **соседней `<tr>`-сестре** относительно строки позиции, не внутри неё. В коде:
   ```python
   required_attrs_jsonb=_extract_position_attrs(row)
   ```
   `row` = текущий `<tr>` позиции, `_extract_position_attrs` делает `scope.find_all("table")` — но таблица характеристик в **следующем** `<tr>`. Поэтому `required_attrs_jsonb` пуст для **всех** 438 позиций в БД. Пример: в [0373100056024000064](https://zakupki.gov.ru/epz/order/notice/ea20/view/common-info.html?regNumber=0373100056024000064) у МФУ есть «Скорость ч/б A4 ≥ 30 стр/мин», «Класс A», «USB+LAN», «ОЗУ ≥ 128 МБ», «Двусторонняя Да» — и ни одна из этих строк не попадает в матчинг. Фикс: передавать в `_extract_position_attrs` объединённый scope из текущей `<tr>` + следующей `<tr class="truInfo_…">` (или искать expander через `row.find_next_sibling("tr", class_=re.compile(r"truInfo_"))`). После фикса нужен ещё нормализатор «zakupki-row → schema-key» (привести «Скорость черно-белой печати в формате А4 по ISO/IEC 24734, стр/мин ≥ 30» к `print_speed_ppm=30`, «Цветность Черно-Белая» → `colorness="ч/б"` и т.д.) — ровно те же эвристики, что в name-parser, но на структурных данных. После этого `merge_required_attrs(jsonb_attrs, parsed_attrs)` (уже есть в matcher) автоматически даст jsonb приоритет. **Ожидаемый эффект:** медиана маржи должна упасть до 30–60 %, потому что атрибутный фильтр заработает на полную мощность. Это ~1 чат с тестами на сохранённых raw_html-фикстурах.
2. **35 / 187 primaries — SKU с `n/a` для critical-атрибутов**: они проходят с `needs_manual_verification=True` по решению собственника №19 (никогда не отбрасывать — пометить ручной проверкой). Это правильное поведение, но оно держит дешёвые SOHO-SKU в ответах. Если собственник примет решение «для лотов с указанным `max_format=A3` SKU без подтверждённого `max_format` (или с `max_format='A4'`) должны отбрасываться» — это меняет решение №19. Я этого не делаю до явной постановки.
3. **6 SKU из топ-маржи всё ещё `hp:7WN42B`** (бюджетный струйник 3960 ₽) — вопрос к domain-эксперту по аукционам (собственнику-1): корректно ли обогащение этого SKU? Зафиксировано в диагностике Волны 2 пункт 6, не дублирую.
