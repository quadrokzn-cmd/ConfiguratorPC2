# 2026-05-11 — Мини-этап 9e.3: раскладка ingest-worker'а на офисном сервере в РФ

## 1. Какая задача была поставлена

Этап 9e (production-ingest архитектура, путь A). 9e.1 завёл роль `ingest_writer` на pre-prod-БД, 9e.2 написал CLI `scripts/run_auctions_ingest.py`. В 9e.3 нужно физически развернуть этот CLI на офисном сервере в РФ (статический IP, не блокируется zakupki.gov.ru) и поставить регулярный запуск через Windows Task Scheduler. После 24 часов наблюдения acceptance → 9e.4 (повтор роли на prod + переключение).

В отличие от предыдущих мини-этапов 9-серии, 9e.3 — **не задача для Claude Code исполнителя**: всё происходит на удалённом физическом сервере (Windows Server 2012 R2 в офисе) через RDP. Claude Chrome тоже не подходит — он управляет браузером, не RDP. Работу выполнял собственник в RDP-сессии под пошаговыми инструкциями оркестратора (текущий чат).

## 2. Как решал

**Sanity check.** Первый шаг — узнать, что вообще есть на офисном сервере. PowerShell-блок собрал: ОС (Windows Server 2012 R2 Standard, build 6.3.9600), пользователь (`server1c\Администратор`, IsAdmin=True, **кириллица в имени** — потом это укусит), PowerShell 4.0, свободное место (C: 246, D: 750, F: 311 ГБ), отсутствие Python и Git, **`Test-NetConnection maglev.proxy.rlwy.net:32320 → True`** (сетевой путь до Railway-PG открыт).

**Установка Python.** Первая попытка дала `Python 3.11.0rc1` (release candidate от августа 2022, без security-патчей) — собственник скачал не то с python.org. Переустановили на `Python 3.11.9` stable (август 2024) в `D:\Python311\`, pip 24.0. Add to PATH + Install for all users + advanced location D:\.

**Установка Git for Windows.** Версия 2.54.0.windows.1, все дефолтные опции инсталлятора (PATH с командной строкой, Bundled OpenSSH, OpenSSL backend, Checkout Windows-style/commit Unix-style, MinTTY, Git Credential Manager). Установка в `C:\Program Files\Git\`.

**SSH Deploy Key.** Первая попытка `ssh-keygen` в Git Bash под `~/.ssh/` упала с `Could not create directory '/c/Users/\300\344\354\350\355\350\361\362\360\340\362\356\360/.ssh'` — MSYS2/Git Bash не справляется с кириллическим именем пользователя. Решение: всю SSH-инфраструктуру вынесли в путь без кириллицы — `D:\AuctionsIngest\.ssh\`:
```
ed25519-ключ → /d/AuctionsIngest/.ssh/id_ed25519_github (без passphrase, read-only)
~/.ssh/config → /d/AuctionsIngest/.ssh/config (с явными IdentityFile и UserKnownHostsFile)
chmod 600 — обязательно
```
Публичный ключ добавлен в GitHub Deploy Keys репо `quadrokzn-cmd/ConfiguratorPC2` без write-доступа. Проверка `ssh -F /d/AuctionsIngest/.ssh/config -T git@github.com` → `Hi quadrokzn-cmd/ConfiguratorPC2! You've successfully authenticated...` — присутствие имени репо в приветствии подтверждает, что ключ распознан как deploy key (не user-key).

**Клонирование репо.** `GIT_SSH_COMMAND="ssh -F /d/AuctionsIngest/.ssh/config" git clone git@github.com:quadrokzn-cmd/ConfiguratorPC2.git` в `D:\AuctionsIngest\ConfiguratorPC2\`. Размер ~150 МБ (включая backlog #8 — 2 SQL-дампа ~55 МБ в истории, не чистим — Task Scheduler работает с shallow surface). Последний коммит `1eb74fd` (9e.2) совпал с remote master.

**venv + зависимости.** `python -m venv venv` (без активации через `Activate.ps1` — ExecutionPolicy Restricted блокирует). Затем `.\venv\Scripts\python.exe -m pip install --upgrade pip` (до 24.0+) и `.\venv\Scripts\python.exe -m pip install -r requirements.txt`. Установка ~5 минут, без проблем со сборкой — все wheel'ы есть для Python 3.11 + Windows. Проверка: APScheduler 3.11.2, beautifulsoup4 4.14.3, fastapi 0.136.1, httpx 0.28.1, lxml 6.1.0, psycopg2-binary 2.9.12, python-dotenv 1.2.2, SQLAlchemy 2.0.49 — все ключевые на месте.

**`.env` через notepad (через катастрофу с Read-Host).** Первая попытка использовала PowerShell-блок с `Read-Host` для DSN. Собственник скопировал весь блок (включая `Read-Host`) одним paste'ом — `Read-Host` остановился, ждал ввода, и следующие команды из буфера обмена «влетели» в Read-Host как «введённое значение». В итоге `.env` содержал 28 байт (только префикс `INGEST_WRITER_DATABASE_URL=` + `\n`, без DSN). Дополнительно: `[System.Text.UTF8Encoding]::new($false)` не работает в PS 4.0 — синтаксис `::new()` появился в PS 5.0. Использовали `New-Object System.Text.UTF8Encoding($false)`.

Урок: paste-в-Read-Host в RDP-сессии под PS 4.0 без PSReadLine — это анти-паттерн. Решение: открыли `notepad .env`, на клавиатуре набрали `INGEST_WRITER_DATABASE_URL=`, после `=` — правым кликом мыши вставили DSN из RDP shared clipboard. Сохранили (Ctrl+S). Размер 158 байт — корректно. Также зафиксировано как memory `reference_office_server_setup.md`.

**Первый ручной запуск ingest.** `.\venv\Scripts\python.exe scripts\run_auctions_ingest.py --env-file .env` с офиса. **159 cards seen / 159 parsed / 0 failed / 0 inserted / 159 updated за 752 секунды**. Никаких `permission denied` — миграция 0035 покрывает всё, что нужно ingest. KTRU использовались `26.20.16.120-00000001` (Принтер) + `26.20.18.000-00000001` (МФУ), флаги выкосили 51 (excluded_region=4 / below_nmck=3 / over_unit_price=20 / no_watchlist_ktru=14 / no_positions=10). **Это был первый production-ingest с российского IP — главная архитектурная цель 9e доказана работающей.**

**Wrapper-скрипт для логов.** `D:\AuctionsIngest\run_ingest.ps1` (~806 байт). Делает:
1. timestamp в имени лога;
2. `Set-Location D:\AuctionsIngest\ConfiguratorPC2`;
3. `& cmd /c "set PYTHONIOENCODING=utf-8 && venv\Scripts\python.exe scripts\run_auctions_ingest.py --env-file .env > \"$logFile\" 2>&1"` — запуск через `cmd /c` для byte-level redirect в файл без вмешательства PS-кодировки **и** без оборачивания stderr в `NativeCommandError`;
4. ротация: оставить 30 последних `ingest_*.log`.

Первая версия wrapper'а использовала `Tee-Object` — лог получился в cp1251/cp866 («каля-баляка» для кириллицы), а PS оборачивал stderr-сообщения python.exe в красный `NativeCommandError`. После фикса (cmd /c + PYTHONIOENCODING=utf-8) — лог чистый UTF-8, видны нормальные `«Возможность сканирования в форматах»` / `«ФЕДЕРАЛЬНОЕ ГОСУДАРСТВЕННОЕ БЮДЖЕТНОЕ УЧРЕЖДЕНИЕ "ЦЕНТРАЛЬНЫЙ ВОЕННЫЙ КЛИНИЧЕСКИЙ САНАТОРИЙ «АРХАНГЕЛЬСКОЕ»"`.

**Task Scheduler через PowerShell.** Вместо GUI (taskschd.msc) — `Register-ScheduledTask` командлет:
```
Trigger: Once -At now+2min -RepetitionInterval 2h -RepetitionDuration 3650d
Action: powershell.exe -NoProfile -ExecutionPolicy Bypass -File D:\AuctionsIngest\run_ingest.ps1
Settings: MultipleInstances=IgnoreNew, ExecutionTimeLimit=1h, RestartCount=2, RestartInterval=10min
Principal: server1c\Администратор, Password (через Read-Host -AsSecureString → PtrToStringAuto), RunLevel=Highest
```
Пароль учётки `Администратор` получен через `Read-Host -AsSecureString` (одна-строка-один-paste, без блок-paste-капкана). После Register-ScheduledTask: `State : Ready` — задача создана.

**Ручной test-запуск + acceptance.** `Start-ScheduledTask -TaskName "AuctionsIngest"`. Лог `D:\AuctionsIngest\logs\ingest_2026-05-11_183559.log` — 433 971 байт, статистика `cards_seen=159, cards_parsed=159, cards_failed=0, inserted=0, updated=159 за 776.5 секунд`. UTF-8 в логе работает. State после завершения — `Ready` (готова к следующему запуску по триггеру через 2 часа). Это **третий успешный прогон production-ingest с офисного IP** — wrapper + Task Scheduler работают идентично ручному запуску.

## 4. Эффективно ли решение, что можно было лучше

**Эффективно:**
- **Sanity check сразу** дал точную картину (ОС, версия PS, кириллица в имени, отсутствие Python/Git, доступность Railway-PG). Это позволило избежать гадания.
- **Перенос SSH в `D:\AuctionsIngest\.ssh\`** — единственный надёжный путь обхода кириллицы в `%USERPROFILE%`. Зафиксирован в memory.
- **`cmd /c` вместо `Tee-Object`** — обходит и PS-кодировку, и `NativeCommandError`. Простое и надёжное решение для wrapper'а.
- **`Register-ScheduledTask` через PS** вместо GUI taskschd.msc — воспроизводимо, документируется одним блоком, единственный интерактив (пароль) — через `Read-Host -AsSecureString` в отдельной строке.
- **Memory обновлена 3 раза по ходу этапа** (feedback_secret_via_grep ещё в 9e.1; reference_windows_psql в 9e.1; reference_office_server_setup в 9e.3 + дополнения по UTF-8 / NativeCommandError) — будущие исполнители на офисе сразу учитывают ограничения.

**Что было плохо и чему научился (помимо собственных скверных моментов с copy-paste):**

- **Read-Host + copy-paste блок** = **серьёзный анти-паттерн** в RDP/PS 4.0 без PSReadLine. Когда собственник копирует мой блок (включающий Read-Host и команды ниже), весь буфер обмена пишется в console-stream одной непрерывной строкой; Read-Host берёт первую строку как ввод, а остальные команды могут «влететь» в Read-Host как продолжение строки. На 9e.1 это привело к тому, что **первый сгенерированный пароль попал в LLM-контекст через grep-stderr**; на 9e.3 — к 28-байтовому `.env` без DSN. **Урок:** все мои блоки с Read-Host теперь явно маркирую 🟡 «по одной строке», блоки без Read-Host — 🟢 «весь блок целиком». Это поведение зафиксировано в инструкциях, но не в memory — стоит рассмотреть как отдельный feedback для будущих deploy-этапов через RDP.
- **Не сразу проверил версию PowerShell в первом sanity-check** — Windows Server 2012 R2 идёт с **PS 4.0**, без PSReadLine, без `::new()`, без `Set-Clipboard`. Из-за этого первый `[System.Text.UTF8Encoding]::new($false)` упал на `MethodNotFound`. Нужно сразу было проверить PS-версию первой командой sanity-check (она была там, но я не отметил последствие).
- **rc1 vs stable**: первая ссылка на python.org не явно указала на 3.11.9 — собственник попал на rc1 (старая страница из архива). На будущее: при инструкции скачать Python — **сразу давать прямую ссылку на `.exe`** конкретной версии, например `https://www.python.org/ftp/python/3.11.9/python-3.11.9-amd64.exe`.
- **Можно было бы предусмотреть 5-минутный smoke-тест paste-в-Read-Host** перед первой реальной попыткой с DSN. Я попросил тест paste только после первого фейла. На будущее — для любой команды с интерактивным вводом через RDP сначала проверять paste коротким значением.

**Что осталось технически открыто (не блокер для 9e.4):**
- 24-часовое наблюдение — Task Scheduler сам сделает ~12 прогонов, я в этот чат-сессию не вижу их результат. Собственник может через сутки проверить `D:\AuctionsIngest\logs\` и сообщить, если что-то сбойнуло.
- `Sentry` для office-ingest не настроен — в случае ошибок мы узнаем только из лог-файлов на офисе. Не критично для текущей фазы, но если в течение 24h что-то ловим — можно добавить Sentry-DSN в `.env` и `import sentry_sdk; sentry_sdk.init(...)` в `scripts/run_auctions_ingest.py`. Это не было в DoD 9e.3.
