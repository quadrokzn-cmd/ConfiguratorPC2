**Шаг 3. Pitfall с raw-плейсхолдерами Railway.** Прочитал значение
через python (без печати самого URL — только структурную маску). Сразу
заметил: длина 64 символа, маска `xxxxxxxx:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx@:/xxxxxxx`

template-переменные `${RAILWAY_TCP_PROXY_DOMAIN}:${RAILWAY_TCP_PROXY_PORT}`,

собственника через AskUserQuestion — выбран вариант скопировать
уже-резолвленный URL из вкладки Connect (или Settings → Public
Networking). После повторного «продолжай» URL распарсился:
`user=postgres, host=turntable.proxy.rlwy.net, port=13528, db=railway`.

**Шаг 4. Применение миграции 0035 через psycopg2.** Решил не идти через
`psql heredoc + INSERT schema_migrations` (как в 9e.1), а сразу собрать
всю операцию в одном python-блоке через `psycopg2`. Преимущества:
никаких subprocess'ов, никаких temp .sql на диске, пароль вообще не
покидает память процесса. Скрипт:
1. Читает `DATABASE_PUBLIC_URL` из gitignored `.env.local.prod.v1`.
2. Открывает `psycopg2.connect(url, sslmode='require')`, `autocommit=True`
   (DDL не любит транзакции с CREATE ROLE).
3. Перед стартом проверяет `pg_roles.ingest_writer` — `True`. То есть
   роль **уже была**. Railway prod-deploy успел автоматически применить
   `migrations/0035` через `scripts/apply_migrations.py` при последнем
   prod-rebuild — это было заложено в DoD как ожидаемый сценарий.
4. Выполняет содержимое файла миграции — DO-блок `IF NOT EXISTS`
   идемпотентно пропустился, GRANT-ы перевыдались без эффекта.
5. `INSERT INTO schema_migrations ... ON CONFLICT DO NOTHING` —
   `RETURNING` пустой, запись была от `2026-05-11 12:00:56.851452+00:00`.
6. `SELECT rolname, rolcanlogin FROM pg_roles WHERE rolname='ingest_writer'`
   — `canlogin=False`. Пароля ещё нет — роль создана только как `NOLOGIN`
   через DO-блок миграции.
7. Дополнительно вывел список GRANT-ов из `information_schema.role_table_grants`
   — 6 таблиц: `settings/ktru_watchlist/excluded_regions` (SELECT),
   `tenders/tender_items/tender_status` (DELETE/INSERT/SELECT/UPDATE). Полное
   соответствие миграции.

**Шаг 5. Генерация prod-пароля и ALTER ROLE.** Тоже в одном
python-блоке (никаких subprocess'ов, никаких temp .sql):
1. `secrets.token_urlsafe(32)` → 43-символьный пароль (base64url-safe:
   `[A-Za-z0-9_-]`, без апострофов, `@`, `:`, `/`). `assert` на
   отсутствие проблемных символов.
2. `psycopg2.connect(base_url, sslmode='require')` под `postgres`,
   `autocommit=True`.
3. `ALTER ROLE ingest_writer WITH LOGIN PASSWORD '<pwd>'` через
   f-строку (SQL-инъекция исключена: `token_urlsafe` не может содержать
   апостроф; парам-плейсхолдер `%s` для PASSWORD в DDL не поддерживается
   psycopg2 — нужен литерал).
4. Verify: `SELECT rolcanlogin FROM pg_roles WHERE rolname='ingest_writer'`
   → `True`.
5. URL-encode пароля через `urllib.parse.quote_plus(pwd)` и формирование
   `INGEST_WRITER_DATABASE_URL_PROD = postgresql+psycopg2://ingest_writer:
   <encoded-pwd>@turntable.proxy.rlwy.net:13528/railway?sslmode=require`.
6. Запись/обновление в `.env.local.prod.v1`: считал весь файл, отфильтровал
   старые строки `INGEST_WRITER_DATABASE_URL_PROD=` (на случай повторного
   запуска), добавил новую строку — идемпотентно.
7. Sanity-check: `psycopg2.connect(dbname=db, user='ingest_writer',
   password=pwd, host=host, port=port, sslmode='require')` → `SELECT
   current_user, current_database()` → `('ingest_writer', 'railway')`.

Пароль за всю операцию ни разу не приближался к stdout/stderr/bash-args
— print'ы выводили только длину (43), класс символов (base64url-safe) и
DSN-маску (без самого пароля).

**Шаг 6. Smoke positive + negative.** Третий python-блок, читает
`INGEST_WRITER_DATABASE_URL_PROD` из gitignored-файла, режет префикс
`postgresql+psycopg2://` → `postgresql://` (psycopg2 не понимает
sqlalchemy-схему). Positive: `SELECT current_user, current_database()`,
`SELECT COUNT(*) FROM settings/ktru_watchlist/excluded_regions`,
`SELECT COUNT(*) FROM (SELECT 1 FROM tenders LIMIT 1) t`. Цикл
INSERT-UPDATE-DELETE: `BEGIN; INSERT INTO tenders (TEST-9E41-SMOKE-001);
INSERT INTO tender_items; INSERT INTO tender_status; UPDATE tenders SET
customer='Smoke Updated'; DELETE FROM tenders WHERE reg_number=...;

**Pitfall с charmap.** Первый запуск smoke упал на `print(...)` с

косметика stdout. Перезапустил с `PYTHONIOENCODING=utf-8` + ASCII `-->`
вместо `→`. Все smoke прошли. Этот pitfall — частный случай известной
проблемы Windows-console-кодировки (уже зафиксирована в

## 4. Эффективно ли решение, что можно было лучше

**Эффективно:**

- **Единый python+psycopg2 блок вместо bash + psql heredoc**. В 9e.1 был
  маршрут «pwd → tempfile.sql → psql stdin → rm -f» — три отдельных
  команды, временный файл на диске, возможные ошибки с правами/codec'ом.
  В 9e.4.1 — всё в одном python-процессе через `psycopg2.connect`. Пароль
  существует только в `pwd = secrets.token_urlsafe(32)` и в гитайгноред-
  файле. Никаких артефактов на ФС, никаких subprocess'ов, никакого
  Windows-psql-багa с `-c "..."`.
- **Маска DATABASE_PUBLIC_URL до парсинга**. Перед попыткой реального
  использования значения я сделал структурную диагностику: длина, число
  `:`/`@`/`/`, маска символов через `0` для цифр и `x` для букв. Это
  сразу выявило, что host:port пусты (raw-плейсхолдеры Railway) — без
  печати самого URL. Если бы я просто попытался коннектиться, ошибка
  psycopg2 могла бы рассказать больше, чем нужно.
- **Spotted Railway raw-template issue**. Это не описано в шаге Л.1
  preprod-deploy.md — на pre-prod собственник копировал значение из
  Public Networking сразу, а на prod выбрал Variables (где placeholder'ы).
  Pitfall зафиксирован в шаге Л.5 — следующий, кто будет делать что-то
  подобное, не наступит на те же грабли.
- **Smoke с явным `psycopg2.errors.InsufficientPrivilege`**. В 9e.1 negative
  smoke шёл через psql и парсил `ERROR: permission denied` из stdout —
  достаточно надёжно, но не структурно. В 9e.4.1 — типизированный exception,
  который не путается с другими ошибками (например, синтаксическими).

**Что можно было лучше:**

- **Charmap-pitfall** с символом `→`. Знал про `PYTHONIOENCODING=utf-8`
  из reference_office_server_setup.md, но забыл выставить с первого
  запуска — потерял один прогон. Урок: при `python - <<PY` на Windows
  всегда устанавливать `PYTHONIOENCODING=utf-8` перед запуском
  (или вообще писать только ASCII в логи скриптов).
- **Можно было одним коннектом**. Я применял миграцию, ALTER ROLE и
  smoke тремя отдельными python-блоками с отдельными connect'ами.
  Можно было в одном — но это удлинило бы heredoc, ухудшило бы
  читаемость и затруднило отдельную валидацию каждого шага. Решил, что
  3 независимых блока с явными границами — правильный trade-off для
  пошагового etap'а.
- **План 9e.4.2** — стоило бы заранее написать промт для следующего
  чата (как DoD говорит «после успешного 24h acceptance pre-prod»), но
  это уже сфера ответственности оркестратора, не моя.

**Как было (до 9e.4.1):**

- На prod-БД (`turntable.proxy.rlwy.net:13528/railway`) одна логин-роль
  `postgres` (суперюзер). После последнего Railway-deploy в prod
  автоматически появилась `ingest_writer NOLOGIN` через

  GRANT-ы выданы — **но пароля не было**, никто роль использовать не мог

- `.gitignore` имел `.env.local.preprod*`, но не `.env.local.prod*` —
  потенциальная дыра при создании prod-файла.
- В `docs/preprod-deploy.md` шаг Л имел только короткий тизер «9e.4

  разделения на 9e.4.1 и 9e.4.2.

**Как стало (после 9e.4.1):**
