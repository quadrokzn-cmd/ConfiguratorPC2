# Этап UI-4.5: перенос auctions/catalog/scheduler из app/ в portal/

**Дата:** 2026-05-11
**Этап Пути B:** 4.5 из 5 (UI-4.5)
**Контекст плана:** `plans/2026-05-11-ui-merge-portal-configurator.md`
**Предыдущий этап:** UI-4 — перенос Конфигуратора (коммит 3f0816e).

## 1. Поставленная задача

Технический пред-этап перед UI-5: вычистить остатки `app/services/` и
`app/scheduler.py`, на которые после UI-4 опирались portal-роутеры,
скрипты ingest и тесты. Цель — устранить кросс-импорт `from
app.services.auctions/catalog ...` (отложенный в UI-4 как «вернёмся
на UI-4.5») и перенести cron-задачу курса USD/RUB на portal-сервис,
чтобы Railway-сервис `configurator` после UI-5 можно было упразднить.

Состав:

- `app/services/auctions/` → `portal/services/auctions/` (ingest, match,
  catalog/enrichment — 4 подпапки, ~22 файла);
- `app/services/catalog/` → `portal/services/catalog/`
  (`brand_normalizer.py`);
- `app/scheduler.py` → код переносится в `portal/scheduler.py` как 5
  новых cron-job'ов `cbr_fetch_<HHMM>` + `ensure_initial_rate()` через
  `portal/main.py` startup;
- Все 36 файлов с `from app.services.{auctions,catalog} ...` — на
  `portal.services.{auctions,catalog}`;
- `app/main.py` — убрать импорты scheduler и `@app.on_event` для него;
- `docs/office-ingest-deploy.md` — добавить раздел про `git pull` на
  офисном сервере (операционно важно).

Ограничения: не менять бизнес-логику auctions/catalog/scheduler — только
пути импортов; не удалять `app/main.py`, `app/auth.py`, `app/config.py`,
`app/database.py` (это UI-5); не менять URL'ы.

## 2. Как решал

1. **Ревизия зависимостей.** `Grep "from app\.services\.(auctions|catalog)"`
   нашёл 42 файла (включая markdown'ы). После фильтрации
   `--glob=*.py` — 36 Python-файлов. Карта потребителей оказалась
   ровно той, что предсказывала UI-4 рефлексия: portal-роутеры
   (`auctions`, `admin_auctions`, `nomenclature`), portal-scheduler
   (через `app.services.auctions.ingest.single_flight`), 7 скриптов
   (`run_auctions_ingest`, `run_matching`, `reparse_cards`,
   `normalize_brands`, `enrich_*`, `auctions_enrich_{export,import}`),
   все тесты `test_auctions/` и `test_catalog/`, плюс внутренние
   импорты в самих auctions/catalog модулях.

2. **`git mv` для переезда.** Два вызова:
   `git mv app/services/auctions portal/services/auctions` и
   `git mv app/services/catalog portal/services/catalog`. Git сохранил
   историю как rename на каждом из 27 файлов. После этого
   `git status -s` показал ровно 27 строк `R  ... -> ...`.

3. **Массовая замена импортов через Python-скрипт.** Скрипт прошёл по
   списку из 36 файлов и для каждого применил две `re.sub`-замены:
   `\bfrom app\.services\.auctions\b → from portal.services.auctions`,
   `\bfrom app\.services\.catalog\b → from portal.services.catalog`
   (плюс зеркальные `\bapp\.services\.X\b → portal.services.X` без
   `from` — для monkeypatch'ей в тестах). Это оказалось надёжнее ручных
   `Edit`-вызовов: один проход — 36 файлов, без рисков опечататься.

4. **`from app.database import engine` → `from shared.db import engine`.**
   В 4 файлах внутри `auctions/catalog/` нашёлся импорт через старый
   re-export `app.database`. Поскольку `app.database` — это просто
   `from shared.db import Base, SessionLocal, engine, get_db`, оба
   варианта работают одинаково. Но идеоматически правильнее: после
   переезда в `portal/` использовать canonical-источник `shared.db`.
   Если этого не сделать, в `portal/services/auctions/...py` останутся
   импорты `from app.database` — выглядит странно (модуль уже в
   `portal/`, а тянет из `app/`).

5. **Cross-import в `price_loaders/orchestrator.py`.** Это был
   главный мотиватор UI-4.5. После переезда 3 импорта в нём
   (`from app.services.auctions.catalog.enrichment.name_parser`,
   `from app.services.auctions.catalog.enrichment.schema`,
   `from app.services.catalog.brand_normalizer`) обновились на
   `portal.services.*` в общей пачке.

6. **Monkeypatch-строки в тестах.** В
   `tests/test_auctions/test_run_auctions_ingest.py` обнаружились
   `monkeypatch.setattr("app.services.auctions.ingest.orchestrator.
   run_ingest_once", ...)` — это литералы для patching, regexp на
   `from app.services...` их не зацепил. Поправил вручную через два
   `Edit`'а.

7. **Перенос cron USD/RUB.** Самая интересная часть. В app/scheduler.py
   жил `BackgroundScheduler` с 5 cron-job'ами (08:30, 13:00, 16:00,
   17:00, 18:15 МСК) + функция `ensure_initial_rate()` для синхронного
   первого fetch при пустой `exchange_rates`. Активировался через
   отдельный флаг `RUN_SCHEDULER`.

   В portal/scheduler.py уже работал `BackgroundScheduler` под флагом
   `APP_ENV=production` / `RUN_BACKUP_SCHEDULER=1`. Перенёс:
   - константа `_CBR_CRON_TIMES` (5 точек в день);
   - функция `_job_fetch_cbr()` — тело cron-задачи (ленивые импорты
     `shared.db.SessionLocal` и `portal.services.configurator.export.
     exchange_rate`, чтобы не утяжелять стартовый импорт scheduler'а);
   - функция `ensure_initial_rate()` (теперь без подчёркивания —
     public, вызывается из `portal/main.py`);
   - регистрация 5 cron-job'ов внутри `init_scheduler()` после
     блока auctions_ingest.

8. **Где жить `ensure_initial_rate()`.** Изначально положил его внутрь
   `init_scheduler()` — между регистрацией jobs и `sched.start()`. Но
   тут же увидел: `tests/test_portal/test_scheduler.py` ставит
   `RUN_BACKUP_SCHEDULER=1` и зовёт `init_scheduler()` для проверки,
   что 6 auto_price-jobs зарегистрированы. С `ensure_initial_rate()`
   внутри — каждый из 4 тестов делал бы реальный HTTP-запрос на ЦБ
   (table труъ-truncate'ется фикстурой `_clean_tables` перед каждым
   тестом, так что `SELECT 1 FROM exchange_rates LIMIT 1` всегда
   возвращает None). Это медленно и хрупко.

   Решение — вынести `ensure_initial_rate()` ИЗ `init_scheduler()` и
   вызывать отдельно из `portal/main.py @app.on_event("startup")`,
   гейтя тем же `_is_enabled()`. Так:
   - на pytest TestClient `_is_enabled()` возвращает False (нет ни
     APP_ENV=production, ни RUN_BACKUP_SCHEDULER=1) — `ensure_initial_rate`
     не запускается, сеть не дёргается;
   - в production startup'е (Railway) флаг True — cron-job'ы регистрируются
     и `ensure_initial_rate` синхронно подтягивает первый курс;
   - `test_scheduler.py` ставит RUN_BACKUP_SCHEDULER=1 и зовёт
     `init_scheduler()` напрямую (не через TestClient), значит startup
     event не фаерит — `ensure_initial_rate` не дёргается.

9. **Удаление `app/scheduler.py`.** `git rm app/scheduler.py` после
   того, как убедился, что нигде в репо нет `from app.scheduler import
   ...` (grep пуст). В `app/main.py` убрал импорт `from app.scheduler
   import ensure_initial_rate, init_scheduler, shutdown_scheduler` и
   два `@app.on_event` handler'а (`_startup_scheduler` и
   `_shutdown_scheduler`). Settings-поле `settings.run_scheduler`
   осталось в `app/config.py` (без потребителей) — удалю в UI-5 вместе
   с `app/config.py`. Не стал трогать сейчас, чтобы не растягивать diff.

10. **Sanity-импорты.** Перед запуском полной регрессии:
    `python -c "import portal.scheduler; import portal.services.auctions.
    ingest.orchestrator; import portal.services.catalog.brand_normalizer;
    print('OK')"` и `python -c "import portal.main; import app.main"` —
    оба отработали без ошибок. Это типичный smoke перед прогоном
    pytest.

11. **Документация.** Обновил четыре файла:
    - `CLAUDE.md` — блок стека (упоминание UI-4.5), таблица «Где что
      искать» (добавлены `portal/services/auctions/` и
      `portal/services/catalog/`), структура папок, блок «Сервисы
      конфигуратора», объединённый раздел «Расписание APScheduler»
      (одна таблица портала, app/scheduler.py больше нет, добавлены
      cron-job'ы `cbr_fetch_<HHMM>`).
    - `docs/ui-architecture.md` — шапка ссылается на UI-4.5, в разделе
      «История» пункт UI-4.5 переписан из «плана» в «выполнено» с
      описанием того, что переехало.
    - `docs/url-migration-map.md` — добавлен раздел «UI-4.5 (2026-05-11)»
      с таблицей переездов модулей, упоминанием перехода с
      `RUN_SCHEDULER` на `RUN_BACKUP_SCHEDULER`, и operational
      предупреждением про офисный сервер.
    - `docs/office-ingest-deploy.md` — новый раздел «Обновление кода
      на офисе после deploy» с пошаговой процедурой `git pull` и
      sanity-проверкой `python -c "from portal.services.auctions.
      ingest.orchestrator import run_ingest_once; print('ok')"`.

12. **Полная регрессия `pytest -q -m "not live"`.** 1892 passed, 1
    skipped, 0 failed за 80 секунд. UI-4 baseline был 1882; разница
    +10 — это, видимо, тесты `tests/test_auctions/` и
    `tests/test_catalog/`, которые до UI-4.5 тоже работали, но
    возможно пара кейсов из них раньше иногда падала на flaky
    сетевых импортах через cross-paths. Никаких regressions не
    появилось, никаких новых тестов я не добавлял (этап чисто
    рефакторный).

## 4. Эффективно или нет, что можно было лучше

**Что прошло хорошо:**

- **Grep + Python-скрипт для массовой замены** оказался идеальным
  инструментом. 36 файлов, 4 паттерна, один проход — без ошибок и
  без необходимости перепрогонять для отдельных файлов. Если бы
  использовал последовательность `Edit`'ов, потратил бы в 5-10 раз
  больше времени.
- **Решение вынести `ensure_initial_rate()` за пределы `init_scheduler()`**
  через анализ test_scheduler.py — сэкономило кучу головной боли с
  HTTP в тестах. Если бы запустил pytest до этого осмысления —
  test_scheduler.py пошёл бы в сеть и либо упал бы по timeout'у на
  CI без интернета, либо был бы медленным.
- **Замена `from app.database import engine` → `from shared.db import
  engine`** в перенесённых файлах — небольшая, но идеоматически
  важная чистка. После UI-5, когда `app/database.py` исчезнет, в
  portal/services/auctions/... не будет внезапно отваливающихся
  импортов через старый re-export.
- **Регрессия 1892 vs 1882** — сразу видно, что покрытие не уменьшилось
  и тесты на auctions/catalog после переезда корректно подцепляются.
- **`git mv` с детальной историей** — `git log --follow
  portal/services/auctions/ingest/orchestrator.py` теперь показывает
  всю историю от создания файла в `app/services/auctions/`, что
  важно для blame.

**Что можно улучшить:**

- **`settings.run_scheduler` в `app/config.py`** остался как dead
  field. Без потребителей он бесполезен, но удалять прямо сейчас
  значит трогать `app/config.py`, который тоже уйдёт в UI-5. Решил
  не размывать diff UI-4.5.
- **Папка `app/services/__init__.py`** теперь пустая, но осталась.
  Раз `app/services/` опустошена, можно было удалить и сам пакет.
  Но: `app/main.py` (живой) импортируется через `app.*`, и если
  Python увидит непустую `app/` иерархию без `services/` — ничего
  не сломается, но и удаление папки даст лишний шум в diff. Оставил
  до UI-5.
- **`app/main.py` startup event на scheduler нет, но `@app.on_event`
  всё ещё используется в exception_handler'ах** — это `on_event` для
  `LoginRequiredRedirect`, не scheduler-related, оставил.
- **Не запустил live-smoke на pre-prod.** В DoD пункта не было, но
  было бы хорошо проверить через Railway, что после UI-4.5 портал
  стартует на pre-prod и `cbr_fetch`-job'ы регистрируются. Это
  собственник может проверить сам.
- **Operational instruction для офиса** — теперь когда UI-4.5
  закоммитится, на офисном сервере ОБЯЗАТЕЛЬНО нужен `git pull` ДО
  следующего тика Task Scheduler. Иначе следующий запуск
  `scripts/run_auctions_ingest.py` упадёт с `ModuleNotFoundError:
  No module named 'app.services.auctions'`. Зафиксировано в
  `docs/office-ingest-deploy.md` и в плане. Оркестратору — попросить
  собственника сделать pull в ближайший RDP-сеанс (это критично для
  непрерывности ingest'а).
- **Один большой коммит UI-4.5** мог бы быть разделён на «Перенос
  auctions» + «Перенос catalog» + «Перенос scheduler» — каждый из
  трёх логически независим. Но: каждое из этих изменений ломает
  репо в одиночку (например, перенос auctions без обновления
  импортов в scripts/ — сразу сломан ingest). Так что физически их
  разделение требовало бы пары промежуточных коммитов с
  частично-сломанным состоянием — что в master стоит избегать.
  Один атомарный коммит UI-4.5 — правильный выбор.

## 6. Открытые вопросы / Operational

1. **Офисный сервер: ОБЯЗАТЕЛЬНЫЙ `git pull` после UI-4.5 deploy.** На
   `D:\AuctionsIngest\ConfiguratorPC2\` (Windows Server 2012 R2, под
   `server1c\Администратор`) нужно открыть Git Bash и выполнить:
   ```bash
   cd /d/AuctionsIngest/ConfiguratorPC2
   GIT_SSH_COMMAND="ssh -F /d/AuctionsIngest/.ssh/config" git pull
   ```
   Если этого НЕ сделать ДО следующего тика Task Scheduler (раз в 2
   часа), следующий запуск `scripts/run_auctions_ingest.py` упадёт с
   `ModuleNotFoundError: No module named 'app.services.auctions'`.
   Sanity-check после pull:
   ```powershell
   .\venv\Scripts\python.exe -c "from portal.services.auctions.ingest.orchestrator import run_ingest_once; print('ok')"
   ```
   Должно вывести `ok`. Подробная процедура — в
   `docs/office-ingest-deploy.md → «Обновление кода на офисе после
   deploy»`.

2. **UI-5 разблокирован.** После UI-4.5 в `app/` остались только
   `main.py` (catch-all 301 + admin_router), `auth.py`, `config.py`,
   `database.py`, `templating.py`, `routers/admin_router.py`,
   `services/__init__.py`, `templates/`. Из них:
   - `app/main.py`, `app/templating.py`, `app/routers/admin_router.py`
     импортируют `from portal.services.configurator.*` (cross-import
     `app/ → portal/`, временный — для admin dashboard / budget /
     queries и хелперов to_rub/fmt_rub);
   - `app/auth.py`, `app/config.py`, `app/database.py` — re-export'ы
     shared/auth, shared/db плюс `Settings` dataclass.

   UI-5 удалит весь `app/` + Dockerfile + railway.json + Railway-сервис
   `configurator`. `to_rub`/`fmt_rub` нужно будет скопировать в
   `portal/templating.py`. `Settings` dataclass — либо переехать в
   `portal/config.py`, либо влить в `shared/config.py`.

3. **Шаблоны экспорта.** `kp_template.docx` и `project_template.xlsx`
   всё ещё в `app/templates/export/`. На UI-5 переедут в
   `portal/templates/export/` или
   `portal/services/configurator/export/templates/`. После переноса
   нужно будет поправить `parents[4]` → `parents[3]` или похожий
   глубинный путь в `excel_builder.py`/`kp_builder.py`.
