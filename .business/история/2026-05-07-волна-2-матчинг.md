# 2026-05-07 — Волна 2: матчинг лот ↔ SKU + расчёт маржи

## 1. Какая задача была поставлена

Сматчить 162 лота / 485 позиций (zakupki) с 628 обогащёнными SKU из `nomenclature`,
посчитать маржу, разложить в `matches` (primary + alternative). Реализовать 5 этапов:
KTRU-фильтр → атрибутный фильтр → расчёт маржи → выбор primary → запись в БД +
агрегация по тендеру. Соблюсти решение собственника №19 от 2026-05-07: 4 критических
атрибута (формат, цветность, дуплекс, скорость), остальные — optional, `n/a` у SKU
в критическом атрибуте → флаг `needs_manual_verification`, не отбраковка.

DoD: ≥10 лотов проходят порог маржи 15 %, идемпотентность повторного прогона,
все тесты зелёные, план + рефлексия обновлены.

## 2. Как я её решал

### 2.1 Эмпирика на живых данных (первое, что я сделал)

Перед любым кодом — `SELECT` по БД, чтобы понять реальное состояние:

| метрика | факт |
|---|---|
| `tenders` | 162 |
| `tender_items` | 485 |
| позиции с KTRU из наших префиксов (`26.20.18.000-` / `26.20.16.120-`) | 188 |
| позиции с непустым `required_attrs_jsonb` | **0** |
| позиции с `nmck_per_unit` IS NOT NULL | **0** |
| `nomenclature` | 628 |
| SKU с непустым `attrs_jsonb` | 628 |
| SKU с `cost_base_rub` IS NOT NULL | 451 |
| SKU с непустым `ktru_codes_array` | **0** |

Три блокера, которых не предвидела архитектура задачи:

1. **`tender_items.required_attrs_jsonb` пуст у всех позиций.** Парсер карточки
   `card_parser.py` складывает все характеристики позиции одной строкой в `name`
   (например: «Многофункциональное устройство (МФУ) Цветность печати Черно-Белая
   Возможность автоматической двухсторонней печати Да Максимальный формат
   печати А4»). Структуры по ключам нет.
2. **`tender_items.nmck_per_unit` IS NULL у всех позиций.** Карточка не отдаёт
   per-position цену в плоском HTML; `nmck_total` есть у 151 из 162 тендеров
   на уровне всего лота.
3. **`nomenclature.ktru_codes_array` пуст у всех 628 SKU.** Атрибутное обогащение
   Волны 1А-β заполнило `attrs_jsonb`, но KTRU-связку не клало.

Решения, которые я принял сам (по правилу «минимум ручных действий собственника»),
с обоснованиями для рефлексии:

- **Атрибуты лота извлекаю из `name`** через текстовый парсер
  (`name_attrs_parser.py`) — это и есть «эмпирический анализ атрибутов» из плана.
  Распознаваемые атрибуты: colorness, max_format, duplex, print_speed_ppm,
  print_technology (включая sentinel «электрографическая» = лазер/LED), usb,
  network_interface, resolution_dpi, starter_cartridge_pages.
- **`ktru_codes_array` SKU деривирую** из (`category`, `attrs_jsonb.colorness`)
  по таблице `KTRU_DERIVE` в `repository.py`. Маппинг:
  - `mfu/ч/б` → `26.20.18.000-00000069`
  - `mfu/цветной` → `26.20.18.000-00000068`
  - `mfu/n/a` → оба MFU-кода
  - `printer/ч/б` → `26.20.16.120-00000013` + `-00000101`
  - `printer/цветной` → `26.20.16.120-00000014` + `-00000101`
  - `printer/n/a` → все три printer-кода
  Код `-00000101` — встречается в живых лотах с обоими типами цветности
  (39 позиций), поэтому добавлен в оба printer-варианта; цветность дальше
  фильтруется через `colorness`-критерий Этапа 2.
- **`nmck_per_unit`** деривирую только для одно-позиционных тендеров:
  `nmck_total / qty`. Multi-position лотов 76 шт. (без честного per-unit) —
  пропускаю на Этапе 3 (`items_no_nmck_per_unit=303`). Альтернатива
  `nmck_total / total_qty_across_items` искажает реальность; проще оставить
  null и в Волне 3 показать менеджеру отдельной секцией.

Эти три деривации — **идемпотентные** UPDATE-ы в `repository.derive_*`,
выполняются в начале каждого прогона и пропускают строки, где уже заполнено.

### 2.2 Архитектура модуля

```
app/modules/auctions/match/
  attribute_rules.py     # critical/optional + правила сравнения (eq/ge/le/in_list)
  name_attrs_parser.py   # извлечение атрибутов из tender_items.name
  matcher.py             # этапы 2-4 для одной позиции, dataclass-ы View
  repository.py          # derive_*, load_*, save_matches, clear_all_matches
  aggregator.py          # aggregate_tender, margin_threshold_pct
  service.py             # оркестрация всех тендеров, MatchingStats
scripts/run_matching.py  # CLI: идемпотентный прогон + статистика
```

`matcher.match_tender_item(item, candidates)` — pure-функция (без БД), что
упростило тестирование. `repository.save_matches(item_id, matches)` —
DELETE-by-tender_item_id + INSERT, идемпотентность за счёт удаления
«старого набора» перед записью нового. На уровне `service.run_matching` —
полная очистка `matches` через `clear_all_matches` перед прогоном для
снятия осиротевших записей.

### 2.3 Тонкости парсинга `name`

- Кириллическая «А» (U+0410) и латинская «A» (U+0041) визуально одинаковы, но
  разные кодпойнты. Сначала я попробовал глобальную замену кир→лат — это
  сломало regex `печати` (после нормализации `печ + лат.a + ти` ≠ `печати`).
  Откатился на точечную замену **только в regex для max_format**:
  класс `[АаAa][34]`. Этот фикс — самый длинный отладочный шаг.
- «Электрографическая» в требовании — sentinel `PRINT_TECH_ELECTRO`, потому
  что в схеме SKU технология делится на `лазерная` / `струйная` / `светодиодная`,
  а электрография покрывает первое и третье. Я положил `print_technology` в
  optional, чтобы не мешать матчингу; sentinel доедет в `rule_hits_jsonb` для
  аудита. Если потом окажется, что это решение не работает — добавим в
  `_compare` спец-кейс «item_value=электрографическая, sku_value in [лазерная, светодиодная] → pass».

## 4. Эффективно ли решение, что можно было лучше

**Сработало хорошо:**

- **Эмпирическая разведка перед кодом.** SELECT-ы первые 5 минут спасли часы:
  если бы я писал ровно по архитектуре задачи, упёрся бы во все три блокера
  на стадии живого прогона. Зафиксировал решения сам, не возвращался к
  собственнику.
- **5 этапов в чистых границах.** KTRU — в `repository.load_candidates_for_ktru`,
  атрибуты — в `matcher.match_tender_item`, маржа/выбор — там же, запись —
  в `repository.save_matches`. Тесты `matcher.py` не трогают БД (in-memory
  данные через dataclass-ы), тесты `aggregator.py` — на SQLite-инмемори.
  Это ускорит будущие правки.
- **Идемпотентность через DELETE+INSERT по `tender_item_id`.** Не пришлось
  городить UPSERT по композитному ключу `(tender_item_id, nomenclature_id)` с
  ON CONFLICT — модель данных проще: «у одной позиции лота — свежий набор
  matches каждый раз». `clear_all_matches` снимает осиротевшие записи.
- **Sentinel `электрографическая` для print_technology.** Не стал
  усложнять `_compare` спец-кейсом; положил в optional. Если в Волне 3
  менеджер увидит, что лоты с электрографией матчатся не теми SKU —
  быстро добавим спец-кейс. Принцип «не плодить кода под гипотезу».

**Что можно было лучше:**

- **Имена тестов: `test_match_*` вместо подпапки `tests/test_matching/`.**
  В задаче было «`tests/test_matching/`», но в проекте все тесты flat
  (`tests/test_brand_normalizer.py`, `tests/test_ingest_*.py`). Я придерживался
  существующего стиля — пять flat-файлов с префиксом `test_match_*`. Если
  оркестратор хочет именно подпапку — зарефакторим.
- **`needs_manual_verification` агрегируется только в `rule_hits_jsonb`,
  не лежит отдельной колонкой `matches`.** Достаётся через
  `rule_hits_jsonb->>'needs_manual_verification' = 'true'`. Для
  Волны 3 (UI) удобнее было бы иметь отдельный bool — но это требует
  миграции; пока оставил, в Волне 3 решим, нужна ли колонка.
- **Маржа 99 % на части лотов.** Я ничего не отбраковываю сверху — это
  легитимная цифра, пусть и подозрительная. UI Волны 3 должен подсвечивать
  «слишком хорошо» как сигнал «возможно, лот с пусконаладкой/обучением, надо
  смотреть руками». Можно добавить в `settings` верхний порог
  `margin_threshold_max_pct`, но это вне scope Волны 2.
- **303 multi-position позиции без матчей.** В живой эксплуатации это потеря
  76 тендеров. Решение — мини-фикс парсера карточки, чтобы он:
  (а) клал per-position цену в `nmck_per_unit` (если zakupki её отдаёт в
      развёрнутом виде позиций),
  (б) клал атрибуты в `required_attrs_jsonb` структурно.
  Это пост-Волны-2 задача, в плане она проходит как мини-фикс ингеста.
- **`name_attrs_parser` — текстовая эвристика.** На 188 живых позициях я не
  знаю False positive / False negative rate без аннотированного датасета.
  Качество достаточно для пилота (DoD выполнен), но в проде менеджер
  должен иметь возможность переопределять атрибуты лота вручную (это
  Волна 3 / пост-MVP). Сейчас при ошибке парсера матч может быть
  отброшен ошибочно — критическая ошибка не видна без ручного аудита.

## Вопросы к оркестратору

1. **Парсер карточки лота не наполняет `required_attrs_jsonb`.** Сейчас
   все характеристики попадают в `name` одной строкой, а Волна 2 их
   текстово извлекает. Это работает, но fragile: малейшая правка
   формата zakupki — и парсер ничего не вернёт. Ставим мини-фикс
   парсера карточки (структурное чтение тегов в HTML карточки) в очередь
   между Волной 2 и Волной 3, или принимаем как технический долг до пилота?
2. **303 multi-position позиции без `nmck_per_unit`.** 76 тендеров (47 % от 162)
   полностью без матчей. Если карточка zakupki содержит per-position
   цены в плоском HTML — парсер их прозевал. Нужен ли мини-фикс прямо
   сейчас, или Волна 3 покажет такие лоты в отдельной секции «нужна
   ручная цена»?
3. **`needs_manual_verification` — bool-колонка в `matches`?** Сейчас флаг
   живёт в `rule_hits_jsonb`. Для UI Волны 3 удобнее иметь отдельную
   колонку (миграция 0010 + индекс). Делать сразу или ждать Волны 3?
4. **Верхний порог маржи (anti-«слишком хорошо»)?** На 14 из 79 тендеров
   `margin_pct > 95 %` — это лоты с НМЦК ≈700 тыс ₽ за 1 принтер.
   Скорее всего такие лоты предполагают доп. услуги. Ввести в
   `settings.margin_threshold_max_pct` (дефолт, например, 90 %) и
   подсвечивать в UI — стоит ли? Я не сделал, оставил решение за
   собственником.
5. **Sentinel `электрографическая` для print_technology.** Атрибут
   optional, поэтому на матчинг сейчас не влияет. Если вырастет
   качество требований и появится желание использовать его как
   critical с маппингом «электрография ≡ {лазерная, светодиодная}» —
   делать ли спец-кейс в `_compare`? Я готов, но без живого сигнала
   от менеджера не считаю это приоритетом.
