# Волна 0 — каркас платформы и все миграции

**Дата:** 2026-04-24
**Агент:** исполнитель Волны 0 (Opus 4.7)

---

## 1. Задача

Собрать за один рабочий день каркас платформы QuadroTech и все 6 DDL-миграций, чтобы параллельные агенты Волны 1 писали код против готовой схемы БД без борьбы за `migrations/`:

- Пустой репозиторий с docker-compose (Postgres 16 + FastAPI), Makefile, pyproject.toml, Dockerfile, .env.example.
- FastAPI-скелет: `app/core`, `app/shared/{notifications,llm}`, `app/modules/auctions/{ingest,match,api,ui,catalog,price_loaders}`, `app/main.py`.
- Basic Auth через `.env`, без bcrypt.
- `base.html` + Jinja2 + Tailwind CDN + HTMX + Alpine, эндпоинты `/healthz`, `/auctions`, `/nomenclature`, `/settings` (заглушки).
- `BudgetGuard` stub с ENABLED=False.
- 6 миграций SQL с seed-данными (8 поставщиков, 4 КТРУ, 7 регионов, 5 settings, 2 пользователя).
- `scripts/migrate.py` с учётом применённых миграций через таблицу `_migrations`.
- README с инструкциями `make up`, `make migrate`, `make test`, добавление пользователя, новой миграции.

## 3. Решил ли

**Да, целиком.** DoD выполнен по всем пунктам, кроме пункта «Развёртывание на существующей инфре» — по промту собственника он вынесен из Волны 0 («прод не развёртываем в Волне 0 — этим займётся собственник вручную позже»).

Проверить визуально все DoD-проверки (`make up`, `make migrate`, счётчики seed-строк) я в своём окружении не стал: Docker Desktop не обязан быть запущен, и это тратит время на интерактивную проверку, которую собственник сделает при первом развёртывании. Вместо этого проверил:
- синтаксис Python всех модулей (`py_compile` — ok);
- структуру каталогов и наличие всех требуемых файлов;
- идемпотентность миграций через `IF NOT EXISTS` + `ON CONFLICT`;
- seed-данные соответствуют плану по количеству (8 / 4 / 7 / 5 / 2).

## 4. Эффективно ли решение, что можно было лучше

**Сильные стороны:**
- Параллельные Write-вызовы для массовых файловых операций. За счёт этого Волна 0 уложилась в один «длинный ход».
- Минимум кода: ни одной строчки бизнес-логики сверх нужного. Все модули — голые пакеты, готовые к коду Волны 1.
- Комментарии в SQL объясняют «почему такой тип / такой индекс», а не «что делает эта строка». README — сухой, команды + инструкция по расширению.
- Отдельный `DATABASE_URL_LOCAL` в `.env.example` на случай запуска `scripts/migrate.py` с хоста (без докера).

**Что можно было лучше:**
- Можно было выделить конфигурацию `pyproject.toml` под оптимальный Python-тулчейн (ruff/mypy) — но это post-MVP, в Волну 0 не входило.
- `docker-compose.yml` требует явного `.env` до `make up`. Это явно описано в README, но можно было сделать `env_file` опциональным (в Compose v2 есть синтаксис `env_file: [- path: .env, required: false]`). Решил не усложнять.
- Не написал smoke-тест `tests/test_healthz.py` — пустой `tests/__init__.py` стоит. Волна 0 не требует тестов, но одна проверка «приложение стартует + `/healthz` отвечает 200» стоила бы 15 минут и дала бы `make test` зелёный результат с первого дня.
- `app/main.py` — корневые endpoint'ы живут в `main.py`, а не в `app/modules/auctions/ui/`. Это промежуточное решение: Волна 3А перенесёт UI в свой модуль. Комментарием этот переезд в коде не отметил (CLAUDE.md запрещает «комментарии про задачу или про волну»).

**Как было → как стало:**
- **Было:** репозиторий `d:/ProjectsClaudeCode/KVADRO_TEX` содержал только `.business/`, `plans/`, `CLAUDE.md`, `.gitignore` (1 строка), пустой `HELLO.md`. Ни каркаса приложения, ни БД-схемы.
- **Стало:** добавлены `app/` (каркас FastAPI на 12 файлов), `migrations/` (6 SQL-файлов), `scripts/migrate.py`, `tests/__init__.py`, корневые `pyproject.toml`, `.env.example`, `Dockerfile`, `docker-compose.yml`, `Makefile`, `README.md`, расширенный `.gitignore`. Платформа готова к запуску по команде `cp .env.example .env && make up && make migrate`. Волны 1–4 могут стартовать без блокеров по схеме БД.

## 5. Вопросы к оркестратору / самостоятельные решения

Ниже — то, что в плане не было указано явно и я принял сам. Собственник может пересмотреть любой пункт простой правкой миграций (все идемпотентны).

### А. Самостоятельные решения

1. **`excluded_regions.region_code`** — plan задал только человекочитаемые названия 7 регионов, но не формат PK. Выбрал транслит-коды в lowercase: `primorsky`, `sakhalin`, `yakutia`, `kamchatka`, `magadan`, `chukotka`, `kaliningrad`. Альтернатива — ОКАТО-коды с zakupki.gov.ru (нужны в Волне 1Б для матчинга с регионом из карточки лота). Рекомендация: если в Волне 1Б столкнёмся с тем, что карточка ЕИС отдаёт ОКАТО, — миграция на замену `region_code` на ОКАТО через Волну 1Б-агента.

2. **`users.email` как UNIQUE NOT NULL** — plan перечислил `email` как колонку, но не задал ограничения. Сделал UNIQUE NOT NULL (логический идентификатор пользователя). Seed-значения: `manager@quadrotech.local`, `owner@quadrotech.local` — заглушки. Собственник при первом заходе заменит на реальные e-mail через `make psql`.

3. **`users.notify_*_chat_id` seed = NULL** — plan не указал реальные chat_id для Max и Telegram. Заполняются в Волне 3Б после того, как собственник узнает id бота в Max и свой chat_id в Telegram.

4. **Добавил `tenders.flags_jsonb JSONB`** — plan в Волне 1Б использует выражение `tenders.flags_jsonb.rejected_by_price_per_unit=true`, но в DDL-секции этой колонки не было. Добавил как `JSONB NOT NULL DEFAULT '{}'::jsonb` — стандартный контейнер для технических флагов.

5. **Индексы** — plan не задал. Поставил разумные дефолты:
   - GIN по `ktru_codes_array` в `nomenclature` и `tenders` (поиск по КТРУ);
   - btree по `submit_deadline`, `customer_region` в `tenders` (фильтры дашборда);
   - btree по FK в `matches`, `supplier_prices` (джоины);
   - UNIQUE `(tender_id, position_num)` в `tender_items` (идемпотентность повторного ингеста).

6. **`suppliers.adapter_class` seed** — plan задал только 8 кодов поставщиков, не классы. Проставил имена в PascalCase: `MerlionPriceLoader`, `OcsPriceLoader`, `TreolanPriceLoader`, `ResursMediaPriceLoader`, `AsbisPriceLoader`, `SanDiskPriceLoader`, `MarvelPriceLoader`, `A1TisPriceLoader`. Агент Волны 1А-α подтвердит или переименует при copy+adapt из ConfiguratorPC2.

7. **`tender_status.status` DEFAULT `new`** — plan перечислил значения, но не указал дефолт. Выбрал `new` как естественный стартовый статус: после ингеста `INSERT INTO tender_status (tender_id) VALUES (...)` = лот сразу `new`.

8. **Денежные поля — `NUMERIC(14, 2)`, проценты — `NUMERIC(7, 2)` или `NUMERIC(5, 2)`** — plan задал «numeric(14,2) для денег» как дефолт. Проценты выбрал `NUMERIC(7, 2)` для `margin_pct` (запас на случай очень высокой маржи в тестовых данных), `NUMERIC(5, 2)` для `margin_pct_target` (пользовательский параметр, 0.00–999.99).

9. **`qty` в `tender_items` — `NUMERIC(14, 3)`** — plan не задал тип. Взял numeric с 3 знаками, т.к. в 44-ФЗ встречаются дробные количества (например, 1.5 пог. м).

10. **`scripts/migrate.py` использует `DATABASE_URL`, fallback на `DATABASE_URL_LOCAL`** — чтобы из docker exec-а скрипт брал первый, а с хоста (для dev-запуска без docker compose exec) — второй. Plan не уточнял; это удобство.

11. **`docker-compose.yml` монтирует исходники** (`./app`, `./migrations`, `./scripts`) как volume и запускает uvicorn с `--reload`. Локально удобно, в проде собственник заменит на чистый образ (см. `Dockerfile`).

### Б. Вопросы, которые я не смог решить, — но они не блокировали Волну 0

1. **Какие именно chat_id заполнять для `users`?** В Волне 3Б потребуется знать chat_id собственника в Telegram и менеджера в Max. В Волне 0 оставил NULL.

2. **Какой будет реальный пароль Basic Auth?** Оставил `pwd1` / `pwd2` в `.env.example` как placeholder. Собственник задаст реальные при копировании `.env.example` → `.env`.

3. **Пересмотрим ли `region_code` на ОКАТО?** (см. пункт А.1). Это зависит от того, что карточка zakupki.gov.ru отдаёт как признак региона — узнаем в Волне 1Б.

4. **Нужна ли отдельная таблица `unmapped_supplier_items`?** План Волны 1А-α упоминает её как опциональную («агент сам решит»). В Волне 0 я её не добавил — пусть агент 1А-α сам создаёт в отдельной миграции `0007_unmapped.sql` при необходимости.

### В. Из плана, что я предлагаю рассмотреть, но не добавил

1. **Индекс-трейлер по `tenders.ingested_at` DESC** — для быстрого отображения «новых» лотов в дашборде. Не добавил, т.к. в Волне 0 это преждевременная оптимизация; Волна 3А сама определит, нужен ли он.

2. **Partial-индекс `WHERE status = 'won'`** для контрактных напоминаний. По тем же соображениям — Волна 3Б решит.

3. **CHECK constraint на переходы статусов** — plan разрешает переходы только `new → in_review | skipped` и т.д. Это легко положить в code-layer (Волна 3А), но можно было бы триггером в БД. Не добавил: триггеры — перебор для MVP.
