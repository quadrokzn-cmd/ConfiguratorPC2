# 2026-05-07 — Диагностика аномалий Волны 2

## Что проверяли

После прогона матчинга Волны 2 собственник увидел две аномалии:
1. **Маржа медиана 89 %, max 99 %** — слишком хорошо. Гипотеза: «парсер не учитывает количество штук одной позиции и берёт сумму вместо цены за единицу».
2. **303 multi-position позиции без `nmck_per_unit`** (76 тендеров, 47 %) — подозрение, что zakupki отдаёт per-position цену, но парсер её не вытащил.

Парсер карточки **не трогали** — только диагностика. Третья мелочь («электрография ≡ лазерная») зафиксирована в коде.

## Часть A — Маржа 99 %: гипотеза собственника НЕ подтверждается

Топ-5 тендеров с самой высокой маржой (по `matches.match_type='primary'`):

**Колонка «per-unit ₽ (zakupki)» = то, что показывает сама карточка.** Для всех пяти случаев `nmck_per_unit_наш == nmck_total/qty == per_unit_на_zakupki`. То есть **деривация цены за единицу математически верна** — гипотеза «складываем сумму вместо цены за единицу» опровергнута.

### Реальная причина аномальной маржи

Цены 600–833 тыс ₽ за 1 МФУ — это **легитимные цены лотов на промышленную/A3-технику**, а матчинг подобрал бюджетные SOHO-модели Pantum/HP за 4–12 тыс ₽. Корень — name-parser **не извлекает критические атрибуты из обрезанного/деформированного name**.

Что в `matches.rule_hits_jsonb->checks` для каждого primary:

| reg_number | rule_checks (что вообще проверилось) |
|---|---|
| 0358200055826000035 | **пусто** — name-parser ничего не извлёк |
| 0373100031126000016 | только `print_speed_ppm`: SKU=7, item=**3** (sic!) — regex поймал «А3» в «формате А3» как «3 ppm» |
| 0119200000126008104 | пусто |
| 0358200055826000034 | только `duplex` (yes-yes); `max_format=A3` не извлекли — фраза «возможность сканирования в форматах A3», парсер ищет «формат [печати] A3» |
| 0122300006326000057 | только `duplex` (n/a с manual-флагом) |

**То есть атрибутный фильтр пропускает почти всё**, и primary становится самый дешёвый SKU в KTRU-кандидатах. KTRU-фильтр сам по себе слишком грубый: он пропускает SOHO-картриджный струйник на лот за 685К ₽ под промышленный A3 МФУ.

### Дополнительные находки

- **HP:7WN42B классифицирован как `category='mfu'`** в каталоге, хотя похоже на код картриджа (HP DeskJet 2320 имеет MPN 7WN42B = это сама модель). Сам по себе это не баг — модель действительно МФУ. Но cost_base 3960 ₽ + max_format A4 + colorness=цветной → этот SKU должен в принципе выпадать из KTRU 26.20.18.000-00000068, на который шли промышленные лоты.
- **Регэкс `print_speed_ppm`** в `name_attrs_parser.py` ловит «А3» как число 3: pattern `[^0-9]{0,40}?(\d+)` хватается за первую цифру; в имени «Скорость цветной печати в формате А3, стр/мин ≥ 20» цифра «3» из «А3» ближе к слову «печати», чем «20». Это самостоятельный баг, не связанный с гипотезой собственника.

## Часть B — multi-position: zakupki **отдаёт** per-position цены

Проверил 3 multi-position тендера. Везде в карточке цены за единицу есть отдельно по каждой позиции:

- Клавиатура × 14 шт по 971 ₽ = 13 594 ₽
- Системный блок × 4 шт по 49 227,25 ₽ = 196 909 ₽
- IP видеорегистратор × 1 шт по 6 509,75 ₽
- Монитор × 4 шт по 8 837,25 ₽
- МФУ A3 × 1 шт по 162 069,50 ₽
- МФУ A4 × 2 шт по 23 962,25 ₽
- Принтер × 1 шт по 12 762,25 ₽
- Web-камера × 1 шт по 1 987,25 ₽
- Батарея аккумуляторная × 10 шт по 1 992,50 ₽
- ИБП

В нашей БД у всех 13 позиций `nmck_per_unit IS NULL`.

### [0107300018926000042](https://zakupki.gov.ru/epz/order/notice/zk20/view/common-info.html?regNumber=0107300018926000042) — 12 позиций, НМЦК 4 788 308,96 ₽
Аналогично — per-unit отдаётся в карточке (системник 90 533,33 ₽, интерактивная панель 240 500 ₽, ноутбук 75 358,33 ₽, принтер 23 633,33 ₽, МФУ 33 700 ₽ и т. д.).

### [0317100032926000169](https://zakupki.gov.ru/epz/order/notice/ea20/view/common-info.html?regNumber=0317100032926000169) — 12 позиций, НМЦК 3 371 417,90 ₽
Аналогично — для каждой строки указана цена за единицу.

### Вывод

**Да, zakupki физически отдаёт per-position цены в карточке для multi-position лотов.** Наш `card_parser._parse_items()` их не извлекает по двум возможным причинам:
1. цены лежат в отдельном блоке «Расчет НМЦК / Расчёт стоимости» вне основной таблицы позиций (нужно проверить на raw_html);
2. в текущем `_parse_items()` колонка «Цена за единицу» ищется внутри той же `<tr>`, где сама позиция; для multi-position лотов структура другая (раздельная таблица).

Поскольку задача — диагностика, а не фикс парсера, рекомендация ниже.

## Часть C — «электрография ≡ лазерная ≡ светодиодная»: зафиксировано в коде

- Добавлена таблица групп эквивалентности `_EQUIVALENCE_GROUPS` для атрибутов, у которых разные строковые значения семантически равны:

## Рекомендации оркестратору

В порядке приоритета (по влиянию на качество матчинга):

### 1. Фикс name-parser: print_speed_ppm путает «А3» с «3 ppm» — БЛОКЕР

**Симптом:** на тендере 0373100031126000016 «Скорость цветной печати в формате А3, стр/мин ≥ 20» — парсер вернул `print_speed_ppm=3` вместо 20. SKU с `print_speed_ppm=7` прошёл фильтр.

**Минимальный фикс:** в regex `parse_print_speed_ppm` добавить требование, что захваченное число НЕ должно стоять сразу после «А»/«A». Один промт, файл `name_attrs_parser.py`, +1 тест.

### 2. Расширить name-parser: max_format не извлекается из «возможность сканирования в форматах A3» / «формате А3»

**Симптом:** на тендерах `0358200055826000034` и `0373100031126000016` критерий `max_format=A3` не сработал, хотя в имени явно «А3». Сейчас regex ищет конкретно «формат [печати] A3» в начале фразы.

**Минимальный фикс:** добавить альтернативные паттерны: «в формате А3», «формат сканирования А3», «поддержка А3». Один промт, +2-3 теста.

### 3. Multi-position парсер карточки: 47 % тендеров упускаем

**Симптом:** zakupki отдаёт per-position цены в карточке, но `card_parser._parse_items()` их не находит. 76 тендеров (303 позиции) полностью без матчей.

**Что нужно:** реверс-инжиниринг HTML-блока «Расчет НМЦК / расчёт стоимости» в карточке закупки, добавление парсинга в `card_parser.py` с тестом на сохранённом raw_html. Это 1-2 часа работы (новый промт), но снимает крупнейшую дыру в покрытии.

### 4. Сразу после фиксов 1-3: перепрогон матчинга, ревью маржи

После фиксов name-parser (1, 2) ожидаю — top-маржа упадёт до реалистичных 30–60 % на тех же 79 тендерах: критические атрибуты A3/format начнут работать, бюджетные SOHO-SKU отвалятся. Multi-position fix (3) даст +50–70 новых тендеров с матчами.

### 5. (опционально) Анти-«слишком хорошо»: верхний порог маржи в `settings`

После фиксов 1-3 + перепрогона часть лотов всё равно может давать `margin_pct > 90 %` (настоящие промышленные лоты с услугой/обучением, которая не указана отдельной позицией). Можно ввести в `settings` параметр `margin_threshold_max_pct` (например, 90 %): тендеры выше — в отдельную секцию UI «нужна ручная проверка». Это полу-час кода + UI-флажок в Волне 3.

### 6. (вне диагностики) Каталог: SKU `hp:7WN42B` — точно ли это HP DeskJet 2320?

В матчинге primary дважды (топ-1 и топ-2) выбран `hp:7WN42B` как «Струйное МФУ/ HP DeskJet 2320» за 3960 ₽. Стоит верифицировать обогащение этого SKU — выглядит дёшево даже для бюджетного струйника. Возможно, при обогащении мог попасть код картриджа под видом модели. Это вопрос к domain-эксперту по аукционам (собственнику-1) — потенциально 5-минутная проверка.

---

**Итого** — гипотеза собственника неверна, причина аномальной маржи в слабом name-parser. Multi-position тендеры можно подобрать целиком, если научить парсер карточки извлекать per-position цены. Электрография ≡ лазерная ≡ светодиодная зафиксирована, тесты зелёные.
