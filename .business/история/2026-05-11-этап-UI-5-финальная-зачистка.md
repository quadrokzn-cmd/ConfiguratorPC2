# Этап UI-5: финальная зачистка app/

**Дата:** 2026-05-11
**Этап Пути B:** 5 из 5 (последний)
**Контекст плана:** `plans/2026-05-11-ui-merge-portal-configurator.md`
**Предыдущий этап:** UI-4.5 — перенос auctions/catalog/scheduler из app/
в portal/ (коммит 3bdf3a8).

## 1. Поставленная задача

Полностью удалить папку `app/` из репозитория и перевести проект на
одно FastAPI-приложение (`portal/main.py`). Это завершающий этап Пути B —
после него `config.quadro.tatar` упраздняется как поддомен, Railway-сервис
`configurator` удаляется собственником вручную, DNS-записи в Reg.ru
снимаются.

Решение собственника 2026-05-11: **Вариант A (полное удаление)**.
Никаких 301-редиректов со старого хоста сохранять не нужно — менеджеры
ещё не работают в сервисе, закладок ни у кого нет. URL admin-страниц
(`/admin`, `/admin/budget`, `/admin/queries`) сохранить в портале без
редиректов (они никому не известны снаружи, но идиоматически правильно
держать их под `/admin/*`).

Состав работ:
1. Перенести `Settings` dataclass из `app/config.py` в `shared/config.py`.
2. Перенести живой код `to_rub`/`fmt_rub`/`current_exchange_rate`/
   `static_url` из `app/templating.py` в `portal/templating.py` (раньше
   были re-export'ы через `from app.templating import …`).
3. Перенести 3 admin-страницы (dashboard, budget, all_queries) в
   `portal/routers/admin.py` + `portal/templates/admin/*.html`.
4. Перенести Word/Excel-шаблоны экспорта в `portal/templates/export/`.
5. Заменить ~40 импортов `from app.{config,database,auth,templating}` на
   `shared.*`/`portal.*` по всему репо.
6. `git rm -rf app/`, удалить `Dockerfile`, `railway.json`, `Procfile`.
7. Почистить `tests/test_web/`, обновить документацию (CLAUDE.md, README,
   docs/), отметить UI-5 как `[x]` в плане.
8. pytest регрессия 0 failed.

## 2. Как решал

1. **Ревизия импортов.** `grep '^(from|import)\s+app(\.|\s)' --include=*.py`
   нашёл 55 файлов (помимо самой app/). Это были не «забытые после
   UI-4.5», а ровно тот долг, который был зафиксирован в рефлексии UI-4.5
   («что осталось собственнику сделать в UI-5»). Я задал оркестратору
   уточняющий вопрос на старте — где жить `Settings`, какой объём
   миграции, нужны ли редиректы. Собственник подтвердил: полный объём,
   Settings в `shared/config.py`, без редиректов. Это было правильное
   решение задать вопрос — без него я бы либо собрал слишком маленький
   диф (только удаление папки, без миграции), либо излишний (с
   compatibility-shim'ами «на всякий»).

2. **shared/config.py.** Скопировал полностью из `app/config.py`,
   выкинув dead-field `run_scheduler`/`RUN_SCHEDULER` (после UI-4.5 не
   читался нигде — UI-4.5 рефлексия это упоминала как «удалю в UI-5»).
   Обновил `shared/db.py`: `from app.config` → `from shared.config`.
   Это устранило последний `shared/ ← app/` импорт.

3. **portal/templating.py.** Раньше там был `from app.templating import
   to_rub as _to_rub, fmt_rub as _fmt_rub` + `from app.templating import
   current_exchange_rate as _current_exchange_rate`. Я перенёс всю
   реализацию (4 функции + `_ENG_NOISE_PATTERNS` + хелперы
   `_safe_get_current_rate`) внутрь portal/templating.py. Импорт
   `shared.db.SessionLocal` и `portal.services.configurator.export.exchange_rate`
   сделал **ленивым внутри функции** — иначе на старте модуля шаблонов
   подтягивается тяжёлый chain зависимостей (SQLAlchemy engine, openai,
   etc.), что замедляет импорт `portal.main` в тестах. UI-4 рефлексия
   объясняла, что аналогичный приём (lazy imports) уже применялся в
   `portal/scheduler.py::_job_fetch_cbr()` — здесь та же логика.

4. **portal/routers/admin.py.** Создал по образцу старого
   `app/routers/admin_router.py`, но:
   - убрал `dashboard_legacy_alias` для `/admin/dashboard` — мёртвая
     ссылка, никаких внешних потребителей этого URL не было.
   - убрал `users_redirect_to_portal` (это был cross-service редирект
     `/admin/users` → `${portal_url}/settings/users`; в портале уже есть
     point-by-point handler для `/admin/users` → `/settings/users`,
     прямо в `portal/main.py` с UI-3).
   - переписал импорты на `shared.auth`, `shared.db`, `portal.templating`.
   - оставил inline-COUNT для `mapping_count` (как было в app/, чтобы
     не плодить новые service-функции; функция нужна только этой одной
     странице).
   Router имеет `prefix="/admin"` и routes `""`, `"/budget"`, `"/queries"`.

5. **Перенос шаблонов через git mv.** `git mv app/templates/admin/*.html
   portal/templates/admin/` и `git mv app/templates/export/{kp,project}_*
   portal/templates/export/`. Git сохранил историю rename'ов
   (`git log --follow portal/templates/admin/dashboard.html` теперь
   показывает всю историю с момента создания файла в app/templates/).
   Затем внутри перенесённых HTML поправил два URL-паттерна (старые
   ссылки в app/ были `/admin/mapping` и `/query/{id}`, после переездов
   UI-2 и UI-4 правильные — `/databases/mapping` и `/configurator/query/{id}`).

6. **parents[3] vs parents[4].** В `excel_builder.py` и `kp_builder.py`
   `_TEMPLATE_PATH = parents[4] / "app" / "templates" / "export" / "X"`.
   После переезда шаблона: parents[3] от файла `portal/services/
   configurator/export/X.py` = `portal/` (3 уровня вверх:
   `export → configurator → services → portal`). Оттуда `templates/
   export/X` напрямую. UI-4 рефлексия предсказывала «parents[4] →
   parents[3]» — совпало в точности.

7. **Подключение admin router в portal/main.py.** Важно: подключить
   `admin.router` ДО точечных catch-all 301-handler'ов `/admin/users`,
   `/admin/backups`, `/admin/audit` (которые задают редиректы на
   `/settings/*` ещё с UI-3). Иначе FastAPI выберет point-by-point
   `/admin/users` обработчик раньше, чем общий `/admin` от admin.router
   — но это не страшно, потому что `/admin/users` и `/admin` — разные
   маршруты. Конфликта быть не могло, я подключил admin.router раньше
   на всякий случай.

8. **Массовая замена импортов.** Написал одноразовый Python-скрипт
   (как в UI-4.5): 4 regex'а, проход по всем `*.py` за пределами `app/`,
   `node_modules/`, `.venv/`, `.git/`, `__pycache__/`. За один прогон —
   54 файла обновлено. Это в 5-10 раз быстрее, чем последовательность
   `Edit`-tool вызовов, и без рисков опечататься (regex точечный, нет
   ложноположительных матчей в строках или комментариях).

9. **Удаление app/.** `git rm -rf app/` — `-f` обязателен, потому что
   два редирект-теста (`test_configurator_redirects.py`, `test_databases_redirects.py`)
   к тому моменту были модифицированы regex'ом (там были `from app.config
   import settings` → `from shared.config import settings`). Git
   справедливо предупредил, что в файлах есть несохранённые
   модификации; флаг `-f` сказал «всё равно удаляй». Тоже самое для
   `Procfile` (хотя я не модифицировал его, видимо из-за CRLF на

      Write tool заменил NBSP-литерал в `int_grouped = "\xa0".join(chunks)`

      `tests/test_portal/` через ещё один `git mv` — там есть autouse
      `_clean_tables` в `test_portal/conftest.py`, который TRUNCATE'ит
      ровно те таблицы, которые тесты используют (`users`, `projects`,
      `queries`, `specification_items`).

## 4. Эффективно или нет, что можно было лучше

**Что прошло хорошо:**

- **Уточняющий вопрос на старте.** Прочитав scope в промте от
  оркестратора, я обнаружил, что 55 файлов содержат `from app.*` — что
  оркестратор охарактеризовал как «забытые импорты, сообщи». Но из
  рефлексии UI-4.5 было очевидно, что это известный долг UI-5, который
  оркестратор просто не зафиксировал в промте детально. AskUserQuestion
  с тремя опциями (полный объём / Settings в portal-config / без
  миграции) — собственник выбрал полный объём, и мы избежали ситуации,
  когда я бы либо сделал слишком мало (только удалить app/), либо
  слишком много (compatibility shims «на всякий случай»). Без вопроса
  риск был — и потерять время на переделку, и сделать не то.

- **Массовая regex-замена через скрипт.** 54 файла за один проход. Если
  бы делал последовательностью `Edit` вызовов — потратил бы в 5-10 раз
  больше времени, и неизбежно где-то ошибся бы. Скрипт прост (4
  регулярки), идемпотентен (запуск второй раз — 0 изменений), легко
  верифицируется (после прогона — grep пуст).

- **Ленивые импорты в portal/templating.py.** `_safe_get_current_rate()`
  не делает `from shared.db import SessionLocal` на уровне модуля.
  В UI-4 портал-templating страдал от cross-import'а с app/templating —
  при тестах подгружалась тяжёлая chain зависимостей. После UI-5 с
  lazy imports эта проблема не вернулась.

- **git mv с follow-history.** Все 5 шаблонов перенесены через `git mv`,
  не через `cp + git rm + git add`. `git log --follow` теперь
  показывает полную историю каждого файла от самого создания, что
  важно для blame и debugging.

- **Минимально-инвазивные правки в perенесённых шаблонах.** В
  `dashboard.html` и `all_queries.html` исправил только URL-ы
  `/admin/mapping` → `/databases/mapping` и `/query/{id}` →
  `/configurator/query/{id}`. Не «причёсывал», не переименовывал
  переменные, не оптимизировал CSS — никаких лишних изменений в diff'е.

- **Удаление dead-кода (`scripts/smoke_stage_6_2.py`).** Скрипт-привидение
  из этапа 6.2, никому не нужен, импортировал `app.main`. Удалил без
  жалости. Если что-то такое в будущем понадобится — лучше как
  pytest-тест, не как одноразовый CLI-скрипт.

**Что можно улучшить:**

- **Имя `portal/routers/admin.py` — slightly tight namespace.** В
  `portal/routers/` уже есть `admin_auctions.py`, `admin_auto_price.py`,
  `admin_diagnostics.py`, `admin_price_uploads.py` — 4 «админ-чего-то».
  Новый `admin.py` без суффикса для admin-обзора конфигуратора может
  визуально путаться. Альтернативы: `admin_configurator.py` или
  `admin_dashboard.py`. Но `admin.py` короче, и его URL'ы — это
  «общий админ-обзор» (не привязан к конкретному модулю как auctions/
  diagnostics/etc.). Оставил `admin.py`. Если будет путаница — можно
  переименовать на любом следующем этапе одним коммитом.

- **`Settings` оказался ~50 полей и переехал as-is.** Если бы делал
  отдельный refactor — разбил бы на `DatabaseSettings`/`AuthSettings`/
  `OpenAISettings`/`SMTPSettings`/`SchedulerSettings`. Но: каждый
  потребитель `settings.X` ожидает плоский namespace. Разбиение
  требует параллельной правки 60+ потребителей, что выходит за scope
  UI-5. Оставил плоско, как было.

- **Стейл-комменты в коде, ссылавшиеся на `app.*`.** Прошёлся по 6
  файлам (`portal/routers/admin_price_uploads.py`,
  `portal/services/configurator/price_loader.py`, 4 скрипта) и поправил
  активно-неверные комментарии. Чисто-исторические упоминания типа
  «раньше было через app.templating —» оставил как контекст. Это
  субъективная граница — будущий читатель может счесть, что я не до
  конца почистил. Но переписывать всю историю в комментариях ради
  «чистоты» — это переплачивать.

- **Не запустил smoke на pre-prod.** В DoD пункта не было, но было бы
  хорошо проверить: после `git push` Railway пересоберёт portal,
  старт-логи покажут все cron-job'ы + ensure_initial_rate; затем
  посмотреть `/admin` на pre-prod — должно отдаваться нормально, не
  500. Это собственник сам сделает после деплоя.

- **Cleanup: `portal/templates/_macros/icons.html`.** После удаления
  `app/templates/_macros/icons.html` остался один источник иконок —
  но я не сравнивал, не остался ли неиспользуемые иконки. UI-1
  рефлексия описывала «выравнивание иконок между двумя файлами», теперь
  это не нужно. Не трогал — это микро-cleanup, который при появлении
  pre-prod smoke лучше делать отдельной задачей.

- **`portal/routers/admin.py` и тесты — мог бы использовать `mock_process_query`
  фикстуру, но не нужно.** В новом `tests/test_portal/test_admin.py`
  тестируется только GET-доступ к страницам и базовый рендер бюджета.
  Никаких POST'ов на админ-страницах нет, OpenAI mock не нужен.

- **Один большой коммит UI-5 vs три маленьких.** Этот коммит затрагивает
  ~80 файлов (переезд + замена импортов + удаление app/). Технически
  можно разбить: (а) shared/config.py + Settings refactor, (б) перенос
  admin-страниц и templating, (в) удаление app/. Но: каждый из этих
  трёх шагов в одиночку оставляет репо в полусломанном состоянии
  (например: после шага (а) `app/config.py` ещё существует и settings
  есть в двух местах — это противоречит solid). Атомарный коммит
  UI-5 — это правильно. Откатить можно через `git revert` (один
  коммит), что проще, чем серия из трёх.

## 6. Открытые вопросы / Operational

1. **Собственнику нужно сделать руками (вне Claude Code scope):**
   - В Railway UI: остановить и удалить сервис `configurator` (prod) и
     `configurator-preprod` (pre-prod). Двух service-удалений достаточно —
     никакой data preservation не нужно, эти сервисы только редиректили,
     БД у них общая с portal.
   - В Reg.ru: удалить DNS-записи `config.quadro.tatar` и
     `config-preprod.quadro.tatar`. Альтернатива — оставить как CNAME
     на портал (но это создаст «зомби-поддомен» без цели; чище удалить).
   - Опционально: после удаления записей сделать `dig config.quadro.tatar`
     с локалки — должно вернуть NXDOMAIN, подтверждая что DNS пропал.

2. **Settings field cleanup на будущее.** После UI-5 в `shared/config.py`
   осталось ~50 полей с разными префиксами (`smtp_*`, `admin_*`, `cookie_*`,
   etc.). Если будет рефактор — разбить на pydantic BaseSettings с
   разделами (`DatabaseSettings`, `OpenAISettings`, etc.). Это вне scope
   UI-5; обходится один длинный плоский класс не больно.

3. **Sanity-check на офисном сервере** — после UI-5 deploy на офисном
   `D:\AuctionsIngest\ConfiguratorPC2\` нужен `git pull` ДО следующего
   тика Task Scheduler (как было после UI-4.5). Опять — иначе
   `scripts/run_auctions_ingest.py` упадёт, но теперь с разными
   ошибками (нет `shared.config` импорта на старой версии). Процедура
   та же — `docs/office-ingest-deploy.md → «Обновление кода на офисе
   после deploy»`.

4. **`Dockerfile.portal` — единственный Dockerfile.** Имя осталось с
   суффиксом `.portal` для исторической различимости (когда были два
   Docker'а). Можно переименовать в просто `Dockerfile` одним коммитом
   на любом следующем шаге. То же про `railway.portal.json` → `railway.json`.
   Я не сделал в UI-5, чтобы не растягивать diff и не путать deploy-pipeline
   (Railway смотрит на specific path в `railway.portal.json`).

5. **`tests/test_portal/test_admin.py` — минимальное покрытие.** Тесты
   проверяют access (admin vs manager) и базовый рендер бюджета (warning
   80%, blocked 100%). Если в будущем admin-страницы расширят (например
   добавят управление пользователями, фильтры, экспорт) — тесты надо
   расширять. Сейчас покрытие соответствует объёму функционала (3
   GET-страницы).
