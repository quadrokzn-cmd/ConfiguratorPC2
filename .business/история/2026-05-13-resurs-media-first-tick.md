# 2026-05-13 — Мониторинг первого боевого тика Resurs Media (07:40 МСК) — НЕ СОСТОЯЛСЯ

## 1. Задача

Оркестратор поставил: «убедиться, что первый prod-тик автозагрузки Resurs
Media в 2026-05-13 07:40 МСК отработал успешно». Контекст: 2026-05-12
собственник получил боевой SOAP API от РМ, подключил Variables на Railway
prod, прогнал `scripts/resurs_media_bootstrap_catalog.py` (Фаза C — 70 501
позиция в `resurs_media_catalog`). Сегодня — день первого scheduled-тика.

Скриншот собственника с `/databases/auto-price-uploads` (на самом деле
дашборд-виджет «свежесть прайсов» на главной) показывал «Ресурс Медиа:
Старый, 30.04.2026 17:09» — что нужно было либо подтвердить как баг UI,
либо объяснить.

## 2. Как решал

Worktree `feature/resurs-media-monitor` от `origin/master`.

Прочитал runner/fetcher/scheduler/UI-роутер, чтобы понять контракт:
- `portal/scheduler.py::_make_auto_price_job` → `_is_supplier_enabled(slug)`
  читает `auto_price_loads.enabled` для конкретного slug. Если FALSE —
  log INFO «тумблер OFF, пропуск» и тихий выход. **Запись в
  `auto_price_load_runs` НЕ создаётся** (это полезно: тумблер OFF = «как
  будто cron не существует»).
- `portal/services/dashboard.py::_get_suppliers_freshness` → читает
  `MAX(uploaded_at) FROM price_uploads WHERE status IN ('success','partial')`
  по каждому имени поставщика. Это и есть источник «30.04.2026 17:09» —
  он не связан с auto_price_loads напрямую.

Подключение к prod-БД через `DATABASE_PUBLIC_URL` из
`.env.local.prod.v1`, чтение через `dotenv_values()` без печати DSN в
stdout (memory `feedback_env_grep_content`). Скрипт
`scripts/_diag_resurs_media_first_tick.py` — одноразовая диагностика,
не закоммичен.

Результаты (см. план, мини-этап 2026-05-13 Фаза D, детали):

1. `auto_price_loads.resurs_media.enabled=FALSE`, last_success_at=NULL.
2. `auto_price_load_runs` за 2026-05-12..2026-05-13 для resurs_media —
   **0 строк**.
3. Все run'ы resurs_media за всё время — 2 строки от 2026-05-08
   (manual experiments перед получением боевого API).
4. `audit_log` за весь май с упоминанием `resurs_media` — **0 строк**.
5. `resurs_media_catalog` — 70 501 row (Фаза C прошла корректно
   2026-05-12 19:05-19:07 МСК).
6. `resurs_media_notifications` — 0 row.
7. `price_uploads` для supplier_id=8 — 2 ручных XLSX-загрузки от
   30.04.2026 (т.е. ровно за 13 дней до сегодня).

## 3. Решил ли

**Частично.** Диагноз поставлен однозначно: первого боевого тика
2026-05-13 07:40 МСК **не было**, потому что `auto_price_loads.enabled
=FALSE`. Запись в журнал не появляется — cron-job делает
`_is_supplier_enabled('resurs_media') → False` и тихо возвращается. Это
**сценарий A** из категорий, заданных оркестратором: «Cron disabled в
settings — не включай сам, это бизнес-решение».

Сам фиксить тут нечего: код корректный, scheduler корректный (если бы
тумблер был ON — тик прошёл бы). Действие — на стороне собственника:
один клик в UI `/admin/auto-price-loads` для «Ресурс Медиа».

## 4. Эффективно ли решение, что можно было лучше

**Эффективно** (≈45 минут от worktree до диагноза):
- Один Python-скрипт, 7 SQL-запросов, одна итерация на правке столбцов
  схемы (`source_filename` → `source` → `filename`; `fetched_at`
  → `synced_at`).
- Никаких изменений на prod, никаких сетевых запросов к SOAP, никакого
  трогания секретов.
- Безопасность секретов соблюдена: `dotenv_values()` без печати, regex
  redact для подозрительных подстрок в error_message.

**Чего могло быть лучше:**

1. **Предусмотреть seed-state.** При seed-данных миграции `enabled=FALSE`
   для всех 6 поставщиков (резонно — пока fetcher не подключён, не
   надо его звать). Но семантика «вот fetcher подключён, креды
   заведены, bootstrap-каталог построен → почему тогда `enabled` всё ещё
   FALSE» нигде не подсвечена. На UI `/admin/auto-price-loads`
   у строки RM на скрине бейдж «Канал не подключён» — это, видимо,
   потому что `enabled=FALSE`, а не потому что fetcher отсутствует
   (`register_fetcher` его зарегистрировал). UI-формулировка
   «Канал не подключён» вводит в заблуждение, когда `enabled=FALSE`
   при существующем fetcher'е.
   
   **Идея на будущее (не делал, не входит в скоуп):** разделить
   состояния в UI — «канал не подключён» (нет fetcher'а) vs
   «канал выключен» (есть fetcher, но enabled=FALSE) — это два разных
   бейджа.

2. **Cron мог бы оставлять след в `auto_price_load_runs`** со
   status='disabled', чтобы факт «scheduler пытался, но тумблер OFF»
   был виден в журнале. Сейчас он `logger.info(...)` уходит в
   Railway Deploy Logs (root WARNING обрезает — memory
   `reference_railway_deploy_logs`), и понять «cron сработал и нашёл
   OFF» vs «cron не запустился» можно только через `print(..., flush=True)`
   или через посекундный анализ Railway-логов. **Не делал**: смена
   контракта (что считать «попыткой запуска») должна быть отдельным
   архитектурным решением, не побочным эффектом мониторинга.

3. **Сделал бы UI-предупреждение «канал подключён, тумблер OFF».** Сейчас
   ничто на UI не подсказывает, что забыли включить — RM просто стоит
   как «выключенный», но различия с «не подключённым» нет. Не входит в
   текущий мини-этап.

4. **Action item: после включения тумблера — обязательно проверить
   первый боевой тик руками** (UI «Запустить сейчас»). Если оставить до
   scheduled 07:40 МСК завтра — а там сюрприз (rate-limit, auth-ошибка
   с боевыми кредами, ещё что-то) — потеряем сутки. Ручной тик сегодня
   покажет ответ за минуту.

## 5. Как было и как стало

**Было:**
- `auto_price_loads.resurs_media.enabled=FALSE` (seed-состояние
  с 2026-05-08, никогда не меняли).
- В журнале — пустота. На дашборде «свежесть прайсов» — последняя
  ручная XLSX-загрузка от 30.04.2026.
- Загадка «отработал ли первый prod-тик» без явного ответа.

**Стало:**
- Однозначный диагноз: тика не было, потому что тумблер OFF.
- В плане `plans/2026-04-23-platforma-i-aukciony.md` зафиксирован
  мини-этап 2026-05-13 Фаза D с точными SQL-данными и action item для
  собственника.
- Никаких изменений на prod: код, БД, секреты — без правок.

**Что осталось собственнику** (промт оркестратора прямо просил
AskUserQuestion в сценарии A; здесь зафиксировано в рефлексии для
оркестратор-чата — пусть он решит, спрашивать собственника
интерактивно или просто скинуть как информацию):

1. `/admin/auto-price-loads` → тумблер ON для «Ресурс Медиа».
2. (Опционально) сразу же кнопка «Запустить сейчас» — проверить fetcher
   с боевыми кредами.
3. Завтра утром мониторинг scheduled-тика 2026-05-14 07:40 МСК
   повторить (новый чат).
