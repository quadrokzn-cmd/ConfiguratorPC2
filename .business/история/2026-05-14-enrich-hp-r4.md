# 2026-05-14/15 — HP enrichment r4 (officials only, 145 SKU) — закрытие волны round 3/4

## 1. Какая задача была поставлена

Завершить отложенный HP-пул из 140 n/a-marked SKU (+5 уже-claude_code SKU
имеющих starter=n/a), который round 3 не закрыл из-за двух инцидентов
2026-05-13/14 (methodology error «store listings как fallback» +
WebFetch hp.com 60-сек timeout + Anthropic rate-limit при retry).
Это **завершающий чат** enrichment-волны round 3/4 (Canon+Epson+Ricoh r4
закрыты сегодня же).

Промт собственника требовал: pre-flight WebFetch test обязательно,
эскалация если домены недоступны / PDF не парсится, при возможности —
3 параллельных subagent'а с whitelist + blacklist + source_url
mandatory + normalizer + batched UNNEST UPDATE.

### Этап 3. Эскалация → решение собственника

По букве r3-правил («WebSearch-snippet'ы без открытия страницы —
ЗАПРЕЩЕНО») это нештатная ситуация, требующая решения. Эскалировал
через `AskUserQuestion` с 4 опциями:

A. WebSearch+allowed_domains как primary (расширение методики).
B. Skip HP до восстановления WebFetch.
C. Офисный сервер + ручная CSV-подгрузка.
D. Hybrid WebSearch + WebFetch validate.

**Собственник выбрал Option A** с расширением методологии (memory
`feedback_only_official_sources.md` обновлён): «snippet OK, если
allowed_domains=["hp.com"] — источник гарантирован API-ограничением,
дух правила соблюдён». Уточнения:
- `urlparse.hostname` whitelist должен включать **все hp.com варианты**
  (`hp.com`, `www.hp.com`, `support.hp.com`, региональные субдомены,
  CDN `h\d+\.www\d+\.hp\.com`).
- Snippet parsing: WebSearch с `allowed_domains=["hp.com"]` по запросу
  `<модель> HP specifications`. Из snippet'ов извлекать spec'и; ключ не
  нашёлся → `n/a`.
- Normalizer (паттерн Epson r4) обязателен.
- Cap WebSearch ≤200 на subagent (вместо WebFetch).
- 3 subagent'а параллельно.
- Audit-trail `source_method: "websearch_snippet"` в done-JSON.

Параллельно собственник проверил Playwright: **support.hp.com → Akamai
Bot Manager блокирует на TLS**, www.hp.com работает. **Исключение
Playwright из стратегии** — только WebSearch+allowed_domains.

### Этап 5. 3 параллельных subagent'а

примеры (`"yes"`/`"no"` для duplex, `"ч/б"`/`"цветной"` для colorness,
`["LAN","WiFi"]` для network_interface, int для starter_cartridge_pages),
cap WebSearch ≤200, WebFetch ≤5 (только на www.hp.com при необходимости),
early termination при >70% blank snippet'ов первых 10 SKU.

### Этап 6. Main-thread валидация + normalizer

`_hp_r4_validate.py`: для каждого item:
1. **urlparse.hostname** → проверка против:
   - exact-set: `hp.com`, `www.hp.com`, `support.hp.com`,
     `developers.hp.com`, `store.hp.com`, `h10032.www1.hp.com`,
     `h20195.www2.hp.com`, `h20566.www2.hp.com`, `h41112.www4.hp.com`,
     `kaas.hpcloud.hp.com`, `h30434.www3.hp.com`.
   - suffix: `.hp.com`, `.hpcloud.hp.com`.
   - CDN regex: `^h\d+\.www\d+\.hp\.com$`.
2. **Blacklist substring** (28 substring'ов): `hp.ru`, retailers,
   маркетплейсы, форумы.
3. Если source_url null или невалидный — force ВСЕ 9 ключей в `"n/a"`.
4. **`normalize_attrs`** — HP-специфичные маппинги:
   - `duplex`: «automatic»/«auto»/«yes» → `yes`; «manual»/«no»/«simplex» → `no`.
   - `usb`: «usb 2»/«hi-speed»/«usb» → `yes`.
   - `colorness`: «mono»/«монохром» → `ч/б`; «color»/«цветн» → `цветной`.
   - `print_technology`: «laser»/«electro-photographic» → `лазерная`;
     «inkjet»/«thermal»/«pagewide» → `струйная`; «led»/«светодиод» →
     `светодиодная`.
   - `max_format`: A3-substring → `A3`; A4/Letter/Legal → `A4`.
   - `resolution_dpi`: max int из строки `«1200 x 1200»` → 1200.
   - `network_interface`: список или строка с ethernet/wifi tokens →
     `["LAN"]`/`["WiFi"]`/`["LAN","WiFi"]`/`[]`.
   - `starter_cartridge_pages`: int или dict `{black: N, color: M}` →
     N (black yield).
5. **Re-validate через `schema.validate_attrs`** — при ошибке force в
   полный n/a.

**Результат валидации:** **145/145 valid, 0 forced_na_url, 0
forced_na_schema, 0 blacklist_hits — 100% покрытие.** Это лучше
Canon r4 (95.5%) и Epson r4 (96.8%), на уровне Ricoh r4 (100%).

### Этап 7. Apply на prod через прямой batched UNNEST UPDATE

**Архитектурное решение принято исполнителем** (по
`feedback_executor_no_architectural_questions`): apply через прямой SQL
UPDATE с полной перезаписью `attrs_jsonb`, **не через importer**. Та же
причина, что в Canon/Epson/Ricoh r4: `importer.merge_attrs` (Backlog #10)
защищает не-n/a от n/a-incoming. У HP 4 SKU с `print_speed_ppm=n/a` +
3 SKU с `network_interface=n/a` + 111 SKU с `starter_cartridge_pages=n/a`
(massive — HP не публикует starter yield на model-pages, как и Ricoh).
Через importer старые `regex_name` n/a остались бы, что не противоречит
методологии (regex_name — не грязный источник), **но** apply'ит
`claude_code_r4` source-tag везде → консистентность audit-trail важна.

**Первая попытка апплая** прокинулась network timeout'ом до Railway
proxy `turntable.proxy.rlwy.net:13528` — транзиентная ошибка. Retry —
**прошло.**

```sql
UPDATE printers_mfu pm
   SET attrs_jsonb      = src.attrs::jsonb,
       attrs_source     = 'claude_code_r4',
       attrs_updated_at = now()
  FROM (SELECT UNNEST(CAST(:skus AS text[])) AS sku,
               UNNEST(CAST(:attrs AS text[])) AS attrs) src
 WHERE pm.sku = src.sku
```

**Результат:** `rowcount: 145`. Один RTT Railway (~100мс).

## 4. Эффективно ли решение, что можно было лучше

**Что сработало:**

1. **Pre-flight test ДО запуска subagent'ов** — главный урок из r3
   incident'а. Один WebSearch+три WebFetch'а за 30 секунд показали:
   WebFetch стабильно недоступен, WebSearch работает. Без этого 3
   subagent'а сожгли бы rate-limit на retry'ях (как в r3).
2. **Эскалация при нештатной ситуации** через `AskUserQuestion` —
   собственник принял архитектурное решение (расширение методики),
   memory обновлена сразу. Исполнитель не «спрятался» в полном n/a и
   не нарушил методику по своей инициативе.
3. **Cap'ы соблюдены:** subagent A 30/200 WebSearch, B 46/200, C
   62/200. Никакой early termination не сработала — snippet'ы давали
   реальные данные.
4. **Family-стратегия subagent'а A** (1 WebSearch на семейство → 7
   запросов для 40 SKU) — экономия ~80% бюджета. Будущий шаблон.
5. **Normalizer-слой** (паттерн Epson r4 + Ricoh r4) дал 0 schema-
   ошибок. Subagent'ы вернули значения почти каноническими (urok Epson
   r4 учтён — в промте указал примеры из schema.py).
6. **Прямой batched UNNEST UPDATE** — один RTT, audit-tag консистентен.
7. **Аккуратный retry network timeout'а** на prod apply (paranoid
   approach: первая попытка fail, повторил — успех, без changes в
   скрипте).
8. **5 разнообразных sample-SKU** — гарантия согласованности
   normalizer↔DB.

**Что можно было лучше:**

1. **DATABASE_URL vs DATABASE_PUBLIC_URL.** В `.env.local.prod.v1` ключ
   называется `DATABASE_PUBLIC_URL`, а не `DATABASE_URL`. Изначально
   упёрся в KeyError. Раньше в Canon/Epson/Ricoh r4 паттерн был
   «DATABASE_URL = prod_env['DATABASE_URL']», что у меня сработало
   потому, что я не пересоздавал .env.local.prod.v1. Сейчас структура
   .env-файлов другая — надо было сразу проверить `dotenv_values().keys()`.
   Урок: всегда проверяю ключи env-файла перед использованием.
2. **Один-два subagent'а могли использовать family-стратегию более
   агрессивно.** Subagent A сделал 30 WebSearch для 40 SKU (0.75 на SKU).
   Subagent C сделал 62 для 56 (1.1 на SKU). Для следующих чатов:
   объяснить family-стратегию в промте явно («1 WebSearch на семейство»),
   не оставлять выбор subagent'у.
3. **starter_cartridge_pages 111/145 n/a.** Это легитимная HP-особенность
   (как и у Ricoh: 0/34). Для будущего bizz-case'а «полное обогащение
   starter» нужен PDF-проход через `pdftotext` на офисном сервере.
   Backlog.
4. **Sub-agent C ошибся с DeskJet IA Ultra 4828 ppm = n/a:** snippet
   дал «противоречивые цифры», но это, скорее всего, моя промт-инструкция
   «не угадывать» сыграла консервативно. Точечно можно было бы открыть
   datasheet, но subagent был ограничен WebFetch=5.
5. **`source_method` в done-артефакте все 145 = `websearch_snippet`.**
   Если в будущем будут чаты с гибрид-стратегией (WebFetch primary +
   WebSearch fallback) — поле будет различать. Для HP r4 — всё один
   метод, по сути audit-trail.
