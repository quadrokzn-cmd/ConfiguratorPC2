# 2026-04-25 — Волна 1А-β: обогащение атрибутов через Claude Code + UI справочника

## 1. Какая задача была поставлена

Собрать инфраструктуру обогащения атрибутов принтеров и МФУ через Claude Code
(паттерн export → JSON → Claude Code в отдельной сессии → import в БД) и UI
справочника `/nomenclature` с фильтрами, бейджами n/a и inline-правкой. Параллельно
работают 1А-α (парсер прайсов) и 1Б (ингест ЕИС) — пересечений по коду быть
не должно. Источник истины паттерна — `ConfiguratorPC2/app/services/enrichment/claude_code/`.

## 2. Как я её решал

1. **Изучил образец.** Прочитал `ConfiguratorPC2/app/services/enrichment/claude_code/{schema,exporter,importer}.py`,
   `enrichment/prompts/_общие_правила.md` и `cpu.md`. Понял контракт: pending → JSON → done → archive,
   валидация по полям, идемпотентность по уже выгруженным id.
2. **Адаптировал под нашу схему.** В отличие от ConfiguratorPC2 у нас одна таблица
   `nomenclature` с `attrs_jsonb` (а не 8 таблиц компонентов). Поэтому:
   - имена файлов в pending — `<brand>_<batch_id>.json` (бренд в роли категории);
   - схема одна: 9 целевых полей для печатной техники;
   - идемпотентность — по SKU в pending/done (archive не учитывается, чтобы можно
     было повторно прогнать после ручной правки).
3. **Добавил миграцию `0007_attrs_source.sql`** с колонками `attrs_source` и
   `attrs_updated_at` — план предписывал флаг источника для аудита; класть его в
   `attrs_jsonb._meta` оказалось грязнее, чем добавить две колонки.
4. **Расчёт `cost_base_rub`** — отдельный модуль `catalog/cost_base.py` с функцией
   `recompute_cost_base`. Четыре режима (sku / nomenclature_id / supplier_id /
   all_rows) — чтобы агент 1А-α мог дёшево пересчитывать после `load_price`,
   а importer — точечно после правки SKU. Контракт согласован в плане:
   функция лежит в catalog/, агент 1А-α её импортирует.
5. **UI**. Главный шаблон `nomenclature.html` + два partial'а:
   - `_nomenclature_row.html` — одна строка, перерисовывается через HTMX после
     правки атрибутов;
   - `_attrs_edit_form.html` — модалка с 9 полями (selects для enum-полей,
     текстовые для int-полей, чекбоксы для `network_interface`-списка).
   Partial `_upload_price_form.html` подключается через `{% include … ignore missing %}` —
   до того как агент 1А-α его положит, вместо формы рендерится подсказка-плейсхолдер.
6. **Промт-шаблон.** Написал `printer_mfu_attrs.md` на русском с белым списком
   доменов (Pantum, Brother, Kyocera, HP, Canon, Epson, Xerox, Ricoh, Lexmark,
   Konica Minolta), алгоритмом, схемой attrs, форматом выхода и чек-листом
   самопроверки. Запретил DNS, e2e4, маркетплейсы и агрегаторы.
7. **Тестовый прогон.** Положил `enrichment/done/test_001.json` (Pantum P3010DW
   с одним `n/a` в `starter_cartridge_pages`). Прогнал `enrich_import.py --dry-run`
   и smoke-тест валидации payload+attrs на позитивных и негативных кейсах. Также
   прогнал `_nomenclature_row.html` через Jinja с типичной строкой и проверил, что
   бейдж n/a виден, источник `claude_code` показан, список `network_interface`
   рендерится через `|join`.

## 3. Решил ли — да / нет / частично

**Частично — все артефакты сделаны, но финальный e2e-прогон с записью в БД и
переносом файла в archive не выполнен**, потому что Postgres в среде агента
не запущен (по плану его поднимает собственник вручную после Волны 0). Что
проверено и работает:
- импорты Python собираются (`fastapi`, `sqlalchemy`, мои модули);
- `validate_attrs` корректно ловит enum-ошибки, отсутствие полей, лишние поля;
- `_validate_payload_structure` ловит «не объект», «нет results», «нет sku/attrs»;
- Jinja-шаблоны парсятся и рендерятся с тестовой строкой (бейдж n/a, источник,
  список интерфейсов отображаются);
- `enrich_import.py --dry-run` на `test_001.json` валидирует структуру и attrs
  без ошибок, а на этапе записи отдаёт понятный отчёт `db_write_failed: 1` с
  причиной «Postgres недоступен» — то есть импортер ведёт себя предсказуемо.

Что осталось из DoD: после поднятия Postgres собственнику нужно прогнать
`python scripts/enrich_export.py --brand pantum --batch-size 30` (создаст пустой
JSON, если SKU ещё нет — это норма) и `python scripts/enrich_import.py`
(должен принять `test_001.json` и переехать в `archive/2026-04-25/`). А также
вручную через `Agent` tool в этой же сессии прогнать Claude Code на 2-3
моделях Pantum — чтобы убедиться, что промт-шаблон работает в реальном
поиске. Это часть DoD, которую можно выполнить только после того, как
агент 1А-α загрузит хотя бы один прайс с Pantum-моделями.

## 4. Эффективно ли решение, что можно было лучше

**Что вышло хорошо:**
- Контракт «JSON в pending → Claude Code → JSON в done → import» прозрачен и
  никак не зависит от Anthropic API в платформе. Budget-guard не активируется,
  ключа не нужно — это и было целью решения собственника от 2026-04-25.
- Структурная валидация и attrs-валидация выполняются ДО любых записей в БД.
  Если файл невалидный — он пропускается целиком и в archive не уезжает,
  оператор видит причину в отчёте.
- UI собран без Node.js, на связке Jinja + HTMX + Alpine + Tailwind CDN —
  как требует CLAUDE.md.
- `recompute_cost_base` имеет 4 режима, что покрывает и точечный пересчёт
  после правки SKU, и массовый после загрузки прайса целиком — без триггера
  в БД (триггер усложнил бы транзакционную логику и был бы дороже на больших
  батчах).

**Что можно было лучше:**
- В режиме `--dry-run` импортер всё равно ходит в БД для проверки существования
  SKU. Это удобно для прода, но в сценарии «проверить только структурную
  корректность файла без БД» приходится колупать smoke-скриптом. Если по факту
  понадобится — добавлю флаг `--validate-only`.
- В UI нет инлайн-правки `cost_base_rub` через клик-по-полю (есть отдельный
  POST-эндпоинт, но в шаблоне я его не вынес в кликабельный input — только
  через атрибуты). Если менеджер попросит — это 5 минут.
- Не учёл сценарий, когда `attrs_jsonb` уже заполнен старым (узким) набором
  ключей, а схема расширилась. Сейчас валидация требует ровно 9 ключей.
  Если в post-MVP добавится 10-й атрибут (вес для логистики ПЭК, например),
  понадобится миграция данных или мягкая валидация. Зафиксирую в риски ниже.

**Было:**

- `enrichment/` с подпапками и `.gitkeep`; `enrichment/pending/*` и

- В `app/main.py` подключён `nomenclature_router` (старый stub-роут
  `GET /nomenclature` удалён).

## Вопросы к оркестратору

1. **Нужен ли отдельный режим `--validate-only` для `enrich_import.py`?** Сейчас
   даже `--dry-run` ходит в БД для проверки существования SKU. Если оркестратор
   хочет валидировать done-файлы без БД — добавлю флаг.
2. **Когда агент 1А-α положит `_upload_price_form.html`?** Я подключил его через
   `ignore missing` — пока показывается заглушка с подсказкой. После того как
   1А-α положит partial, он подцепится автоматически.
3. **Контракт `recompute_cost_base` устраивает 1А-α?** Я положил функцию в
   `app/modules/auctions/catalog/cost_base.py`. Агент 1А-α должен импортировать
   `from app.modules.auctions.catalog.cost_base import recompute_cost_base` и
   вызывать `recompute_cost_base(supplier_id=...)` после успешной загрузки прайса.
4. **Расширение схемы атрибутов в post-MVP — как?** Сейчас `validate_attrs`
   требует ровно 9 ключей. Если расширим (вес, габариты для логистики ПЭК) —
   нужна миграция данных по существующим SKU (досыпать `n/a` в недостающие
   ключи) или мягкий режим валидации (присутствие старых ключей не обязательно).
   Не делаю в Волне 1А-β, оставляю в риски post-MVP.
5. **Реальный прогон Claude Code на 2-3 Pantum-моделях** — оркестратор должен
   запустить его после того, как 1А-α загрузит первый прайс с Pantum-SKU.
   Без живых SKU мне нечего класть в pending. Тестовый файл `test_001.json`
   валидируется корректно, это лучшее, что можно сделать без живой БД.
