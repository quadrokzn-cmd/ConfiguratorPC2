## 1. Какая задача была поставлена

Развернуть pre-prod environment Railway по `docs/preprod-deploy.md` (написанному в 9d.1) и подтвердить готовность инфраструктуры через smoke в браузере. Главные обязательства DoD:

- Создать environment `pre-prod` в существующем проекте Railway (НЕ копировать prod-секреты).
- Поднять Postgres-сервис `postgres-preprod` (отдельный от prod).
- Развернуть оба application-сервиса (configurator-preprod через `Dockerfile`, portal-preprod через `Dockerfile.portal`) в статусе Online.
- Привязать кастомные домены `config-preprod.quadro.tatar` / `app-preprod.quadro.tatar` через Reg.ru DNS, дождаться SSL Let's Encrypt.
- Настроить ~30 env-vars в обоих сервисах (без копирования prod-секретов; новый APP_SECRET_KEY, новый ADMIN_PASSWORD).
- Залить seed-дамп в pre-prod БД (через `scripts/preprod_seed_dump.sh` → `psql -f`).
- Гарантировать выключенные auto-price-loads и включённый `auctions_ingest_enabled`.
- Smoke в браузере: login, виджеты, /auctions (5 секций), /nomenclature (628 SKU), /auctions/settings (6 settings + 7 регионов + 2 KTRU), cross-subdomain auth, конфигуратор отвечает на тестовый запрос.
- Финальный коммит этапа 9d.2 в git с обновлением плана и рефлексией.

Жёсткие рамки: НЕ затрагивать prod environment и его секреты, НЕ деплоить уведомления Telegram/Max (9b ещё не реализован), НЕ запускать auto-price-loads на pre-prod, НЕ слать реальные SMTP-письма.

**Хронология реальных событий 9d.2** (по логам собственника):

1. **Создание environment `pre-prod`** через Railway UI («+ New Environment» → Empty environment, не Copy from production). Postgres-сервис `postgres-preprod` поднят за ~30 секунд (PostgreSQL 18). Получены `DATABASE_URL` (внутренний) и `DATABASE_PUBLIC_URL` (для psql с dev-машины).

2. **Создание двух application-сервисов** (`configurator-preprod` через Dockerfile, `portal-preprod` через Dockerfile.portal). Healthcheck path `/healthz`, healthcheckTimeout 60 для конфигуратора и 300 для портала (на холодном старте портал применяет миграции 001-034 + bootstrap_admin.py — это занимает заметное время).

3. **Variables и secrets**. 29 переменных в Shared Variables environment-а `pre-prod` (общие для всех сервисов). Поначалу пытались подвязать `DATABASE_URL` через Shared Variables как reference (`${{ postgres-preprod.DATABASE_URL }}`), но reference не разрешалась вне сервисного scope; перевели в per-service Variables — заработало. Сгенерирован отдельный `APP_SECRET_KEY = secrets.token_urlsafe(48)` (НЕ копия prod). `ADMIN_PASSWORD` pre-prod — отдельный длинный random. **Важная защёлка**: суггестед-блок «Variables → Suggested Variables → Add all» НЕ нажимать — Railway применяет placeholder-значения из `.env.example` в открытом виде (включая placeholder-секреты типа `APP_SECRET_KEY=changeme`). Только point-by-point добавление.

4. **Custom domains через Reg.ru DNS**. Railway → Settings → Networking → Custom Domain → ввод `config-preprod.quadro.tatar` → Railway показал CNAME-target вида `<sometoken>.up.railway.app` + TXT-токен `_railway.config-preprod.quadro.tatar` для verify. В Reg.ru добавлены 4 записи: 2 CNAME (`config-preprod` и `app-preprod` → Railway-CNAME-targets) + 2 TXT (`_railway.config-preprod` / `_railway.app-preprod` → verify-токены). DNS пропагировался за 1-3 минуты, Railway автоматически выдал SSL Let's Encrypt.

5. **Цикл «remove → re-add custom domain»** при дебаге одного из доменов. Урок: при re-add Railway даёт **новый** CNAME-target — старый CNAME в Reg.ru становится мёртвым, нужно обновлять вручную. TXT-токен Railway сохраняет (он привязан к FQDN, не к custom-domain-записи как таковой).

6. **Залив seed-дампа** — самый болезненный участок:
   - **Первая попытка**: `psql -f scripts/preprod_seed.sql` через `DATABASE_PUBLIC_URL`. Тихо повисло на ~3 минуте, файл-лога едва рос. `tasklist | findstr psql` показал, что psql.exe (версия 16) занят. Смотрим первые строки `preprod_seed.sql` — pg_dump 17 (на dev-машине) добавил `\restrict` / `\unrestrict` meta-команды, на которые psql 16 не реагирует, но и не двигается. Решение: **`scripts/_strip_restrict.py`** — 10-строчный sanitizer на regex (`re.compile(rb'^\\(?:un)?restrict[^\r\n]*\r?\n', re.MULTILINE).subn`), генерит `scripts/preprod_seed.clean.sql` для совместимости. После запуска sanitizer-а psql пошёл, но...
   - **Вторая попытка**: `psql -f preprod_seed.clean.sql` — теперь идёт, но **очень медленно** (24 тыс. INSERT-ов через Railway-PG прокси, latency ~50-100 мс на каждый round-trip). После 20 минут публичная сессия порвалась. Диагностика: `--column-inserts` (флаг pg_dump из 9d.1) даёт по INSERT-у на строку, что не подходит для public-URL заливки.
   - **Третья попытка**: убрал `--column-inserts` из `scripts/preprod_seed_dump.sh` (теперь дефолтный COPY-формат), регенерил seed → одиночный COPY-блок на таблицу, ~30 секунд на всю заливку. Это исправление 9d.1-овой ошибки — тогда `--column-inserts` ставился «для устойчивости к schema-различиям», но на практике schema управляется apply_migrations.py и **одинакова** между dev и pre-prod.
   - **Четвёртая попытка**: `TRUNCATE 15 таблиц RESTART IDENTITY CASCADE` перед заливкой (`auto_price_loads` + 14 seed-таблиц). Без TRUNCATE psql упадёт по `duplicate key` — миграции 001-034 уже залили 6 таблиц своими data-INSERT-ами при первом старте контейнера portal-preprod (cases, coolers, ktru_watchlist, excluded_regions, settings, auto_price_loads). Затем psql -f → за ~30 секунд серия `COPY N` без ошибок. **Контрольные SELECT-ы совпали** (printers_mfu=628, cpus=228, supplier_prices=13953, ktru_watchlist active=2, excluded_regions excluded=7, settings 6 строк включая `auctions_ingest_enabled='true'`).
   - **Зомби-psql.exe на Windows** — между попытками Bash bridge не убивал зависший процесс при разрыве bash-скрипта; пришлось `taskkill //F //IM psql.exe` вручную.

7. **Контрольные SQL-апдейты после seed**: `UPDATE auto_price_loads SET enabled=FALSE WHERE supplier_slug IN (...)` вернул `UPDATE 0` — таблица пустая после TRUNCATE и миграция 028 не перезалила (запись в schema_migrations уже есть). Это **корректно**: пустая `auto_price_loads` = APScheduler не находит slug-ов = ноль автозагрузок. Двойная защита от конкуренции с prod: пустая таблица + пустые `TREOLAN_API_LOGIN/PASSWORD` в env. `SELECT key, value FROM settings WHERE key='auctions_ingest_enabled'` → `'true'` ✅.

8. **Smoke в браузере** (через Claude Chrome + собственник):
   - `app-preprod.quadro.tatar/login` — открылась форма входа (без редиректа на prod) ✅.
   - Логин как `admin` / `<ADMIN_PASSWORD pre-prod>` ✅.
   - Главная портала — виджет «Аукционы» виден с нулями (ингест ещё не отработал), 5 виджетов рендерятся ✅.
   - `/auctions` — 5 пустых секций (urgent / ready_to_review / new_low_margin / in_work / archive) ✅.
   - `/nomenclature` — 628 SKU видны, фильтры по brand/category работают ✅. **Известное ограничение**: pagination показывает первые 200 SKU без UI «следующая страница» (открытый вопрос для будущей итерации).
   - `/auctions/settings` — 6 settings с правильными значениями + 7 регионов excluded + 2 активных KTRU ✅.
   - **Cross-subdomain auth**: переход с app-preprod на `config-preprod.quadro.tatar` без перелогина (cookie `kt_session` шарится через `APP_COOKIE_DOMAIN=.quadro.tatar`) ✅.
   - Конфигуратор отвечает на тестовый NLU-запрос ✅. **Известное ограничение**: multi-storage (SSD+HDD) NLU схлопывает в один накопитель — открытый вопрос для будущей итерации.

## 4. Эффективно ли решение, что можно было лучше

**Эффективно:**

- **Разделение Claude Chrome ↔ Claude Code**: Claude Chrome работал в Railway UI и Reg.ru DNS пошагово, Claude Code параллельно генерировал seed-дамп, фиксил скрипты, обновлял документ под живой troubleshooting. Собственник как passerby — только логин и копи-паст. Это удобный шаблон workflow для подобных «инфраструктурных + кодовых» задач.
- **Pivot на COPY-формат для seed-заливки**: за один цикл «попытка → диагноз → фикс скрипта» решена проблема с публичной сессией Railway-PG прокси. Сэкономили 20+ минут на каждой следующей заливке (если понадобится повторить).
- **TRUNCATE 15 таблиц RESTART IDENTITY CASCADE** — единая команда покрывает auto_price_loads + 14 seed-таблиц, не оставляет sequence-конфликтов. Зафиксировано в `docs/preprod-deploy.md` шагом З.3 — будущим деплоям pre-prod-style не наступать на грабли.
- **Двойная защита от конкуренции с prod**: пустая `auto_price_loads` после TRUNCATE + пустые `TREOLAN_API_LOGIN/PASSWORD` в env. Даже если кто-то вручную включит тумблер на slug — fetcher упадёт без credentials, реальной автозагрузки не будет.
- **Cross-subdomain auth через `APP_COOKIE_DOMAIN=.quadro.tatar`** — работает с первого раза. SSO между config-preprod и app-preprod протестирован в браузере.
- **Wildcard в `.gitignore` для `.env.local.preprod*`** — на будущее ловит любые версии (`.v2`, `.v3`, ...) без пере-правок при ротации секретов.

**Что можно было лучше:**

- **`--column-inserts` ошибка на этапе 9d.1**. На 9d.1 я (тот же я в предыдущем чате) поставил `--column-inserts` «для устойчивости к schema-различиям», не подумав про latency Railway-PG прокси. На 9d.2 это вылилось в 20 минут зря потраченного времени и обрыв сессии. **Урок**: для public-URL заливок всегда COPY, `--column-inserts` оправдан только на схемах, где dev и target расходятся (а у нас они одинаковы благодаря apply_migrations.py).
- **Нечёткий security-регламент по Railway Raw Editor**. Раньше не было правила «не открывать Railway Raw Editor / Apply Details modal в LLM-чатах» — собственник открыл их в Claude Chrome для точечной правки, и LLM получил `APP_SECRET_KEY` + `OPENAI_API_KEY` в открытом виде в context. Все 3 секрета пришлось ротировать. **Зафиксировано** в auto-memory как `feedback_railway_raw_editor_secrets.md` — на будущих деплоях это правило заранее ограничит ущерб.
- **`TREOLAN_API_PASSWORD="s4]O]h|W"` в `.env.example`** — обнаружен в публичной git-истории. На 9d.1 я (тот же я в предыдущем чате) НЕ читал `.env.example` напрямую (permission policy блокировала), и не словил эту утечку. На 9d.2 это всплыло при чтении `.env.example` через Railway Suggested Variables (там значения отображаются в открытую). **Урок**: сразу при старте проекта — grep по `.env.example` на «реальные»-выглядящие пароли, не только на `changeme`/`xxx`/`placeholder`.
- **Нет автоматизированного smoke-теста на pre-prod**. Сейчас smoke — это ручной обход в браузере по ~9 пунктам. Можно было бы написать `scripts/smoke_preprod.sh`: curl-проверка `/healthz` обоих сервисов, GET `/login`, login + GET `/auctions`, верификация структуры HTML (там должны быть 5 секций). Не критично, но при следующем деплое pre-prod (если понадобится пересоздать) — экономия времени.

**Что я бы исправил, если бы переписывал:**

- На 9d.1 — поставил бы COPY-формат сразу (без `--column-inserts`), и не было бы эпизода «20 минут на заливку → обрыв сессии».
- На 9d.1 — добавил бы `scripts/_strip_restrict.py` сразу как часть seed-pipeline (`bash scripts/preprod_seed_dump.sh` сразу бы генерил clean-версию), а не реактивно после факта зависания psql.
- На 9d.1 — внёс бы в `.gitignore` сразу `.env.local.preprod*` с wildcard-ом, чтобы любые версии файла секретов ловились с порога.

## Открытые вопросы (для оркестратора и для будущих итераций)

1. **Pagination на `/nomenclature`** — показывает первые 200 SKU без UI «следующая страница». Известное ограничение, открытый вопрос. Не блокер для acceptance pre-prod (можно ловить нужный SKU через поиск/фильтры). Будущая итерация — добавить серверный paginator + кнопки «next/prev».
2. **Multi-storage в NLU конфигуратора** — SSD+HDD схлопывается в один накопитель. Известный баг NLU, существующее ограничение. Не блокер для pre-prod acceptance модуля «Аукционы» (это про конфигуратор, не аукционы), но открытый вопрос для общей UX-доработки.
3. **`TREOLAN_API_PASSWORD` в `.env.example` (публичный git history)** — security task. Действия: (a) сменить пароль Treolan через их саппорт; (b) убрать из `.env.example` (placeholder); (c) опционально — `git filter-repo` чистка истории. Отдельный мини-этап после 9d.2.
4. **`git filter-repo`/BFG чистка истории** от больших SQL-дампов 9c (~117 МБ суммарно: `quadrotech-full.sql` + `quadrotech-data-only.sql`, удалённых из working tree, но физически остающихся в git-истории). Требует force-push на master, **отдельное согласие собственника**. Не блокер: репо клонируется Railway shallow-копией, вес не мешает.
5. **Acceptance ингеста** — нужно подождать 24-48 часов живого ингеста с zakupki (или нажать кнопку «Запустить ингест сейчас» в `/auctions/settings`). После наполнения inbox-а — собственник прогоняет UX-сценарии менеджера. Только после этого pre-prod считается полностью acceptance-ready.
