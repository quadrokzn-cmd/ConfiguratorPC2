# 2026-05-11 — Мини-этап 9e.2: CLI `scripts/run_auctions_ingest.py` для запуска ingest из произвольного окружения

## 1. Какая задача была поставлена

Этап 9e (production-ingest архитектура, путь A — офисный сервер в РФ
пишет в Railway-PG через `DATABASE_PUBLIC_URL` под ограниченной ролью
`ingest_writer`). 9e.1 закрыт: роль создана на pre-prod-БД, GRANT'ы
выданы, DSN сохранён в gitignored `.env.local.preprod.v2` под ключом
`INGEST_WRITER_DATABASE_URL_PREPROD`. Прод-ingest пока заблокирован,
потому что Railway-датацентр в US/EU недоступен для zakupki.gov.ru.

В рамках 9e.2 нужно сделать самостоятельный CLI, который умеет:
прочитать .env, создать SQLAlchemy-engine с DSN ingest_writer-роли,
вызвать существующий `run_ingest_once(engine)` один раз и завершиться.
Никакого APScheduler, никакого FastAPI, никаких импортов
`portal/main.py`/`app/main.py`. Это база под 9e.3 (раскладка по Task
Scheduler на офисном сервере) и 9e.4 (тот же CLI с DSN
`INGEST_WRITER_DATABASE_URL_PROD` после применения миграции 0035 на
prod-БД).

## 2. Как я её решал

**Шаг 1. Чтение существующего кода.** Перечитал
`app/services/auctions/ingest/orchestrator.py` — там готовая функция
`run_ingest_once(engine) -> IngestStats` без побочных эффектов на
конфиг (не дёргает `app.config`, не импортирует `app.main`). Прошёл по
импорт-цепочке `repository.py → card_parser.py → http_client.py` —
никто из них не требует ENV-vars, всё через engine. Это значит CLI
может вообще не трогать `app/config.py` (в нём `_require_env` падает
без `DATABASE_URL`, `OPENAI_API_KEY` и т.д., а нам нужен только
изолированный engine).

Посмотрел `scripts/run_matching.py` как образец предыдущего тонкого
CLI — там `sys.path.insert(0, ...)`, `load_dotenv()`,
`from app.database import engine` (читает `DATABASE_URL`), вызов
сервиса, печать сводки, exit. Для 9e.2 паттерн похож, но engine
строим из произвольного DSN-ключа.

**Шаг 2. Дизайн CLI.** argparse-опции по DoD:
`--env-file` (default `.env`), `--db-url-env`
(default `INGEST_WRITER_DATABASE_URL` — продовое имя; для pre-prod
передаём `INGEST_WRITER_DATABASE_URL_PREPROD`), `--log-level`
(`DEBUG|INFO|WARNING|ERROR`, default `INFO`). Логика вынесена в
отдельную функцию `run_ingest(env_file, db_url_env, log_level)` —
чтобы можно было покрыть тестами без `subprocess`.

Поток:

1. `load_dotenv(env_path, override=False)` — env-файл с
   gitignored-секретом подмешивается в `os.environ`, существующие
   значения не перезаписываются.
2. Читаем DSN из `os.environ[db_url_env]`. Пусто или нет — exit 2 с
   понятным сообщением (в stderr указано имя env-переменной и имя
   env-файла).
3. Перенастраиваем `loguru.logger` на нужный уровень (orchestrator
   пишет через тот же logger, так что `remove()+add(level=...)`
   применится и к его сообщениям).
4. `create_engine(dsn, pool_pre_ping=True)`.
5. Ставим SIGINT-handler, который вызывает `engine.dispose()` и
   `sys.exit(130)`.
6. Вызываем `run_ingest_once(engine)`, ловим всё исключения через
   `try/except Exception`, при падении — `logger.exception(...)`,
   `engine.dispose()`, `return 1`.
7. На успехе — логируем `stats.as_dict()` + `elapsed_sec`,
   `engine.dispose()`, `return 0`.

**Безопасность.** DSN/пароль не печатается ни в одной ветке. В лог
идёт только имя env-переменной из которой взят DSN, имя env-файла, и
итоговый `IngestStats.as_dict()` (он не содержит секретов). Никаких
`os.environ.get(..., default)`-печатных fallback'ов на DSN.

**Шаг 3. Тесты.** Создал `tests/test_auctions/test_run_auctions_ingest.py`
с 4 кейсами:

1. `test_run_ingest_missing_env_file` — несуществующий путь → exit 2,
   в stderr «не найден».
2. `test_run_ingest_missing_dsn` — пустой env-файл, env-переменная
   удалена через `monkeypatch.delenv` → exit 2, в stderr имя
   переменной и «не задана».
3. `test_run_ingest_success` — env-файл с фиктивным DSN,
   `monkeypatch.setattr("sqlalchemy.create_engine", ...)` на fake-engine
   и `monkeypatch.setattr("app.services.auctions.ingest.orchestrator.run_ingest_once", ...)`
   на фейк, возвращающий `SimpleNamespace(as_dict=lambda: {...})`. CLI
   возвращает 0, `dispose()` вызван, fake-engine передан в orchestrator.
4. `test_run_ingest_unhandled_exception` — fake `run_ingest_once`
   кидает `RuntimeError` → exit 1, `dispose()` вызван (важно — engine
   не висит на исключении).

Скрипт грузится через `importlib.util.spec_from_file_location` (он не
в пакете — `scripts/` без `__init__.py`).

**Шаг 4. Live-прогон на pre-prod.** Запустил CLI с dev-машины:

```bash
python scripts/run_auctions_ingest.py \
    --env-file .env.local.preprod.v2 \
    --db-url-env INGEST_WRITER_DATABASE_URL_PREPROD
```

паттерну `*.log` и `logs/`):

Exit code = 0. **`permission denied` ни разу не возникло** — роль

Цифры: `cards_parsed=159 / cards_failed=0` — парсер прошёл по всем
карточкам без сбоев. `inserted=0 / updated=159` — pre-prod-БД уже
знала все эти 159 reg_numbers от предыдущих прогонов APScheduler-job
`auctions_ingest` в `portal-preprod`, ingest идемпотентно перезаписал
поля; для офисного worker'а это правильное поведение — он будет
донаполнять и обновлять, а не дублировать. `elapsed_sec=726.7` ≈ 12.1
минуты (159 карточек × ~4.6 сек/карточка — медленнее, чем оценка Волны
2 в ~6.5 минут, потому что throttling `ZakupkiClient` плавающий + сеть
до Railway-PG проходит через US-прокси).

Флаги: `excluded_region=4` (4 лота в стоп-регионах из 7), `below_nmck=3`
(НМЦК < 30 000 ₽), `over_unit_price=20` (есть позиция дороже 300 000 ₽

`no_watchlist_ktru=14` (карточка лота не содержит ни одного из 2
целевых КТРУ — возможно, ингест поднял её по дочернему коду, который
zakupki вернул в выдаче, но в самой карточке агрегации нет),
`no_positions=10` (позиции вообще не распарсились — обычно лоты с
PDF-приложением вместо HTML-таблицы). Это всё штатные фильтры, ничего
не сломалось.

## 4. Эффективно ли решение, что можно было лучше

**Что хорошо.**

- Архитектурно правильно вынес тело в `run_ingest(env_file, db_url_env, log_level)`
  — без этого пришлось бы гонять CLI через `subprocess`, что съедало
  бы секунды на каждом тесте и не давало бы `monkeypatch`.
- Тесты грузят скрипт через `importlib.util.spec_from_file_location`,
  потому что `scripts/` не пакет. Альтернатива — добавить `__init__.py`
  — заведомо хуже (изменение структуры репо ради одного теста).
- DSN-безопасность: единственный путь, по которому DSN попадает в
  логи — `create_engine(dsn, ...)`, и SQLAlchemy маскирует пароль в
  `OperationalError`. В прямом виде из CLI он никогда не выводится.
- `pool_pre_ping=True` поверх Railway-прокси: на офисном сервере DSL
  и пара минут простоя — обычное дело, ping предотвратит «stale
  connection» при следующем запуске. На локальном dev-прогоне эффекта
  нет, но в продакшене (9e.3) пригодится.
- SIGINT-handler. Ctrl+C на dev-машине во время отладки — раз в час;
  без handler'а engine остаётся висеть с открытыми соединениями к
  Railway-прокси.

**Что можно было лучше.**

- Метрики прогона (elapsed_sec, inserted, updated, cards_failed) пока
  пишем только в лог — нет JSON-stats-файла рядом с лог-файлом. На
  офисном сервере (9e.3) для мониторинга может понадобиться парс
  лога; проще было бы `--stats-file <path>` опцией. Пока не делаем,
  чтобы не раздувать CLI; добавим в 9e.3, когда станет понятно, в
  каком формате это удобнее читать (Sentry-метрики? plain JSON?).
- Тесты не проверяют SIGINT-ветку (signal-handler не покрыт). На
  Windows-pytest сигналы тяжело имитировать, и польза от теста
  маленькая — пропустил.
- Тест на успешный путь не проверяет, что `pool_pre_ping=True`
  передалось в `create_engine` — это проверяется в фейке косвенно
  через `assert kwargs.get("pool_pre_ping") is True`. Достаточно.

## 5. Как было / как стало

**Было** (до 9e.2): запустить ingest можно только через FastAPI-портал
(`portal/main.py` + APScheduler-job `auctions_ingest`). Это значит, что
для prod-rollout (этап 9e) офисный сервер должен был тащить с собой
весь портал-стек, поднимать uvicorn, читать `DATABASE_URL` через
`app/config.py` (с обязательными `OPENAI_API_KEY`, `APP_SECRET_KEY` и
прочими env-vars, которые в офисном worker'е не имеют смысла).

**Стало** (после 9e.2): один файл `scripts/run_auctions_ingest.py` +
`.env`-файл с одной строкой `INGEST_WRITER_DATABASE_URL_PROD=...`.
Запустил — он сам подцепил env, создал engine, выполнил один цикл
ingest, выгрузил статистику, закрылся. Готов к раскладке через Task
Scheduler/systemd в 9e.3. И к переключению на prod-DSN в 9e.4 без
правки кода — просто другая env-переменная.
