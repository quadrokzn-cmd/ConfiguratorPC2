# 2026-05-11 — Мини-этап 9e.1: ограниченная PG-роль `ingest_writer` для офисного ingest-worker'а

## 1. Какая задача была поставлена

Этап 9e (production-ingest архитектура). Решение собственника от
2026-05-11 — путь A: офисный сервер в РФ со статическим IP запускает
ingest и пишет напрямую в Railway-PG через `DATABASE_PUBLIC_URL` под
отдельной ограниченной ролью. zakupki.gov.ru с офисного IP не
блокируется (smoke от 2026-05-11: HTTP 200 на главной, поиске и
карточке).

В рамках 9e.1 нужно завести в pre-prod-БД роль `ingest_writer` с
минимальными правами (только то, что нужно
`app/services/auctions/ingest/repository.py` для full upsert лотов),
сгенерировать ей пароль, прописать новый DSN в gitignored env-файл,
smoke-проверить положительные SELECT/INSERT/DELETE и отрицательные
`permission denied` на чужих таблицах. 9e.4 повторит то же на prod-БД
позже.

## 2. Как я её решал

**Подготовка.** Прочитал `app/services/auctions/ingest/repository.py`
и миграции 030/031/032/034 — вычленил список реально дёргаемых таблиц:
`settings/excluded_regions/ktru_watchlist` (SELECT на старте цикла) +
`tenders/tender_items/tender_status` (INSERT/UPDATE/DELETE в `upsert_tender`).
Колонок `customer_contacts`/`customer_addresses` нет — это JSONB-поле
в `tenders`. Из BIGSERIAL найдена одна нужная sequence —
`tender_items_id_seq` (у `tenders` и `tender_status` PK типа TEXT).
Sequence у `matches` (`matches_id_seq`) — не нужна, ingest в matches
не пишет.

**Шаг 1. Миграция.** Написал `migrations/0035_ingest_writer_role.sql`
с идемпотентным DO-блоком (`IF NOT EXISTS (SELECT 1 FROM pg_roles ...)`)
и точечными GRANT'ами (CONNECT на `railway`, USAGE на `public`,
SELECT/INSERT/UPDATE/DELETE на цели, USAGE+SELECT на sequence). Никаких
прав на чужие таблицы.

**Шаг 2. Применение на pre-prod.** Файл `.env.local.preprod.v2`
gitignored и недоступен мне для прямого Read — это правильно, секреты
не должны утекать в LLM-контекст. Попросил собственника дописать туда
`DATABASE_PUBLIC_URL=...` (он добавил). Дальше — извлекал URL из файла
только внутри bash-команд через python-обёртку, в чат пароль не
выводил. Применил миграцию через psql heredoc + сразу `INSERT INTO
schema_migrations` под админом — DO + 9 GRANT + INSERT, всё OK.
`applied_at = 2026-05-11 11:42:37+00`.

**Шаг 3. Пароль и ALTER ROLE — здесь была серьёзная ошибка.** Первый
заход: сгенерировал пароль `secrets.token_urlsafe(32)`, попытался
прокинуть его в `psql -c "ALTER ROLE ... PASSWORD :'PWD'"`. На Windows
git-bash + psql 16 флаг `-c "..."` потерял аргумент (psql выдал
warning «лишний аргумент … пропускается») — ALTER ROLE не отработал.
Дополнительно я завернул вывод в `psql ... 2>&1 | grep -v "$NEW_PASSWORD"`
с целью «спрятать пароль, если он случайно мелькнёт». Сторонний
Windows-`grep` оказался не GNU grep'ом и интерпретировал значение
`$NEW_PASSWORD` как аргумент опции `--directories`. В stderr вывелось
сообщение **с самим паролем в тексте ошибки**:
`grep: invalid argument 'WPmHou...' for '--directories'`. Этот текст
прошёл в LLM-контекст. Пароль был скомпрометирован.

**Реакция:** немедленно сгенерировал новый пароль и заменил старый
через ALTER ROLE. Старый пароль работал не дольше нескольких секунд
(если вообще успел применеиться — первый ALTER ROLE упал из-за
warning'а в psql). Новый пароль никогда не приближался ни к stdout,
ни к stderr.

**Рабочий способ ALTER ROLE.** Python-скрипт пишет SQL во временный
файл через `tempfile.mkstemp(suffix='.sql')`, отдельно сохраняет
пароль в `.txt` для последующего формирования DSN. `psql "$URL" <
"$SQL_FILE"` — stdin redirection (не `-f`, не `-c`). После
применения — `rm -f` обоих временных файлов. На третьей попытке
`ALTER ROLE rc=0`.

**Шаг 4. URL.** Сформировал `postgresql+psycopg2://ingest_writer:
<url-encoded-pwd>@maglev.proxy.rlwy.net:32320/railway?sslmode=require`
и записал в `.env.local.preprod.v2` под ключом
`INGEST_WRITER_DATABASE_URL_PREPROD`. Файл gitignored по паттерну
`.env.local.preprod*`.

**Шаг 5. Smoke.** Через psql под `ingest_writer`:
- positive — `current_user='ingest_writer'/db='railway'`, `SELECT
  COUNT(*)` дал `settings=6 / ktru_watchlist=10 / excluded_regions=7 /
  tenders LIMIT 1=1 / tender_items LIMIT 1=1 / tender_status LIMIT 1=1`;
- positive write — `BEGIN; INSERT INTO tenders (TEST-9E1-SMOKE-001)
  + tender_items + tender_status; SELECT verify; DELETE FROM tenders
  WHERE reg_number='TEST-9E1-SMOKE-001'; COMMIT;` — INSERT 0 1 ×3,
  DELETE 1, leftover везде 0 (CASCADE сработал);
- negative — `SELECT 1 FROM users / audit_log / printers_mfu /
  supplier_prices / matches / suppliers / cpus / auto_price_loads
  LIMIT 1` — все 8 вернули `ERROR: permission denied for table
  <name>`. Лишних GRANT'ов нет.

**Шаг 6. Документация.** Добавил в `docs/preprod-deploy.md` секцию
«Шаг Л» (после «Шаг К»): что роль умеет, что не умеет, точные
команды применения миграции, генерации пароля через tempfile,
ALTER ROLE через stdin redirection (с примечанием про
Windows-psql `-c "..."`), smoke-проверки, ссылка что 9e.4 повторит
для prod-БД с отдельным паролем.

**Шаг 7. План.** В `plans/2026-04-23-platforma-i-aukciony.md`
добавил блок «Мини-этап 9e.1 завершён 2026-05-11» в раздел «Что
реализовано» (по образцу предыдущих 9a-блоков) и переформулировал

## 4. Эффективно ли решение, что можно было лучше

**Эффективно по сути:** все GRANT'ы точечные (smoke negative подтвердил),
миграция идемпотентна, пароль маршрутизирован через tempfile без
утечки в bash-args/git. Документация и план обновлены так, что 9e.4
по тем же шагам соберёт роль на prod без работы заново.

**Что было плохо и чему научился:**

- **Главная ошибка**: первая попытка ALTER ROLE использовала пайп
  `2>&1 | grep -v "$NEW_PASSWORD"` как «защиту» от случайного вывода
  пароля. На Windows `grep` оказался не GNU-вариантом — он
  интерпретировал значение `$NEW_PASSWORD` (которое мы туда и
  передали целиком, а не как опцию `-v`) как аргумент для опции
  `--directories`. В результате stderr вывело: `grep: invalid argument
  'WPmHou...' for '--directories'` — с самим паролем в тексте ошибки.
  Если бы я использовал `2>/dev/null` или просто не выводил stderr
  вовсе, утечки бы не было. **Урок:** «спрятать через grep» — это
  анти-паттерн, потому что grep на ошибке распечатает значение, против
  которого тебя пытались защитить. Правильный способ — direct
  redirection в `/dev/null` или вообще не пускать пароль ни в один
  субпроцесс, кроме psql через файл.

- **Второй просчёт:** на Windows-psql 16 `-c "ALTER ROLE ... PASSWORD
  :'PWD'"` и `-f "<path>"` оба раз сломались — psql отдавал warning
  «лишний аргумент пропускается». Я полагал что командная строка
  работает одинаково на Linux и Windows. **Урок:** в проекте под
  Windows стандартный безопасный паттерн для psql — stdin redirection
  (`psql "$URL" <<'SQL' ... SQL` или `< file.sql`). Зафиксировано в
  `docs/preprod-deploy.md`.

- **Третий мелкий:** `mktemp --suffix=.sql` на git-bash под Windows
  завис (нет такой опции в Windows-сборке coreutils). Перешёл на
  `python -c "import tempfile; tempfile.mkstemp(...)"`. Урок: при
  работе из bash под Windows тащить системные tools через python.

**Что было хорошо:**
- разбор миграций + repository.py до начала кода — это сразу дало
  правильный список GRANT'ов без лишних итераций;
- ротация скомпрометированного пароля произошла в течение секунд
  после обнаружения, не позже;
- smoke-проверки сделаны не только positive (что работает), но и
  full negative-список (что НЕ работает) — это и подтвердило
  отсутствие лишних GRANT'ов.

## 5. Как было / как стало

**Как было (до 9e.1):**

- На pre-prod-БД одна роль `postgres` (суперюзер). Любой сервис /
  worker / Claude-Code-скрипт ходит под ней — read/write по всей БД.
- Pre-prod ingest запускается APScheduler-job'ом `auctions_ingest` в
  `portal-preprod` (Railway-DC, ходит к Railway-postgres по
  внутреннему `DATABASE_URL`). zakupki.gov.ru блокирует Railway-DC
  (US West, EU West Amsterdam — оба timeout), prod-ingest не работает.
- Архитектурно нет принципа least-privilege для ingest-цикла:
  крадётся пароль `postgres` → доступ ко всему (audit_log, users,
  supplier_prices).

**Как стало (после 9e.1):**

- На pre-prod-БД появилась вторая роль `ingest_writer` (LOGIN), её
  права (видно из `psql` под ней):
  - **СВОИ (положительный доступ):**
    - DATABASE `railway` — CONNECT;
    - SCHEMA `public` — USAGE;
    - TABLE `settings`, `excluded_regions`, `ktru_watchlist` — SELECT;
    - TABLE `tenders`, `tender_items`, `tender_status` —
      SELECT/INSERT/UPDATE/DELETE;
    - SEQUENCE `tender_items_id_seq` — USAGE, SELECT (для BIGSERIAL).
  - **ЧУЖИЕ (permission denied, проверено в smoke):** `users`,
    `audit_log`, `printers_mfu`, `supplier_prices`, `matches`,
    `suppliers`, `cpus`, `auto_price_loads` (выборочный список — на
    остальных таблицах тот же эффект, потому что роль не имеет
    дефолтного `PUBLIC SELECT`).
- Маршрут пароля:
  1. Сгенерирован `python -c "import secrets;
     print(secrets.token_urlsafe(32))"` (длина 43 символа).
  2. SQL `ALTER ROLE ingest_writer WITH LOGIN PASSWORD '...';` записан
     во временный `.sql`-файл через python `tempfile.mkstemp` (пароль
     в нём ровно одну секунду, между mkstemp и psql).
  3. Применён через `PGSSLMODE=require psql "$URL" < "$SQL_FILE"`.
  4. Временный SQL-файл удалён `rm -f` сразу после применения.
  5. DSN с url-encoded паролем сформирован python'ом, дописан в
     `.env.local.preprod.v2` (gitignored), временный txt с паролем
     удалён.
  6. **В git, в LLM-чате этого пакета и в любом stdout пароль НЕ
     присутствует.** (Первая попытка дала утечку через
     stderr-сообщение grep, пароль ротирован сразу же — см. раздел 4.)
- Документ `docs/preprod-deploy.md` теперь содержит «Шаг Л» —
  достаточно подробный, чтобы 9e.4 (prod) сделал то же, не возвращаясь
  к моему чату; в нём же оговорка про Windows-psql и stdin redirection.
- Plan обновлён: блок «9e.1 завершён 2026-05-11» в «Что реализовано»,
  backlog #5 переформулирован — путь A зафиксирован, 9e.2/9e.3/9e.4
  открыты.
- Ничего в Python-коде не менялось — это чистый DBA-этап.
