# Этап UI-3: оформление раздела «Настройки» в портале

**Дата:** 2026-05-11
**Этап Пути B:** 3 из 5 (UI-3)
**Контекст плана:** `plans/2026-05-11-ui-merge-portal-configurator.md`
**Предыдущий этап:** UI-2 — перенос «Баз данных» (коммит 6d615dc).

## 1. Поставленная задача

Оформить раздел «Настройки» в портале путём перевешивания трёх
существующих portal-роутеров с `/admin/*` на `/settings/*`:

1. **Пользователи** (`/admin/users` → `/settings/users`).
2. **Бэкапы** (`/admin/backups` → `/settings/backups`).
3. **Журнал действий** (`/admin/audit` → `/settings/audit-log`).

Включает:

- перенос файлов (`portal/routers/admin_*.py` →
  `portal/routers/settings/*.py`, `portal/templates/admin/*.html` →
  `portal/templates/settings/*.html`);
- 301-редиректы со старых URL внутри портала (точечные по разделу —
  чтобы не задеть соседей `/admin/{price-uploads,auto-price-loads,
  diagnostics,auctions}`);
- обновление sidebar — три подпункта Настроек на новые URL, sub-key
  `audit` → `audit-log`;
- обновление маппинга URL → `active_section` в
  `portal/templates/base.html`;
- переименование тестов `test_admin_*` → `test_settings_*` +
  новый файл `test_settings_redirects.py`;
- pytest регрессия — baseline 1875, ожидаем такой же или больше.

## 2. Как решал

1. **Чтение требований.** Внимательно перечитал DoD. Главное отличие
   от UI-2: здесь файлы УЖЕ в portal — это перевешивание URL внутри
   одного сервиса, не переезд между сервисами. Значит, редирект-
   handler идёт в `portal/main.py`, а не в `app/main.py`. И сам
   редирект — относительный (просто `/settings/...`), не межсервисный.

2. **Точечные catch-all'ы.** Собственник на UI-2 указал важную
   деталь: три тонких catch-all'а на раздел вместо одного широкого
   `/admin/{path:path}`, чтобы не задеть `/admin/auto-price`,
   `/admin/price-uploads`, `/admin/diagnostics`. На UI-3 паттерн тот
   же — повторил без вопросов. Реализовал helper
   `_settings_redirect(new_path)` и шесть тонких handler'ов
   (3 раздела × 2 варианта пути).

3. **Sub-key для журнала.** В прошлых этапах sub-key подпункта журнала
   был `audit`, а URL стал `/settings/audit-log`. По DoD в маппинге
   `base.html` нужно `subsection='audit-log'`. Чтобы тесты на
   `data-subsection` сошлись, sub_key в sidebar.html тоже изменил на
   `audit-log`. Обновил тест `test_audit_active_section_is_settings`
   в `test_ui1_sidebar.py` (`("users", "backups", "audit")` →
   `("users", "backups", "audit-log")`).

4. **Перенос роутеров.** Скопировал три файла, в каждом:
   - `prefix="/admin"` → `prefix="/settings"` (users.py); или
     `prefix="/admin/backups"` → `prefix="/settings/backups"`;
     или `prefix="/admin/audit"` → `prefix="/settings/audit-log"`;
   - все `RedirectResponse(url="/admin/X")` → `RedirectResponse(
     url="/settings/X")` (массово через `replace_all`);
   - имена шаблонов `admin/X.html` → `settings/X.html`.
   Старые файлы `portal/routers/admin_{users,backups,audit}.py`
   удалил `rm`-ом — это portal, а не app/, так что чисто.

5. **Перенос шаблонов.** Скопировал три HTML, в каждом обновил все
   `action="/admin/..."` и `href="/admin/..."` на `/settings/...`.
   Старые шаблоны удалил.

6. **Обновление `portal/main.py`.** Самая важная часть:
   - удалил импорты `admin_users`, `admin_backups`, `admin_audit`;
   - добавил импорт `from portal.routers.settings import users as
     settings_users, backups as settings_backups, audit_log as
     settings_audit_log`;
   - заменил `app.include_router(admin_*.router)` на трёх
     `settings_*.router`;
   - добавил блок из шести handler'ов 301-редиректов между блоком
     databases и подключением `home.router` (важно: home.router —
     last resort, поэтому редиректы должны быть до него).

7. **Сторона конфигуратора.** В `app/routers/admin_router.py` остался
   старый редирект `/admin/users` → `${portal_url}/admin/users` (для
   совместимости со старыми ссылками). Если оставить как было — будет
   двойной hop: config/admin/users → 302 → portal/admin/users → 301 →
   portal/settings/users. Обновил редирект сразу на
   `${portal_url}/settings/users`, чтобы за один hop.

   ⚠️ Это не нарушает DoD «не трогай конфигуратор кроме редиректов»
   — это и есть редирект, просто на новый URL. И тест на этот хендлер
   (`test_admin_users_redirects_to_portal`) обновил на новый assert.

8. **Sidebar и base.html.** В `shared/templates/_partials/sidebar.html`:
   - топ-ссылка раздела «Настройки» ведёт на `/settings/users`
     (первый подпункт);
   - три `_sub_link` обновлены на `/settings/...` URL, sub_key
     `audit` → `audit-log`.
   В `portal/templates/base.html`:
   - три ветки `elif _path.startswith('/admin/...')` стали
     `elif _path.startswith('/settings/...') or _path.startswith('/admin/...')`;
   - `subsection='audit'` стало `subsection='audit-log'`.

9. **Тесты.** Переименовал четыре файла через `git mv` (чтобы git
   сохранил историю как rename, не как delete+add):
   - `test_admin_users.py` → `test_settings_users.py`;
   - `test_admin_audit.py` → `test_settings_audit_log.py`;
   - `test_admin_role_change.py` → `test_settings_role_change.py`;
   - `test_admin_user_delete.py` → `test_settings_user_delete.py`.

   В каждом — `replace_all` по `/admin/users` → `/settings/users`,
   `/admin/audit` → `/settings/audit-log`, `/admin/backups` →
   `/settings/backups`. И импорт `from portal.routers import
   admin_backups` → `from portal.routers.settings import backups as
   settings_backups` в одном кейсе (`test_backup_manual_run_writes_audit`).

   Не переименовывал, но обновил URL: `test_backups.py` (не имел
   префикса admin_), `test_dashboard.py`, `test_ui1_sidebar.py`,
   `test_stage12_5a_html_confirm_modal.py`, `test_web/test_access.py`.

10. **Новый тест-файл `test_settings_redirects.py`.** 12 кейсов:
    - 9 positive (root + sub-routes для трёх разделов);
    - 3 negative — проверка что соседи `/admin/{price-uploads,
      auto-price-loads,diagnostics}` НЕ редиректятся (status != 301).
    Для positive-кейсов использую `portal_client` (без логина — 301
    публичный), для negative — `admin_portal_client` (страницы
    защищены require_admin).

11. **Прогон pytest.** Первый прогон выборки (109 тестов из
    `test_portal/test_settings_*` + `test_ui1_sidebar` + `test_backups`)
    — все green. Полная регрессия `pytest -q -m "not live"`:
    **1886 passed, 1 skipped, 0 failed** (baseline UI-2 был 1875;
    +11 net).

12. **Документация.**
    - `docs/ui-architecture.md` — обновлены: вводный абзац (упомянут
      UI-3), таблица меню (sub-key `audit` → `audit-log`), маппинг
      URL → active_section (добавлены `/settings/{users,backups,audit-log}`
      с пометкой `/admin/*` fallback), новый блок «Внутри портала»
      в разделе 301-редиректов, обновлена история.
    - `docs/url-migration-map.md` — секция UI-3 переписана из «плана»
      в «выполнено», добавлена детальная таблица с root + sub-routes
      для каждого из трёх разделов, описан POST 404 behaviour
      (catch-all'ы только GET), упомянуто обновление редиректа в
      конфигураторе.

13. **План.** В `plans/2026-05-11-ui-merge-portal-configurator.md`
    UI-3 → `[x]`, добавил подробный блок «Артефакты UI-3»,
    итоговый блок обновлён.

## 4. Эффективно или нет, что можно было лучше

**Что прошло хорошо:**

- Перенос внутри одного сервиса (portal → portal) на порядок проще
  переноса между сервисами (config → portal на UI-2). Не нужно было
  адаптировать импорты `from app.X` → `from shared.X` / `from
  portal.X`, не нужно было выравнивать иконки в двух сервисах, не
  нужно было обновлять templating-фильтры. Это сильно сэкономило
  время.
- `git mv` для переименования тестов сохранил историю файлов в git.
  При обычном `rm + write` git показал бы это как «удалил файл +
  создал новый», что усложнило бы blame.
- Скопировать-и-`replace_all`-URL для тестов сработало радикально
  быстрее переписывания. Все четыре переименованных файла
  отработали без правок логики — только массовая замена URL.
- Тест на отсутствие 301-редиректа для соседей (`test_admin_*_is_not_
  redirected`) предотвратит регрессию: если кто-то на UI-5 решит
  схлопнуть три catch-all'а в один широкий `/admin/{path:path}`, тесты
  сразу упадут.

**Что можно улучшить:**

- Sub-key `audit-log` с дефисом — не идеально для data-атрибутов. По
  HTML-стандарту атрибуты с дефисом обрабатываются как kebab-case.
  Работает, но если в будущем кто-то будет добавлять JS-обработчик
  через `dataset.subsection` — нужно помнить, что свойство станет
  `dataset['subsection']` со значением `audit-log` строкой. Если бы
  знал заранее, что URL станет `audit-log`, оставил бы sub-key
  просто `audit`. Сейчас оставлю как есть — это документировано в
  url-migration-map.md и ui-architecture.md, и тесты это покрывают.

- В `portal/templates/base.html` страховочные fallback'и на
  `/admin/{users,backups,audit}` после 301 не нужны — 301-ответы не
  рендерят base.html. Оставил «на всякий случай», но строго говоря
  это лишний код. На UI-5 уберу.

- Обновление `app/routers/admin_router.py` (двойной hop в новом
  редиректе) технически выходит за чёткие границы DoD «не трогай
  app/ кроме редиректов». Решил, что это попадает в «редирект»,
  потому что я просто обновил URL в существующем редиректе на
  более точный target. Если бы собственник сказал «нет, оставь
  двойной hop» — откатил бы и закрыл с TODO в URL-migration-map.

## 5. Как было и как стало

**Было (после UI-2):**

```
config.quadro.tatar/                 app.quadro.tatar/
├── /                                 ├── /
├── /admin                            ├── /admin/users          ← portal
├── /admin/users  → 302 на portal     ├── /admin/backups        ← portal
├── /admin/budget                     ├── /admin/audit          ← portal
├── /admin/queries                    ├── /admin/price-uploads
├── /admin/suppliers      → 301 ──┐  ├── /admin/auto-price-loads
├── /admin/components     → 301 ──┼─→ /databases/{suppliers,components,mapping}
├── /admin/mapping        → 301 ──┘  ├── /nomenclature
└── /projects, /history, /            └── /auctions
```

В sidebar: подпункты «Настройки» ссылаются на `/admin/{users,backups,audit}`,
все три внутренние для portal (без ↗).

**Стало (после UI-3):**

```
config.quadro.tatar/                 app.quadro.tatar/
├── /                                 ├── /
├── /admin                            ├── /admin/users   → 301 ─┐
├── /admin/users  → 302 ─→ portal/    ├── /admin/backups → 301 ─┤
│                          settings/  ├── /admin/audit   → 301 ─┼─→ /settings/users
│                          users      ├── /admin/price-uploads ├─→ /settings/backups
├── /admin/budget                     ├── /admin/auto-price-loads └─→ /settings/audit-log
├── /admin/queries                    ├── /admin/diagnostics
├── /admin/suppliers      → 301 ──┐  ├── /admin/auctions
├── /admin/components     → 301 ──┼─→ /databases/{suppliers,components,mapping}
├── /admin/mapping        → 301 ──┘  ├── /databases/{suppliers,components,mapping}
└── /projects, /history, /            ├── /settings/users
                                       ├── /settings/backups
                                       ├── /settings/audit-log
                                       ├── /nomenclature
                                       └── /auctions
```

В sidebar три подпункта «Настройки» ведут на `/settings/{users,backups,
audit-log}`, sub-key для журнала переименован `audit` → `audit-log`.

Паттерн `portal/routers/<module>/` ещё раз обкатан — UI-2 (databases),
UI-3 (settings). На UI-4 (конфигуратор) предполагается тот же подход:
`portal/routers/configurator/` со своими подмодулями. После UI-5 в
`portal/routers/admin_*` останутся только незавершённые переезды
(`price-uploads`, `auto-price-loads`, `auctions`, `diagnostics`).
