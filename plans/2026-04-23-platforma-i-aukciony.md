# План: Платформа QuadroTech и автоматизация аукционов

**Дата:** 2026-04-23 (последнее крупное обновление: 2026-04-24)
**Название функции:** Платформа QuadroTech (минимальный каркас) + модуль «Аукционы» (ингест → матчинг по КТРУ → дашборд для менеджера)
**Владелец:** Собственник-1 (продукт, реализатор, domain-expert)
**Пользователи:** менеджер-тендерщик (основной пользователь UI), собственник-1 (ревью/контроль, получает сводный дайджест)

**Цель:** Построить минимальную платформу QuadroTech и модуль «Аукционы» — ежедневный ингест извещений 44-ФЗ по всей РФ с фильтром по конкретным КТРУ-кодам (принтеры и МФУ), автоматический матчинг с нашим справочником оборудования по точному КТРУ-коду и атрибутам карточки лота, расчёт ориентировочной маржи, дашборд с фильтром по марже и уведомлениями в Telegram/Max через существующий почтовый агент. Убрать у менеджера ручной поиск и первичный отсев лотов; оставить за ним принятие решения и подачу заявки в ЕИС штатным путём.

**Рамка (важно):** MVP — это **этап 1 из 2** главной гипотезы года. Этап 1 = поиск + подбор + расчёт маржи, финальное решение и подача — за менеджером. Этап 2 (автоучастие в аукционе с контролем человеком) — отдельный проект Q4 2026, запускается только при подтверждённых метриках пилота этапа 1. Не путать этап 1 и этап 2 в обсуждениях и приоритетах.

## Executive summary

Строим узкий рабочий инструмент: один Postgres, один процесс FastAPI с APScheduler внутри, Jinja2+HTMX+Alpine в вебе, без Node.js.

**Фокус MVP — только принтеры и МФУ**, ровно 4 целевых КТРУ-кода (ч/б и цветные). Расширение категории и атрибутов в будущем — через добавление записей в таблицу `ktru_watchlist` без изменений в коде.

**Справочник номенклатуры** наполняется автоматически из Excel-прайсов **8 дистрибьюторов** (Merlion, OCS, Treolan, Ресурс-Медиа, ASBIS, SanDisk, Марвел, А1Тис). Парсер **copy+adapt из `ConfiguratorPC2`** — три адаптера уже готовы, пять пишем заново по тому же интерфейсу (`BasePriceLoader`). Обновление прайсов — кнопкой в UI (`/nomenclature/upload-price`), синхронно, с отчётом «добавлено / обновлено / не смаппилось».

**Атрибуты моделей** (скорость печати, цветность, формат, дуплекс, разрешение, интерфейсы, ресурс картриджа и пр.) собираются **через оркестрацию Claude Code, не через runtime LLM** (паттерн взят из `ConfiguratorPC2/app/services/enrichment/claude_code/`): `scripts/enrich_export.py` выгружает SKU без атрибутов в `enrichment/pending/*.json`, Claude Code в отдельной сессии или через `Agent` tool обходит официальные сайты производителей (через WebSearch + WebFetch) и кладёт результат в `enrichment/done/*.json`, `scripts/enrich_import.py` мерджит в `nomenclature.attrs_jsonb` и переносит файлы в `enrichment/archive/`. Что не нашли → `n/a`. В матчинге (Волна 2) `n/a` трактуется как «подходит, но менеджер должен уточнить» — в карточке лота показывается жёлтый бейдж рядом с такой характеристикой.

**Ингест аукционов** — скрейп zakupki.gov.ru по фильтру из `ktru_watchlist`. Парсим карточку: НМЦК, заказчик, регион, дедлайны, позиции с КТРУ и атрибутами. Idempotency по номеру извещения.

**Матчинг** — чистый SQL: точное совпадение КТРУ-кода → проверка атрибутов лота против атрибутов SKU (≥/=/≤ по каждому атрибуту, правила определит агент Волны 2 эмпирически на живых данных). Маржа на позицию: `margin_rub = nmck_per_unit - cost_base`, логистика в MVP не учитывается.

**Фильтры лота** (все редактируются в UI, хранятся в таблице `settings` или отдельных таблицах):
- `nmck_min_rub` — минимум НМЦК (дефолт `30 000`)
- `max_price_per_unit_rub` — если в лоте есть позиция дороже этой цены, выкидываем весь лот (дефолт `300 000`)
- `margin_threshold_pct` — порог маржи для inbox (дефолт `15`)
- `excluded_regions` — стоп-лист регионов (7 регионов на старте: Приморский край, Сахалинская область, Якутия, Камчатский край, Магаданская область, Чукотский АО, Калининградская область)
- `ktru_watchlist` — список активных КТРУ (seed — 4 кода)

**Дашборд** — inbox с секциями (срочные / готовы к ревью / в работе / архив) и статусной машиной `new → in_review → will_bid → submitted → won | skipped`. После статуса `won` — форма ключевых контрактных дат (ручной ввод + попытка автопарсинга страницы контракта на zakupki.gov.ru по номеру реестра). При смене статуса на `will_bid` — чек-лист-напоминалка «проверь ТЗ на пусконаладку/обучение/монтаж».

**Уведомления** — два разных дайджеста + срочные алерты через HTTP endpoint в существующем `email-agent` (`POST /notify/auction`):
- **Менеджер:** Max, 09:00 MSK, охват «за вчера»
- **Собственник:** Telegram, 20:00 MSK, охват «за сегодня», короткая сводка + топ-5 по марже
- **Срочный алерт** (дедлайн подачи <24ч для лотов в `in_review`/`will_bid`): Max менеджеру
- **Контрактные напоминания** за N дней до ключевой даты: Max менеджеру + Telegram собственнику (инфо-копия). N = `settings.contract_reminder_days` (дефолт 3)

**Что сознательно НЕ делаем в MVP:** LLM-разбор ТЗ (любой результат чекпоинта — всё равно post-MVP), логистику по ПЭК, ПК/мониторы/сканеры/ИБП (только печать), XLSX-экспорт, top-3 альтернативные SKU со сравнением атрибутов, feedback-кнопки на матчинге, клавиатурные шорткаты, pre-submission summary, статус `lost`/`missed_deadline`, RBAC через permissions, bcrypt-сессии (Basic Auth), отдельный worker-процесс, Sentry, Langfuse, SOPS, Metabase, S3-бэкап, Google Sheets sync, OCR, автоподача в ЕИС, mp/CRM.

**Основные риски:**
1. **Прайсы дистрибьюторов 5 новых поставщиков** — форматы Excel разные, могут быть сюрпризы. Митигация: пишем по одному адаптеру, начинаем с самого объёмного. Буфер 21–22.05 в запасе.
2. **Схема атрибутов КТРУ для сравнения** (какие ≥/=/≤) — живая эмпирика, до ингеста не знаем точно. Митигация: агент Волны 2 анализирует 20–30 реальных карточек и выдаёт правила, собственник утверждает.
3. **Парсинг страницы контракта на zakupki.gov.ru** для автозаполнения дат — может оказаться, что даты в PDF, а не на странице. Митигация: если не выходит, оставляем ручной ввод.
4. **zakupki.gov.ru блокирует скрейп** — ретрай с экспонентой, user-agent, задержки; fallback на ZIP-реестры (отложено, но не выкинуто).

## Технические решения (компактно)

**Веб-слой.** FastAPI + uvicorn, Jinja2 SSR, Tailwind CDN, HTMX + Alpine. Без React, без Node.js.

**БД.** Один Postgres. Голый SQL в `migrations/*.sql`, SQLAlchemy 2.0 только `text()`. Никаких pg_trgm / tsvector / pgvector в MVP.

**Фоновые задачи.** APScheduler внутри FastAPI-процесса. Отдельный worker — post-MVP при деградации UI.

**Справочник и прайсы.** Парсер скопирован и адаптирован из `ConfiguratorPC2` (`BasePriceLoader` + `orchestrator` + `matching`). 8 адаптеров дистрибьюторов, UI-кнопка «Обновить прайсы». Атрибуты моделей — парсингом DNS.ru по MPN.

**Уведомления.** Новый HTTP endpoint `/notify/auction` в существующем `email-agent`; QuadroTech-платформа шлёт туда готовый текст с каналом и chat_id. 4 типа событий: дайджест менеджера, дайджест собственника, срочный алерт, контрактное напоминание.

**Матчинг.** Чистый SQL: точное совпадение КТРУ-кода → атрибутный джоин по правилам (≥/=/≤), зафиксированным в `attribute_rules.py` → выбор по марже.

**LLM.** В MVP не используется вообще. Budget-гард (лимит 100 ₽/день) — заглушка в `app/shared/llm/budget_guard.py`, активируется только в post-MVP LLM-fallback-проекте.

**Авторизация.** Basic Auth через `.env` на 2 пользователя. Без bcrypt/сессий/RBAC.

**Audit.** Минимальный: `changed_at`/`changed_by` колонки в `tender_status`, `settings`, `excluded_regions`, `nomenclature`.

**Наблюдаемость.** loguru в файл + pg_dump в cron локально. Без Sentry / Langfuse / SOPS / Metabase / S3-бэкапа.

**Деплой.** Существующая инфра, docker-compose, nginx. Откат — `git revert` + `docker compose up` предыдущего образа.

## Пользовательский сценарий менеджера (happy path)

1. Утром менеджер открывает Max → видит дайджест «18 новых лотов с маржой ≥ 15%, 3 срочных (дедлайн <48 ч)».
2. Заходит на `/auctions` → секция «Срочные» сверху, потом «Готовы к ревью».
3. Открывает лот → видит шапку, таблицу позиций с primary SKU и маржой. Под каждой позицией — список alternative SKU.
4. Если нужно — меняет выбор SKU, либо заходит в `/nomenclature` для ручного подбора.
5. Помечает лот `will_bid` → модалка «проверь ТЗ на пусконаладку/обучение/монтаж», три чекбокса, менеджер подтверждает.
6. Идёт в ЕИС, подаёт заявку штатным путём.
7. Возвращается в `/auctions/{id}`, ставит статус `submitted`.
8. После оглашения результатов — статус `won` или `skipped`. При `won` — вводит номер реестра контракта, жмёт «Подтянуть даты из ЕИС», корректирует, сохраняет.
9. В последующие дни получает напоминания в Max за 3 дня до каждой контрактной даты.

## Принятые решения собственника (зафиксировано 2026-04-24)

1. **Логистика в MVP не считается.** Формула C (сложная, по тарифам ПЭК) выносится в отдельный модуль «Логистика» в post-MVP. В MVP в карточке лота показываем только регион поставки заказчика; формула маржи = `nmck_per_unit - cost_base_rub`.
2. **Порог маржи — 15% по умолчанию, редактируется из UI** (не из `.env`). Хранится в `settings.margin_threshold_pct`.
3. **Почтовый агент — доступен собственнику-1.** Путь `Почтовый агент/email-agent/`, стек Python 3.11+ / httpx / python-telegram-bot / Railway. `TelegramNotifier` и `MaxNotifier` уже реализованы. Волна 3Б добавляет HTTP-endpoint `POST /notify/auction`.
4. **LLM-бюджет-гард — лимит 100 ₽/день, счётчик стоимости на запрос.** В MVP — только заглушка, активируется в post-MVP LLM-fallback-проекте.
5. **Чекпоинт Волны 2 — при любом результате:** MVP остаётся как есть (работает на структурированной части потока), LLM-fallback выносится отдельным post-MVP проектом. Буфер 21–22.05 и 25.05 — только под UX-хотфиксы и хотфиксы дня запуска.
6. **Фокус MVP — только принтеры и МФУ**, 4 КТРУ-кода. ПК / мониторы / сканеры / ИБП — post-MVP. КТРУ-коды хранятся в таблице `ktru_watchlist`, не в коде — расширяется без деплоя.
7. **НМЦК ≥ 30 000 ₽, цена за позицию ≤ 300 000 ₽/шт** (если хоть одна позиция дороже — выкидываем весь лот). Редактируется в UI.
8. **Стоп-лист регионов** (таблица `excluded_regions`): Приморский край, Сахалинская область, Якутия, Камчатский край, Магаданская область, Чукотский АО, Калининградская область. Редактируется в UI чекбоксами.
9. **Все пороги и фильтры — в UI, не в `.env`.** Правило для всех будущих параметров QuadroTech-платформы.
10. **Справочник SKU — автоматический из Excel-прайсов 8 дистрибьюторов** (Merlion, OCS, Treolan, Ресурс-Медиа, ASBIS, SanDisk, Марвел, А1Тис). Атрибуты — парсингом DNS.ru. UI-кнопка «Обновить прайсы» — синхронно, с отчётом. Прайсы не под NDA.
11. **Архитектура парсера прайсов:** copy+adapt из `ConfiguratorPC2` в `app/modules/auctions/price_loaders/`. Вынос в общий пакет `quadro_price_loaders/` — post-MVP, если целесообразно по опыту.
12. **Эвристики отбора лотов:** основная логика «подходит товар → проходит по цене → вписывается в срок». Тип заказчика, обеспечение заявки, обеспечение контракта — не смотрим. Срок поставки — в карточке, не в списке. Пост-победные дедлайны — важны, но вносятся вручную (см. п.14).
13. **Чек-лист «пусконаладка/обучение/монтаж»** показываем модалкой при смене статуса на `will_bid`. Автопарсинг документов — post-MVP.
14. **Пост-победные дедлайны — вариант (b):** в MVP добавляем статус `won` + форма с номером реестра контракта и ключевыми датами (ручной ввод). Попытка автопарсинга страницы контракта на zakupki.gov.ru по номеру реестра — да, простой скрипт. Если не тянет — ручной ввод. Напоминания за N дней до даты (дефолт 3) через email-agent.
15. **Google Sheets** менеджер сейчас использует для своего учёта лотов — в MVP платформа полностью заменяет эту таблицу для стадий «поиск → подача → победа → закрытие». Sync/export в Sheets — post-MVP если понадобится.
16. **Каналы уведомлений:** менеджеру Max (09:00 MSK, охват «за вчера»), собственнику Telegram (20:00 MSK, охват «за сегодня», сводка + топ-5). Срочные алерты (<24ч) — Max менеджеру. Контрактные напоминания — Max менеджеру + Telegram собственнику (инфо-копия).
17. **Карточка лота** содержит минимум: шапка, позиции, ссылка на ЕИС. Скачанных PDF, контактов заказчика, шаблонов заявки в MVP не добавляем.
18. **Обогащение атрибутов — через Claude Code, не через runtime LLM в платформе** (решение от 2026-04-25). Архитектура из `ConfiguratorPC2/app/services/enrichment/claude_code/`: пара скриптов `enrich_export.py`/`enrich_import.py` + JSON-батчи в `enrichment/pending`/`done`/`archive` + промт-шаблон в `enrichment/prompts/`. Claude Code (в отдельной сессии или через `Agent` tool оркестратора) берёт `pending/`, обходит сайты производителей, кладёт в `done/`. API-ключ Anthropic в платформе не нужен, budget-guard остаётся заглушкой. Источник атрибутов — **только официальные сайты производителей**, не DNS / не e2e4. Не нашли → `n/a`, в матчинге трактуется как «подходит, но уточнить».
19. **Матчинг атрибутов — критические vs второстепенные** (решение от 2026-05-07). Не отбрасывать SKU из-за несоответствия по второстепенным/малозначимым характеристикам или из-за `n/a` у SKU. **Критические атрибуты** (всегда сверяем строго; SKU должен удовлетворять требованию лота с правилом ≥/=/≤): формат, цветность, дуплекс, скорость печати. **Второстепенные** (не блокируют матчинг; помечаются жёлтым бейджем «уточнить»): всё остальное — отдельные форматы сканирования, наличие конкретных интерфейсов, ресурс картриджа в комплекте и т.п. Если у SKU стоит `n/a` по любому атрибуту (включая критический) — SKU **не выбрасывается**, но в карточке лота помечается бейджем «уточнить» рядом с этим атрибутом, чтобы менеджер проверил вручную. Why: на сайтах производителей характеристики часто опускаются для самоочевидных параметров; жёсткий фильтр потерял бы валидные SKU.
20. **«Электрография» ≡ «лазерная» в `print_technology`** (решение собственника-domain-expert от 2026-05-07). При сравнении атрибута print_technology значение «электрографическая» из требования лота и «лазерная» из карточки SKU считаются эквивалентными. Маппинг фиксируется в `app/modules/auctions/match/attribute_rules.py`, sentinel-кейс из Волны 2 заменяется на нормальный equivalence-маппинг. Why: это одно и то же по сути; в текстах лотов госзаказчики используют формулировку «электрографическая печать», производители — «лазерная».
21. **Inbox `/auctions` — 5 секций, не 4** (решение от 2026-05-09). К исходной четвёрке (срочно / готовы к ревью / в работе / архив) добавлена 5-я — `new_low_margin` («новые без сильной маржи»). В неё попадают лоты со статусом `new`, не просроченные, не срочные (`submit_deadline > NOW() + deadline_alert_hours`), без primary-матчей с `margin_pct >= margin_threshold_pct`. Why: без этой секции такие лоты исчезают из view, хотя могут быть валидными — primary часто плохой из-за слабого парсинга атрибутов, менеджер должен иметь возможность пересмотреть руками. Решено оставить как есть; если на свежих данных окажется, что секция засоряется мусором — отключим.
22. **Контакты заказчика в карточке лота — плоский `<details>`-блок без confirm/audit** (решение от 2026-05-09). ФИО / email / phone / должность из `tenders.customer_contacts_jsonb` показываются раскрывающимся блоком, по клику. Без дополнительного «показать персональные данные» подтверждения и без audit-записи на каждое раскрытие. Why: данные публичны в силу 44-ФЗ (предпосылка №2), согласие субъекта по 152-ФЗ ст. 8 не требуется; менеджеру контакты нужны для связи с заказчиком при подаче заявки, лишний клик и audit-шум — overkill.
23. **TREOLAN_API_PASSWORD не считается секретом** (решение от 2026-05-09). Пароль `s4]O]h|W` зафиксирован в `.env.example` и доступен публично через git history. Менять не требуется, отдельного этапа ротации НЕ заводим. Why: этот пароль — технический логин для получения каталога Treolan API (прайс-лист), не даёт доступа к деньгам / счетам / чужим данным; компрометация без последствий. Решение **не распространяется** на остальные креды (`OPENAI_API_KEY`, `SMTP_APP_PASSWORD`, `DATABASE_URL`, `APP_SECRET_KEY`, `ADMIN_PASSWORD`, IMAP-пароли поставщиков и т.д. — для них стандарт «не коммитить, ротировать при утечке» сохраняется).
