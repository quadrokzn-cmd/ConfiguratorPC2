# Общие правила обогащения характеристик через Claude Code

Это база для всех категорий. Конкретные поля и нюансы — в файлах
`gpu.md`, `cooler.md`, `case.md`, `psu.md`, `storage.md`, `motherboard.md`,
`cpu.md`.

> Эти промпты предназначены для **отдельных чатов Claude Code**, которые
> НЕ имеют доступа к нашему репозиторию. Каждый чат:
> 1. получает на вход прикреплённый JSON-файл `enrichment/pending/<категория>/batch_NNN_*.json`,
> 2. через WebSearch/WebFetch ходит на официальные сайты вендоров,
> 3. возвращает заполненный JSON в ответе (или сохраняет в `enrichment/done/<категория>/`).
>
> **Параллельная работа:** одновременно можно открыть несколько чатов на
> разные batch-файлы одной или разных категорий. Каждый чат работает
> независимо, поэтому конфликтов не будет.

## Контекст
Сервис-конфигуратор ПК для компании КВАДРО-ТЕХ. Подбирает совместимые сборки
из реального прайса поставщиков. Достоверность характеристик критична: ошибка
в TDP / типе памяти / форм-факторе приведёт к несовместимой сборке у клиента.

## Жёсткие правила поиска
1. **Только официальные сайты производителей.** Допустимые домены (по
   категориям):
   - **Чипы**: `nvidia.com`, `amd.com`, `intel.com`, `ark.intel.com`
   - **Сборщики GPU/MB**: `asus.com`, `msi.com`, `gigabyte.com`, `aorus.com`,
     `asrock.com`, `palit.com`, `zotac.com`, `pny.com`, `biostar.com.tw`,
     `matrox.com`, `afox.eu`, `afox.ru`, `afox-corp.com`, `sapphiretech.com`,
     `inno3d.com`, `maxsun.com`, `maxsun.com.cn`, `supermicro.com`
   - **Кулеры**: `thermalright.com`, `arctic.de`, `arctic.ac`, `noctua.at`,
     `corsair.com`, `deepcool.com`, `gamerstorm.com`, `bequiet.com`,
     `coolermaster.com`, `idcooling.com`, `pccooler.com.cn`, `alseye.com`
   - **Корпуса**: `jonsbo.com`, `fractal-design.com`, `lian-li.com`,
     `nzxt.com`, `phanteks.com`, `thermaltake.com`, `chenbro.com`,
     `aerocool.io`, `montechpc.com`, `azza.com.tw`, `ocypus.com`,
     `in-win.com`, `aicipc.com`, `hpe.com`
   - **БП и PoE**: `seasonic.com`, `zalman.com`, `chieftec.com`,
     `chieftec.eu`, `fsp-group.com`, `fsplifestyle.com`, `ubnt.com`, `ui.com`,
     `ubiquiti.com`, `cisco.com`
   - **Накопители**: `kingston.com`, `westerndigital.com`, `seagate.com`,
     `netac.com`, `apacer.com`, `samsung.com` (только в whitelist? см. ниже)
   - **SBC и одноплатные**: `raspberrypi.com`, `radxa.com`, `orangepi.org`
   - **Прочее**: `hp.com`, `lenovo.com`, российские ОЕМ-сборщики
     (`fox-line.ru`, `formula-pc.ru`, `accord-pc.ru`, `kingprice.ru`,
     `acd-group.com`)
   > Если конкретный домен из списка отсутствует — это нормально:
   > предметная регулярная проверка whitelist'а живёт в коде, и валидатор
   > отклонит URL, не входящий в неё. Лучше оставить `null` с пояснением,
   > чем рискнуть с серым доменом.
2. **Запрещены**: маркетплейсы (DNS, Citilink, Ozon, Wildberries, Amazon,
   Newegg), агрегаторы (3DNews, Tom's Hardware, TechPowerUp, GPU-Z,
   Videocardz, iXBT), форумы, Wikipedia, кэш Google, блоги вендоров на
   сторонних доменах.
3. **Если на оф. сайте данных нет** — возвращать `value: null` с пояснением
   в `reason`. Ничего не выдумывать. Не достраивать «по аналогии».
4. **Каждое значение** должно сопровождаться `source_url` — конкретной
   страницей, где это значение видно. Не корневая страница бренда, а
   страница продукта или его spec-листа.
5. **HTTPS обязателен.** URL должен начинаться с `https://`.

## Что есть в каждом item входного JSON
| Поле | Тип | Назначение |
|---|---|---|
| `id` | int | id компонента в нашей БД, нужен для возврата (НЕ менять) |
| `manufacturer` | str | производитель (часто пустой, не путать с брендом чипа) |
| `model` | str | внутреннее имя в нашей БД (короткое) |
| `mpn` | str | manufacturer part number — **самый надёжный идентификатор** |
| `gtin` | str | штрихкод, тоже надёжный идентификатор, но менее распространён |
| `raw_names` | list[str] | все названия позиции от поставщиков как они есть в прайсах |
| `current` | dict | поля, уже заполненные в нашей БД — не трогаем, не возвращаем |
| `to_fill` | list[str] | поля, которые нужно найти и вернуть |

**Важно про `raw_names`:** разные поставщики дают разную полноту описания.
В одном raw_name может быть TDP, в другом — частоты, в третьем — список
выходов. Используй ВСЕ raw_names как контекст: они часто содержат ключи
к поиску (артикул, под-модель, объём памяти и т. п.), которых нет в
коротком `model`.

## Алгоритм работы по батчу
1. Открой входной JSON: `enrichment/pending/<категория>/batch_NNN_*.json`.
2. Для каждого `item`:
   - Сначала ищи по `mpn`/`gtin` — это даёт точное попадание.
   - Если не помогло — собирай поисковый запрос из `manufacturer + model`.
   - Доглядывай в `raw_names` — там часто бывает уточнение модели.
3. На странице продукта найди значения только для полей из `to_fill`.
   Поля из `current` уже заполнены — НЕ возвращай их.
4. Заполни выходной JSON в формате ниже.
5. Сохрани результат в `enrichment/done/<категория>/<имя_входного_файла>`.

## Формат выходного JSON
```json
{
  "category": "<тот же>",
  "batch_id": "<тот же>",
  "filled_at": "ISO-8601 UTC, например 2026-04-30T13:45:00Z",
  "items": [
    {
      "id": <int, тот же что во входе>,
      "fields": {
        "<имя_поля>": {
          "value": <значение нужного типа, либо null>,
          "source_url": "https://...",
          "reason": "<необязательно; обязательно при value=null>"
        }
      }
    }
  ]
}
```

Если для item ни одно поле найти не удалось — оставь `fields: {}` либо
не включай item вовсе.

## Чек-лист самопроверки перед сохранением
- [ ] Каждое непустое `value` имеет валидный `source_url`.
- [ ] Все URL — `https://` и из белого списка доменов.
- [ ] Типы значений соответствуют требованиям категории
      (например, `tdp_watts` — целое число, не строка «150W»).
- [ ] Ни одно поле из `current` не задублировано в `fields`.
- [ ] Если значение не нашёл — `null` + `reason`, не пустая строка
      и не нулевое значение.
