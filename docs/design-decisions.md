# Ключевые архитектурные решения

Что решено и почему. Если вы хотите поменять что-то из этого списка —
прочитайте сначала причину, чтобы не наступить на старые грабли.

## 1. USD как единственная валюта в БД

**Решение:** все цены в `supplier_prices` хранятся в долларах США.
Рубли вычисляются на лету через jinja-фильтры `to_rub` / `fmt_rub`
по последнему курсу из `exchange_rates`.

**Почему:**

- Курс ЦБ меняется ежедневно. Хранить рублёвые цены — значит либо
  пересчитывать всю таблицу на каждое обновление курса (медленно
  и шумно), либо мириться с устаревшими цифрами.
- Дистрибьюторы оргтехники (OCS, Merlion, Treolan) ведут закупочный
  учёт в долларах. Хранение в USD — естественная единица их же прайсов.
- Курс обновляется раз-два в день, jinja-фильтр работает в один проход
  по списку компонентов на странице — overhead незначителен.

**Не делайте так:** не вводите рублёвую колонку «для скорости».
Преимущество мнимое, а синхронизация с курсом превращается в источник
багов.

## 2. Фильтры `is_active` / `is_hidden` везде в подборе

**Решение:** все запросы подбора кандидатов включают:

- `suppliers.is_active = TRUE`
- `<component>.is_hidden = FALSE`

**Почему:**

- Поставщик может быть временно отключён (просрочен прайс,
  административная пауза). Подбор должен молча его игнорировать.
- Часть компонентов в БД не предназначена для отдельной продажи —
  например, корпусные вентиляторы, идущие в комплекте с корпусом.
  Скрыть их от NLU удобнее флагом, чем удалять (история цен и
  сопоставлений сохраняется).

**Не делайте так:** не пишите запрос подбора без обоих фильтров.
Появятся «призрачные» предложения с отключёнными поставщиками или
неприменимыми компонентами.

## 3. Шрифт Calibri 11pt в Word-КП через Normal-стиль + явный rFonts

**Решение:** в python-docx экспорте КП:

- Шрифт **Calibri 11pt** прописан в `Normal`-стиле шаблона.
- Таблица КП собирается **программно с нуля** (не через копирование
  готового образца из шаблона).
- На каждый run в каждой ячейке ставится **явный `rFonts`** через
  `w:rFonts w:ascii="Calibri" w:hAnsi="Calibri" w:cs="Calibri"`.

**Почему (Этап 9А.2.7):**

- Без явного `rFonts` python-docx иногда наследует шрифт из родительской
  таблицы образца, а не из Normal — на разных версиях MS Word это
  выглядит по-разному.
- Программная сборка избавляет от артефактов, остающихся при копировании
  таблицы-образца (лишние свойства ячеек, скрытые колонки, форматирование
  с сюрпризами).
- Calibri — стандартный «офисный» шрифт MS Word, не требует поставки
  файла шрифта вместе с .docx.

**Не делайте так:** не возвращайте копирование таблицы из шаблона
ради «удобства» — потеряете контроль над rFonts и колонки опять
поедут (это уже было).

## 4. Курс ЦБ автообновлением через APScheduler — 5 раз в день

**Решение:** `services/export/scheduler.py` поднимает APScheduler с
5 cron-задачами:

- 8:30 МСК
- 13:00 МСК
- 16:00 МСК
- 17:00 МСК
- 18:15 МСК

Каждая тянет XML с `cbr.ru/scripts/XML_daily.asp`, парсит USD и пишет
строку в `exchange_rates`. Актуальный курс — последняя строка.

**Почему (Этап 9А.2.3):**

- ЦБ публикует курс на следующий день около 11:30 МСК. Утренняя выгрузка
  в 8:30 берёт ещё «старый» официальный — это нужно, чтобы ранние
  пользователи видели валидный курс.
- В течение дня ЦБ перевыставляет курс несколько раз. 5 точек ловят
  все потенциальные обновления без захлёста.
- Логи всех обращений к ЦБ — в `logs/`. Если ЦБ недоступен, последний
  валидный курс остаётся в БД и используется до следующего обновления.

## 5. Тёмная enterprise-палитра, brand=#2052E8, точечный синий

**Решение** (этапы 9А.1–9А.2.7 — редизайн UI):

- Базовая палитра: **тёмная** (`surface.base/1/2/3/4` — 5 уровней
  глубины), `ink.primary/secondary/muted/inverse` для текста.
- Акцент: один цвет — **brand** (`#2052E8` основной, диапазон
  50–900). Используется точечно: на ссылках, активных состояниях,
  тонких границах карточек.
- **Свечение**, не заливка. Активность элементов передаётся подсветкой
  (`box-shadow` с brand-оттенком), а не заполнением фона.
- 4 уровня микро-анимаций (от мгновенных feedback-моментов до плавных
  переходов между состояниями).
- Сайдбар — **фиксированный** (на всю высоту окна, не скроллится
  вместе с контентом).

**Почему:**

- Менеджеры работают с конфигуратором по 6–8 часов. Светлая палитра
  утомляет глаза за такое время.
- Один акцентный цвет дисциплинирует — нет соблазна «раскрашивать»
  интерфейс. Информационная иерархия строится через свечение и контраст
  поверхностей.
- Фиксированный сайдбар держит навигацию всегда под рукой при длинных
  проектных страницах.

См. [ui_design_system.md](ui_design_system.md) и [ui_components.md](ui_components.md).

## 6. Tailwind-сборка локальная (а не CDN)

**Решение** (Этап 9А.1):

- `npm install tailwindcss postcss autoprefixer` — dev-зависимости.
- `npm run build:css` собирает `static/dist/main.css` под прод.
- **Собранный CSS коммитится** в репозиторий.

**Почему:**

- CDN добавляет внешнюю зависимость рантайма (риск недоступности,
  медленнее старт, трудно делать офлайн-демо клиенту).
- На Railway/Docker Node.js не нужен — собранный CSS уже в репо,
  Python-контейнер копирует его и отдаёт как статику.
- Tailwind CDN не покрывает все нужные плагины (typography, formes
  и т.п.) и затрудняет кастомизацию через `tailwind.config.js`.

## 7. SQLAlchemy text() без ORM

**Решение:** все запросы к БД — через `sqlalchemy.text(...)`
с именованными параметрами. **ORM-моделей нет.**

**Почему:**

- Запросы подбора кандидатов сложные, с динамическими фильтрами и
  JOIN'ами по 8 таблицам компонентов. ORM-модели сделали бы их
  более многословными без выигрыша в безопасности.
- Параметризованный `text()` защищает от SQL-injection ничуть не хуже
  ORM.
- SQL виден в коде «как есть» — легче читать, легче профилировать,
  легче отдавать DBA на ревью.

**Дисциплина:** **никогда** не собирайте SQL конкатенацией строк
с пользовательским вводом — только параметры через `text(...).bindparams()`
или `:name` placeholder.

## 8. Бюджет OpenAI — дневной лимит в рублях

**Решение:** `services/budget_guard.py` проверяет дневной лимит
расходов на OpenAI **перед каждым вызовом**. Лимит — `DAILY_OPENAI_BUDGET_RUB`
из `.env` (по умолчанию 100 ₽).

**Почему:**

- Бизнес — товарный, не digital-first. OpenAI — внешняя платная
  зависимость, неконтролируемый рост расходов = прямая потеря маржи.
- Дневной лимит даёт верхнюю границу убытка от любого бага (зацикленный
  fallback, спам пользователя). При исчерпании NLU откатывается на
  локальный fuzzy_lookup без LLM.
- Журнал в `api_usage_log` (таблица БД) — админ видит расходы по дням
  на странице `/admin/budget`.

## 9. Программная сборка таблицы КП с нуля

**Решение** (Этап 9А.2.7): таблица КП в Word собирается **полностью
программно** через python-docx, начиная с `add_table()` и установки
ширины колонок руками. Никакого копирования таблицы-образца из
шаблона.

**Почему:**

- Копирование готовой таблицы из шаблона тащит за собой невидимые
  свойства (cell margins, table-level rPr, скрытые колонки), которые
  ломают вёрстку при пересоздании строк.
- Программная сборка — детерминирована. Один и тот же входной список
  компонентов даёт байт-в-байт одинаковый .docx. Это критично для
  тестов экспорта.
- На Этапах 9А.2.5–9А.2.6 копирование образца давало непредсказуемые
  артефакты (узкие колонки Цена/Сумма) — программная сборка эту
  проблему сняла окончательно.

## Что НЕ является архитектурным решением

Эти вещи **изменяемы** без согласования — просто соблюдайте дисциплину
существующего кода:

- Конкретные названия классов компонентов (`btn-primary`, `card`).
- Структура шаблонов (наследование от `base.html`).
- Имена роутов (`/project/{id}`, `/admin/...`).
- Конкретные fields в `BuildRequest`.

Если нужно поменять — посмотрите как сделано в существующих местах
и сделайте симметрично.
