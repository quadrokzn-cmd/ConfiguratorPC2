# Excel-выгрузка и загрузка каталога

Дата: 2026-05-13 (Фазы 1–5 плана `plans/2026-05-13-catalog-excel-export-import.md`).

Админ-инструмент для массовой правки характеристик товаров. Скачивается
xlsx со всей базой, правится локально (Excel/LibreOffice), загружается
обратно — изменения попадают в каталог.

UI-страница: [/databases/catalog-excel](../portal/templates/databases/catalog_excel.html).
Sidebar → «Базы данных» → «Выгрузка/загрузка xlsx». Доступ — только `admin`.

---

## Два файла, два потока данных

| Файл | Источник в БД | Листы |
|---|---|---|
| `Комплектующие_ПК_YYYY-MM-DD.xlsx` | `cpus`, `motherboards`, `rams`, `gpus`, `storages`, `cases`, `psus`, `coolers` | CPU, Motherboard, RAM, GPU, Storage, Case, PSU, Cooler |
| `Печатная_техника_YYYY-MM-DD.xlsx` | `printers_mfu` (фильтр `category='printer'` / `category='mfu'`) | Принтеры, МФУ |

Каждый файл — единое целое: скачиваем тот же файл, что собираемся
загрузить обратно. Листы между файлами не смешиваются.

---

## Структура каждого листа

```
Строка 1: A1 = «Курс ЦБ (USD→RUB)»,  B1 = число (например 97.4523)
Строка 2: (пусто, разделитель)
Строка 3: Заголовки колонок (autofilter активен на этой строке)
Строка 4..N: Данные, по одной строке на товар
```

### Колонки

- **`id` (колонка A, скрытая)** — внутренний PK. По нему импорт
  находит запись для UPDATE. Удалять или править эту колонку нельзя.
- **Редактируемые колонки** — белый фон. Содержат характеристики, имя,
  бренд, SKU, GTIN, `is_hidden`, для печатной техники — `cost_base_rub`,
  `margin_pct_target` и ключи `attrs_jsonb`.
- **Read-only колонки** — жёлтая заливка (`#FFF4CE`). Цены min USD/RUB,
  имя поставщика, дата обновления цен, `attrs_source`. Импорт их
  игнорирует и пишет warning, если значение было изменено.

### Колонка цен и формула RUB

- `Цена min, USD` — статическая (число), берётся из `supplier_prices`
  активного поставщика с минимальной USD-ценой.
- `Цена min, RUB` — если у товара есть USD-цена, ячейка содержит формулу
  `=<USD_cell>*$B$1` (`$B$1` — курс из шапки). Если только RUB —
  статика (формула не нужна).
- При ручной правке `B1` пересчитываются все RUB-формулы автоматически
  (это удобно: можно «прикинуть» каталог по другому курсу).

### Курс ЦБ (`B1`) — fallback при пустой `exchange_rates`

В норме на проде таблица `exchange_rates` не пустая — `ensure_initial_rate()`
зовётся при старте scheduler'а. Но если в момент экспорта таблица всё
же пуста (например, до первого тика на свежей среде), сервис подставит
**фиксированный курс `90.0`** и выставит флаг `rate_is_fallback=True`
в `ExportReport`. В `audit_log` это уйдёт как `rate_fallback: true`.

Альтернативы (вынос в `settings` / возврат 503) отклонены — это
edge-case на холодном старте, defensive default достаточен.

### Сериализация массивов

`TEXT[]` в БД (например `cases.supported_form_factors`,
`coolers.supported_sockets`, `printers_mfu.ktru_codes_array`,
`attrs_jsonb.network_interface`) сериализуются в одну ячейку через
запятую:

```
ATX,mATX,ITX
```

Importer делает `split(',')` + `strip()` + фильтр пустых.

### Пустые ячейки и `n/a`

| В БД | В ячейке Excel | При импорте |
|---|---|---|
| `NULL` / отсутствует ключ | пусто | **ничего не делаем** (COALESCE) |
| `n/a` | `n/a` | пишется `n/a` (маркер «искали — не нашли») |
| значение | значение | UPDATE |

«Обнулить» значение через Excel невозможно — пустая ячейка трактуется
как «не трогать». Это сознательная защита от случайных стираний;
для очистки поля используется отдельный UI.

## Поведение importer'а

### Поиск строки

- **`id` присутствует, в БД есть запись** → `UPDATE` с COALESCE-семантикой.
  Пустые ячейки — пропускаются (значение в БД сохраняется), заполненные —
  записываются.
- **`id` присутствует, в БД нет записи** → строка пропускается, в
  `ImportReport.warnings` появляется запись `«id=N не найден на листе X»`.
- **`id` пустой** → `INSERT` нового товара. Валидируются NOT NULL поля
  (`model`, `manufacturer` для ПК; `sku`, `brand`, `name`, `category` для
  печатной техники). DEFAULT-поля (`is_hidden=FALSE`,
  `ktru_codes_array=[]`) подставляются автоматически.
- **Полностью пустая строка** → skip без ошибки и без warning.

### Транзакция

Один `session.commit()` на весь файл. SQL-ошибка (constraint violation,
неверный enum и т.п.) → `rollback()` всей транзакции, возвращается 500.

Валидационные ошибки на уровне ячейки (неверный тип, неизвестный enum)
**не** прерывают остальные строки — собираются в `ImportReport.errors`,
валидные строки коммитятся.

### Last-write-wins

Если между «скачал → правил → загрузил» товар был обновлён (например,
автозагрузкой прайса), импорт перетрёт значения **только тех ячеек,
которые админ поменял в Excel**. Цены/остатки read-only — они не
страдают. Конфликт-резолюшн (diff-визуализация) — отдельный мини-этап,
если первый MVP покажет реальные коллизии.

### Аудит

Каждая загрузка фиксируется в `audit_log` (`action='catalog_excel_import'`,
`target_type='catalog_excel'`, `target_id='pc'|'printers'`). В payload —
`{updated, inserted, skipped, errors, saved_path, user_id}`.

Скачивание — `action='catalog_excel_export'`, payload —
`{rows_count, sheet_counts, rate_used, rate_fallback}`.

Файл загрузки сохраняется на сервере в `data/catalog_imports/<timestamp>_<filename>.xlsx`
— для отката вручную, если что-то пошло не так.

## Чего impоrter **не** делает

- **Удаление товаров.** Строка, которой нет в файле, остаётся в БД.
  Удаление через Excel-разметку «DELETE» — отдельный мини-этап, если
  потребуется.
- **Изменение цен/поставщиков.** Эти колонки read-only — приходят
  только от поставщиков через автозагрузку прайсов. Ручная правка в
  Excel игнорируется с warning'ом.
- **Конфликт-резолюшн.** Last-write-wins. Diff между Excel и БД на
  момент загрузки не показывается.
- **CSV.** Только `.xlsx`. CSV в первой версии не поддерживается.

## Типичные ошибки и как их читать

JSON-отчёт importer'а имеет структуру:

```json
{
  "updated":      12,
  "inserted":     3,
  "skipped":      1,
  "errors_count": 2,
  "errors": [
    {"sheet": "CPU", "row": 14, "message": "id=999 не найден"},
    {"sheet": "RAM", "row": 22, "message": "cores: не число «abc»"}
  ],
  "warnings": [
    "Лист «GPU», строка 7: попытка изменить read-only колонку «Цена min, USD» — проигнорирована"
  ],
  "saved_path": "data/catalog_imports/2026-05-13_103412_Комплектующие_ПК_2026-05-13.xlsx"
}
```

Что смотреть в первую очередь:

1. **`errors_count > 0`** — есть невалидные строки. Открой каждую по
   `(sheet, row)`, поправь и загрузи снова. Уже импортированные строки
   повторно загружать не вредно — COALESCE-семантика обновит то же
   значение тем же значением.
2. **`warnings`** — попытка изменить read-only колонку. Это безвредно,
   но сигналит «зря тратил время в Excel — это поле не редактируется».
3. **`skipped > 0` без ошибок** — нашлись строки с `id`, которого нет в
   БД (товар был удалён в UI между скачиванием и загрузкой). Сейчас
   они молча пропускаются; если такие пропуски некстати — удали `id`
   в Excel, импорт создаст товар заново как новый.

## CLI-обёртка

Для офлайн-экспорта (без UI) есть `scripts/catalog_excel_export.py`:

```powershell
python scripts/catalog_excel_export.py --target pc --output ./out/
python scripts/catalog_excel_export.py --target printers --output ./out/
python scripts/catalog_excel_export.py --target both    --output ./out/
```

CLI работает с локальной БД (`DATABASE_URL` из `.env`), пишет файл и
выводит JSON-отчёт в stdout. CLI-импорт пока не сделан — импорт идёт
только через UI (тот же сервис `excel_import.import_components_pc` /
`import_printers_mfu` можно дёрнуть из ноутбука/скрипта при необходимости).
